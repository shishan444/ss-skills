# 三层升维认知分析

用中文思维分析问题。目标不是把同一件事机械分成三栏，而是通过逐层挖掘和升维回校，重新定义问题，找到真正决定结果的变量，并给出当前条件下最优的解决方向。

核心路径：

```text
收题
→ 事实层初探
→ 升维到关系因果层并熔炼因素结构
→ 回校事实层焦点
→ 升维到格局演化层并构建局势
→ 回校关系因果层和事实层的形态
→ 分层重构
→ 给出最优方向
```

## 基本原则

- 先补信息，再升维；事实不清时，不做宏大判断。
- 更高层不替代低层，但会反向修订低层的重点、解释和行动方向。
- 每次升维都要说明：上一层解释不了什么，所以需要进入更高层。
- 不把普通原因分析误判为关系因果层；只有进入位置、压力、诉求、定性权和资源分配，才算关系因果层。
- 关系因果层不能只罗列角色和利益。必须展开相关因素，再筛选、合并、熔炼为能解释事实走向的关键因素结构。
- 不把长期、宏观、行业趋势误判为格局演化层；第三层必须落到局、位、道、势、路。
- 格局演化层不能直接给“顺势/借势/破局”等策略词。必须先构建真实局势：边界、参与者、规则、资源分布、主要矛盾、演化方向。
- 高维回校不能只写“焦点改变”。必须说明事实形态、因果形态、决策方向分别发生了什么变更。
- 最终建议必须回到用户的位置、目标、资源、约束、代价和可验证行动。
- 医疗、法律、金融、政治、国际形势等高风险或强时效场景，具体事实必须先查证；如果不能查证，只能给条件化分析，并标明不能替代专业建议或投资建议。
- 对非典型任务先判断分析强度。用户只要复述、安装、改一句触发条件或确认边界时，不要强行铺满三层；用三层判断校准重点即可。
- 推断必须标边界。输入越压缩，越要说明哪些是用户明说，哪些是为了升维分析做的假设。
- 用户只要求“理解/复述意图”时，默认不输出完整方案、最优方向、止损线。最多补一句“后续若推进，应验证 X”。
- 用户给出具体例子时，优先用这些例子构建判断，不要用通用标签替代用户材料。

- 需要确认三层定义和边界时，读 `references/three-layer-model.md`。
- 需要系统追问或补充缺失信息时，读 `references/question-bank.md`。
- 只要选择“完整三层”，必须读 `references/output-template.md` 并按其中的完整结构输出；不得用自由段落替代关系因果结构熔炼、格局构建和升维回校表。
- 需要验证、迭代或优化本 skill 时，读 `evals/rubric.md`、`evals/evals.json` 和 `references/iteration-protocol.md`。

不要一次性读取全部参考文件。按当前缺口读取。

## 1. 收题

先判断用户带来的对象是什么：

- 事件：已经发生了什么，需要判断或处理。
- 任务：需要完成某项工作，需要找到执行方向。
- 决策：要不要做、怎么选、先做什么。
- 关系：人与人、组织、权力、利益之间的问题。
- 局面：多方力量共同作用，单点处理无法解释。

同时明确：

- 用户当前的位置是什么。
- 用户要优化的是单次处理结果、关系影响、资源配置、长期位置，还是未来选择空间。
- 用户是否要求一次性完整分析，还是需要先确认切口。

再选择分析强度：

- 完整三层：复杂决策、系统设计、组织/关系局面、人生选择、用户明确要求“想透/升维/三层分析”。
- 压缩三层：用户材料很短，但背后有定位、方向、方法论或策略含义。
- 边界校准：用户只要求复述意图、安装工具、修改触发条件、确认某个概念边界。此时输出重点是“这件事在三层里应当看到什么”，不要强行写长篇格局判断。

默认规则：

- 开放型策略问题、复杂决策、系统设计、人生选择、商业/内容/产品分析，默认使用完整三层。
- 只有输入明显不完整、用户只要复述、或任务只是边界校准时，才使用压缩三层。
- 完整三层不得省略或简化关键表头：关系因果结构熔炼必须包含“决策含义”，高维回校必须包含“形态变更、被降权的信息、被升权的信息、修订理由”。
- 如果已经选择完整三层，但没有使用 `references/output-template.md` 的核心结构，应当在输出前重写，而不是交付半结构化结果。

边界校准再分两类：

- 复述型：只翻译用户意图，不给完整方案。最多补一句“后续若推进，应验证 X”。
- 执行型：用户给的是命令或改动请求，三层只用于校准改动风险和验收点，最后回到执行清单。

如果缺少会改变判断的关键事实，先问 1-3 个问题。不要为了完整而连续追问细枝末节；能先分析的，先分析并标出不确定点。

## 2. 事实层初探

在具体事情内部整理：

- 已确认事实。
- 用户原话中的判断、偏好和目标。
- 为了继续分析而暂时采用的推断。
- 用户给出的具体样本、例子、措辞和反复强调的点。
- 未确认但重要的信息。
- 当前目标。
- 现实约束。
- 责任、重点、风险。
- 事件内部可行方案。

输出一个事实层初判：

```text
在事实层，这件事暂时可以定义为：[定义]
直接处理方向是：[方向]
会显著改变判断的事实缺口是：[缺口]
```

## 3. 升维到关系因果层

检查事实层是否解释不了以下问题：

- 事实不变，但事情性质、优先级或资源分配会不会因人而变？
- 谁拥有定义这件事的力量？
- 谁能把低优先级问题变成高优先级问题？
- 谁的位置、压力、利益、责任会改变这件事的呈现方式？

如果答案为“是”，进入关系因果层。

按这条生成链分析：

```text
位置与责任
→ 压力与利益
→ 关注点和诉求
→ 对事件的解释与定性
→ 决策和资源分配
→ 事件最终呈现状态
```

必须完成四个动作：

1. 展开：列出会改变事实走向的参与者、位置、压力、利益、定性权、资源分配。
2. 筛选：去掉只解释表象、不改变决策的因素，保留真正影响事实方向的因素。
3. 合并：把重复、从属、同源的因素合并为少数关键因果链。
4. 熔炼：输出一个能支撑判断和决策的因素结构，而不是散点清单。

必须回答：

- 关键参与者是谁。
- 他们分别处在什么位置。
- 他们承担什么压力。
- 他们真正关心什么结果。
- 他们如何定义这件事。
- 他们掌握什么资源。
- 哪些关系决定事情的定性、优先级和资源分配。
- 哪些因素组合会把事实推向不同方向。
- 最终熔炼出的关键因素结构是什么。

## 4. 第一次回校：用关系因果层修订事实层

关系因果层不是在事实层后面追加一段分析，而是要反向检查事实层是否看偏。

必须回校：

- 原先事实层最关注的点是否仍然重要。
- 哪些事实原本看似关键，但在决定者视角下变次要。
- 哪些事实原本被忽略，但会影响定性权、资源或关系。
- 问题定义是否要从“如何处理事情”改为“如何改变解释、关系或资源配置”。
- 原问题应该被重写成什么新问题。
- 下一步应该验证哪一个关系变量，而不是继续补充低价值事实。

用短句输出：

```text
事实层焦点修订：[从 A 改为 B]
事实形态变更：[事实从“孤立事件/表面问题”变成什么结构]
原问题重写：[从“如何处理 X”改为“如何影响/改变 Y”]
修订原因：[关系因果层熔炼出的关键因素结构]
```

## 5. 升维到格局演化层

检查关系因果层是否仍解释不了以下问题：

- 如果换掉具体人物，相似结果是否仍会反复出现？
- 参与者行为是否由更大的制度、行业、组织结构、资源分布、文化惯性或路径依赖生成？
- 用户的问题是否不只是事件处理，而是位置、方向、势能或选择空间问题？
- 局势是否正在变化，导致现在的局部最优未来会失效？

如果答案为“是”，进入格局演化层。

格局演化层必须完成五个动作：

1. 识局：局面边界、参与者、资源分布、规则、依赖、主要矛盾。
2. 定位：用户的位置、资源、能力、约束、目的、可承担代价。
3. 明道：系统长期如此运行的稳定机制，以及适用边界和反证条件。
4. 察势：如果主要条件不变，局面大概率向哪里发展；时间站在哪一边。
5. 择路：顺势、借势、避势、破局、造势，哪一种更适合用户。

执行顺序不能倒置：先识局，再定位，再明道，再察势，最后择路。没有完整局势构建时，不得直接给策略动作。

格局层不能只写趋势词。写完后做一次自检：

- 局：是否说清楚了至少 2 个参与方和 1 个关键约束？
- 位：是否说清楚用户在局中的位置，而不是泛泛说行业趋势？
- 道：是否说清楚一个可迁移的稳定机制，并给出适用边界？
- 势：是否说清楚哪股力量正在增强或衰弱？
- 路：是否把顺势、借势、避势、破局、造势中的一个落成具体路径？
- 变：是否说明格局判断如何改变事实层和关系因果层的形态？

如果做不到以上五点，不要伪装成格局层；改为说明“当前只能形成格局假设，还缺少 X 信息”。

## 6. 第二次回校：用格局演化层修订低层

格局演化层一旦识别出局、位、道、势，就必须回头校准关系因果层和事实层。

必须回校：

- 关系因果层：原先以为要影响的关键人，是否只是系统位置的代理人。
- 关系因果层：真正要改变的是某个人的态度，还是位置、激励、规则或资源结构。
- 事实层：原先要解决的表面问题，是否只是局面中的一个症状。
- 事实层：当前行动应该追求局部最优，还是为改变位置、等待时机、积累资源、退出旧局服务。
- 原问题现在应该被重写成什么决策问题。
- 哪些低层事实可以暂时降权，哪些事实必须优先补齐。
- 下一步验证对象是事实、关系、位置、规则，还是趋势。

输出：

```text
关系因果层修订：[从 A 改为 B]
因果形态变更：[因果从“散点因素/单线原因”变成什么结构]
事实层修订：[从 C 改为 D]
事实形态变更：[事实从“现象描述”变成什么局势症状或结构节点]
原问题重写：[从“X”改为“Y”]
下一步验证对象：[事实/关系/位置/规则/趋势中的一个]
修订原因：[格局演化层识别出的局/位/道/势]
```

## 7. 分层重构

最终按层级展示，不要只给一段综合判断：

| 层级 | 已确认信息 | 关键缺口 | 核心判断 | 对应行动 |
| --- | --- | --- | --- | --- |
| 事实层 |  |  |  |  |
| 关系因果层 |  |  |  |  |
| 格局演化层 |  |  |  |  |

关系因果层必须单独展示因素熔炼结果：

| 因素组 | 参与者/位置 | 压力与利益 | 定性权/资源 | 会推动的事实方向 | 决策含义 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

格局演化层必须单独展示局势构建：

| 局势要素 | 判断 |
| --- | --- |
| 边界 |  |
| 参与者 |  |
| 资源分布 |  |
| 规则/约束 |  |
| 主要矛盾 |  |
| 演化方向 |  |
| 用户位置 |  |
| 可选路径 |  |

然后单独展示高维回校：

| 回校对象 | 原焦点 | 修订后焦点 | 形态变更 | 被降权的信息 | 被升权的信息 | 修订理由 |
| --- | --- | --- | --- | --- | --- | --- |
| 事实层 |  |  |  |  |  |  |
| 关系因果层 |  |  |  |  |  |  |

完整三层分析缺少以下任一项，都视为未完成：

- 事实层初判。
- `## 关系因果结构熔炼`，且表头必须包含“因素组、参与者/位置、压力与利益、定性权/资源、会推动的事实方向、决策含义”。
- `## 格局构建`，且必须呈现边界、参与者、资源分布、规则/约束、主要矛盾、演化方向、用户位置、可选路径。
- `## 升维回校`，且表头必须包含“形态变更、被降权的信息、被升权的信息、修订理由”；不得只写“原焦点/修订后焦点”。
- 原问题重写。
- 最优方向、第一步、验证信号、止损线。

再给出原问题重写：

```text
原问题：[用户表面问题]
升维后的问题：[更准确的问题定义]
这会改变的行动方向：[变化]
```

## 8. 给出最优解决方向

最优方向不是“最高层方向”，而是当前条件下最能作用于真实决定变量的方向。

必须包含：

- 对问题的重新定义。
- 当前最关键的决定变量。
- 推荐方向。
- 第一步行动：优先是 30 分钟到 1 天内能完成的最小验证动作；如果任务较大，则给出最小样例、最小清单或最小路由表。
- 验证信号。
- 止损线或反转条件。
- 仍需补充的信息。

如果证据不足以给出强建议，给出条件化判断：

```text
如果 X 为真，优先走 A；如果 Y 为真，优先走 B。
当前最该验证的是 Z。
```

## 输出风格

- 判断先行，解释跟上。
- 用自然中文，不写报告腔。
- 明确区分用户材料、事实、已有知识和推断。
- 不确定就标注不确定，不把推断写成事实。
- 列表用于结构化展示，关键判断必须用完整句子说清。
- 输出结构服务判断，不服务整齐。对压缩输入和边界任务，可以用压缩三层；对复杂局面，才使用完整表格。
- 对个人画像、自我定位、能力转产品类输入，至少引用 1 个用户原始例子作为判断依据。
- 对复述型任务，最后停在“意图复述 + 边界提醒”，不要擅自进入完整行动方案。
