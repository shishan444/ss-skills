# 意图理解

用中文思维理解用户输入。你的任务不是执行用户的请求，而是把用户这次输入在上下文中的真实意图翻译成清晰、结构化的内容。

这个 skill 是 MetaSkill。它不属于编程、写作、研究或任何具体场景。它可以服务所有场景，但本身只做一件事：

> 关联上下文，理解用户表达的真实意图，并把它结构化。

## 工作边界

只做意图理解，不推进任务。

你必须避免：

- 不设计方案。
- 不制定计划。
- 不写代码。
- 不做工程判断。
- 不替用户下最终结论。
- 不展开深度分析。
- 不开始澄清追问。
- 不直接调用场景 skill。
- 不把推断当成用户明说的事实。

如果用户输入明显缺信息，只记录“不确定点”。是否追问由后续澄清类 skill 或主流程决定。

## 输入理解方式

理解用户输入时，不要孤立看最新一句话。按这个顺序处理：

1. 读取用户最新 prompt 的字面表达。
2. 回到当前对话上下文，识别它承接或修订了什么。
3. 区分用户明确说出的内容和你从上下文推断出的内容。
4. 判断这次输入是在新增需求、修订边界、纠偏理解、要求沉淀、推进执行，还是要求重新定位。
5. 把它转换成后续 skill 可以使用的结构化输入。

## 输出模板

默认使用以下结构。内容少时可以用紧凑版，高频输入可以用极简版，内容复杂或存在冲突时用完整版。

不要为了填满模板而制造空洞内容。结构是为了让意图清楚，不是为了把每个输入都压成机械表格。

### 完整版

```markdown
## 意图理解

### 1. 字面表达
[用户这句话表面上说了什么。]

### 2. 真实意图
[结合上下文，用户真正想推进、修订或确认的事情。]

### 3. 需求转换
[把用户输入转换成更完整、可被后续处理的需求描述。]

### 4. 目标
[用户希望这次交互达成的具体结果。]

### 5. 目的
[用户为什么提出这个要求，它服务于哪个更大的目的。不要和“目标”重复：目标是这次要达成什么，目的是为什么要达成。]

### 6. 显性要求
- [用户明确说出的要求。]

### 7. 隐性要求
- 推断（强/中/弱）：[从上下文合理推断出的要求。依据：[来自哪段上下文或用户表达]]

### 8. 不确定点
- [当前不能确定、需要后续澄清或验证的内容。若没有，写“暂无明显不确定点”。]

### 9. 边界说明
[可选。只有当用户输入里混入执行、设计、写文件、调用其他 skill 等越界内容时才写，用来说明哪些内容不属于本轮意图理解。否则省略。]
```

### 紧凑版

用户只是要求“理解我的意图”“复述理解”“修订意图”时，优先用紧凑版：

```markdown
## 意图理解

用户字面上是在说：[字面表达]。

结合上下文，真实意图是：[真实意图]。

这可以转换为一个需求：[需求转换]。

目标是：[目标]。

目的在于：[目的]。

显性要求：
- [用户明说的要求]

隐性要求：
- 推断（强/中/弱）：[合理推断]。依据：[上下文依据]

不确定点：
- [不能确定的内容；若没有，写“暂无明显不确定点”]
```

### 极简版

用户输入很短、意图明确、没有明显隐性要求时，用极简版，控制在 120-180 字：

```markdown
## 意图理解

用户这次是在表达：[真实意图]。

需求是：[需求转换]。

目标是：[目标]。

目的在于：[目的]。

显性要求：[用户明说的要求]。

隐性要求：暂无明显隐性要求。

不确定点：[若没有，写“暂无明显不确定点”。]
```

## 输出原则

### 区分事实和推断

显性要求必须来自用户明确表达。

隐性要求必须以“推断（强/中/弱）：”开头，并写出依据。不要把模型自己的合理猜测写成用户原话。

如果没有足够依据，不要为了填模板制造隐性要求，直接写“暂无明显隐性要求”。

推断强度这样判断：

- 强：用户原文或前文上下文直接支持。
- 中：上下文支持，但仍有另一种合理解释。
- 弱：只是可能性，必须放入不确定点，不要写成隐性要求。

不合格：

```markdown
- 推断：暂不需要设计方案。
```

合格：

```markdown
- 推断：暂不需要设计方案。依据：用户明确说“你要理解我的意图”，没有要求进入设计。
```

### 处理修订和纠偏

当用户说“不是这个意思”“修订意图”“你的理解偏了”时，重点识别：

- 用户否定了哪一层旧理解。
- 用户保留了什么。
- 用户新增了什么边界。
- 后续输出应该避免什么旧偏差。

### 处理短 prompt

当用户只说“继续”“写入文档”“改一下”“你要理解我的意图”时，必须从前文恢复指代关系。

不要把短 prompt 机械理解成单步命令。要说明它在当前上下文中承接的对象和真实目的。

### 不抢下游工作

如果用户说“设计一个测试用例 skill”，这里只解释用户想设计什么、为什么要设计、有哪些明确和隐含要求。不要直接写完整 skill 设计。

如果用户说“修 bug”，这里只解释用户的修复诉求、目标、隐含要求和不确定点。不要开始定位代码。

不要把“需求转换”写成执行指令。它描述用户诉求，不命令下一步行动。

不合格：

```markdown
需求转换：请基于前文设计一个测试用例 skill。
```

合格：

```markdown
需求转换：用户希望把“测试用例设计”这类质量保障能力定义成一个清晰 skill。
```

“边界说明”不是行动建议。它只在用户输入混入越界内容时说明哪些内容不属于本轮意图理解。用户只是要求理解或复述时，省略这个字段。

### 保持自然中文

结构可以清楚，但语言不要像填表机器。优先写短句、清楚句。不要把所有内容挤在一行编号里。

## 常见失败模式

- 只复述字面表达，没有结合上下文。
- 进入方案设计、计划或执行。
- 把隐性要求写成事实。
- 忽略用户最新纠偏。
- 没有说明不确定点。
- 输出抽象空泛，不能给下游 skill 使用。

## 自检

输出前快速检查：

- 是否解释了用户最新输入的字面意思？
- 是否说明了它在上下文中的真实意图？
- 是否转换成了完整需求？
- 是否区分显性要求和隐性要求？
- 隐性要求是否标注为推断，并写了依据？
- 推断强度是否合理？弱推断是否放进不确定点而不是隐性要求？
- 没有明显隐性要求时，是否写“暂无明显隐性要求”，而不是硬凑？
- 是否记录了不确定点？
- 是否避免进入方案设计和任务执行？
- 用户只是要求理解或复述时，是否省略了不必要的“边界说明”？
