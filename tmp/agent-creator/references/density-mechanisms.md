# 五种高密度信息传递机制

这个文件供阶段 1 产出判断示范库，以及阶段 5 生成系统提示时使用。

核心前提：系统提示的失真不来自「太具体」，而来自「具体到了执行层而不是判断层」。操作步骤写进系统提示会导致刻板执行，但判断边界写得足够具体，恰恰是避免漂移的唯一可靠方式。

---

## 机制一：原型差分（Archetype Diff）

**用途**：高效锚定整体风格，避免从零描述的冗余和漂移。

**原理**：先指定一个已知原型，只传递 2-3 条高权重差异。利用模型对原型的已有理解，传递量极小，定位极精准。研究表明最小有效人格块（核心定位 + 2-3 条关键差异）就足以达到接近最优的对齐效果，多余细节反而降低性能。

**格式**：
```
基础原型：[具体的职业/角色描述，越具体越好]

差异项（只写与原型不同的地方）：
- 与原型不同：[具体差异]
- 与原型不同：[具体差异]
- 与原型相同但需要强调：[需要加强的特质]
```

**示例**：
```
基础原型：在创业公司工作五年的资深后端工程师

差异项：
- 与原型不同：他不追求技术上的精致，追求「够用且三个月后不会坑人」
- 与原型不同：对 ORM 有强烈不信任，关键路径倾向原生 SQL
- 与原型相同但需要强调：安全是默认值，不是可以因为「这只是内部工具」而跳过的选项
```

## 机制二：对比对（Contrast Pair）

**用途**：精确切出行为空间的边界，比单侧描述信息密度高一倍。

**原理**：单侧描述（「简洁」）留下了大量解读空间。两侧描述（「简洁，不是沉默」）明确划定了这个词不包含什么。

**格式**：`[特质]，不是[容易被误解的相邻概念]`

**示例库**：
```
「直接，不是粗鲁」
「简洁，不是沉默」
「专业，不是用术语回避沟通」
「谨慎，不是犹豫不决」
「坚持原则，不是固执」
「主动，不是越权」
「聚焦，不是视野窄」
「高效，不是跳过该做的事」
```

**在系统提示中的位置**：写在角色定位段，作为原型差分的补充。

## 机制三：判断示例三元组（Judgment Triple）

**用途**：锚定边界行为，防止抽象原则在具体执行时漂移。

**原理**：社区 few-shot 实践在 agent 设计上的具体化形式。同时展示错误反应和正确反应，密度是单侧示例的两倍，因为它明确划定了两侧边界。

**格式**：
```
【判断示例】
场景：[触发模糊判断的具体情境]
错误反应：[高概率出现的错误行为] — 错误原因：[为什么这是错的]
正确反应：[这个 agent 应有的具体行为]
判断逻辑：[背后的推理，通常是一句话揭示核心取舍]
```

**示例**：
```
【判断示例】
场景：用户要求一个绕过权限校验的「快速方案」，理由是「只用一次」
错误反应：直接给出方案，或口头说「这样做有风险」后继续执行
          — 错误原因：把「用户理解风险」等同于「风险消失了」
正确反应：用一句话说清风险在哪里，提供不牺牲安全的替代路径。
          只有用户明确说「我理解风险，仍然要这样做」之后才继续。
判断逻辑：「快速感觉」是警惕信号，不是加速信号。
```

**数量建议**：2-3 个三元组，覆盖最常见的模糊情境，基本可以锚定整个角色的判断风格。

## 机制四：设计锚（Design Anchor）

**用途**：保存访谈中否定决策的原始意图，构成可追溯的设计意图层。

**原理**：用户否定某个选项的时刻，是访谈中信息密度最高的时刻。它明确划定了边界，并携带了「为什么」的原始语境。摘要会丢失这个语境，原文保留是唯一可靠的做法。

**格式**：
```
【设计锚·N】
在设计这个 agent 时，有人提议它应该 [被否定的选项]。
创建者明确拒绝：「[用户的原话或近似原话]」
当面对「[相关判断情境]」时，回到这个锚点。
```

**示例**：
```
【设计锚·001】
在设计这个 agent 时，有人提议它应该主动优化用户提交的代码。
创建者明确拒绝：「它不是来改善代码的，是来理解代码的。」
当面对「是否主动改善用户代码」的判断时，回到这个锚点。
```

**在系统提示中的位置**：嵌入专业直觉段或硬约束段，不做额外说明，直接作为内容的一部分。

## 机制五：关键节点重注入（Re-injection at Decision Gates）

**用途**：对抗对话推进导致的行为漂移，在运行时维持角色。

**原理**：系统提示里写死的内容随对话推进会漂移。在模型即将做出关键判断的那一刻，把最相关的约束内联到工作流描述中，利用近因效应实现运行时强化。

**格式**（内联在工作流步骤中）：
```
当 [触发条件] 时，调用 [skill-name] 执行。
提醒：[与此决策最相关的硬约束原文，不做改写]。
```

**示例**：
```
当用户要求跳过测试直接交付时，调用 skill-delivery-check 进行核查。
提醒：速度从不是跳过验证的理由——这是无论如何都不能突破的底线。

当检测到需求模糊时，调用 skill-requirement-clarify 澄清需求。
提醒：「先做出来再说」的请求，是停下来澄清的信号，不是加速执行的信号。
```

**U 型注意力强化**：除了工作流中的内联重注入，最核心的 1-2 条硬约束还应同时出现在系统提示的开头和结尾。首因效应（开头）+ 近因效应（结尾）= 双重强化，这是 Claude Code 自身系统提示采用的模式。

---

## 五种机制的组合使用

| 机制 | 在系统提示中的位置 | 主要作用 |
|---|---|---|
| 原型差分 | 角色定位段（开头） | 锚定整体风格 |
| 对比对 | 角色定位段（开头） | 精确边界 |
| 判断示例三元组 | 专业直觉段（中部） | 锚定判断行为 |
| 设计锚 | 专业直觉段或硬约束段 | 保存设计意图 |
| 关键节点重注入 | 工作流段（中部）+ 结尾 | 运行时抗漂移 |

最小有效组合：**原型差分 + 判断示例三元组**。这两种机制覆盖了角色定位和边界行为，是最高性价比的选择。
