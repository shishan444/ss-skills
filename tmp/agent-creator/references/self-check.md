# 审查评估完整清单

这个文件供阶段 7「审查评估」使用。逐条判断是/否，所有项通过后才进入交付。
发现问题时，直接在本轮修正，不需要用户手动修改。

---

## 第一类：分离正确性（最重要）

这类问题检查「什么放在哪里」是否正确。这是整个设计方案的核心，错误分离会导致 agent 行为混乱或 skill 无法复用。

**检查项 1.1**
系统提示正文中是否出现了操作步骤？
- 操作步骤的识别标准：具体的执行动作、检查项列表、产出格式规定
- 发现时：将该内容提取到对应 skill 的 SKILL.md，系统提示工作流段落改为「调用 skill-xxx 执行」

**检查项 1.2**
任何 skill 的 SKILL.md 中是否出现了判断倾向语言？
- 判断倾向语言的识别标准：「我倾向于」「默认情况下」「你是一个...」「面对 X 时，优先选择 Y」
- 发现时：将该内容移入系统提示的专业直觉段，skill 里删除

**检查项 1.3**
系统提示工作流段落引用的每个 skill 名，是否都有对应的 SKILL.md 文件？
- 检查方法：列出系统提示中所有 skill 引用，逐一核对是否有对应文件
- 发现缺失时：补充对应 SKILL.md，或修改系统提示改用已有 skill

---

## 第二类：约束完整性

**检查项 2.1**
硬约束的措辞是否包含弱化词？
- 弱化词清单：「通常」「一般情况下」「大多数时候」「尽量」「尽可能」「原则上」
- 发现时：改为无条件句式。「通常不做 X」→「不做 X」

**检查项 2.2**
核心硬约束（最重要的 1-2 条）是否同时出现在系统提示的开头和结尾？
- 不需要逐字相同，意思相同即可
- 发现缺失时：在系统提示结尾添加一个「核心约束重申」段落

**检查项 2.3**
系统提示工作流的每个主要决策节点，是否有重注入语句？
- 识别主要决策节点：「当...时」「如果...则」「遇到...」这类条件分支
- 发现缺失时：在分支描述后添加「提醒：[相关约束原文]」

---

## 第三类：触发有效性

**检查项 3.1**
每个 skill 的 description 是否包含：
- 触发词（用户会用什么词触发这个 skill）
- 不触发情况（看起来相似但不该触发的情况）
- 发现不完整时：补充两类信息，参考格式：「[功能定位]。触发词：[词1]、[词2]。不触发：[情况]。」

**检查项 3.2**
subagent 的 description 是否清晰说明「何时调用」这个 agent？
- 有效 description 的标准：以「Use this agent when...」或「当用户...时使用」开头，包含具体场景
- 发现模糊时：改写为场景化描述，加入 2-3 个具体触发词

---

## 第四类：产物完整性

**检查项 4.1**
subagent YAML frontmatter 的必填字段是否都已填写：
- [ ] name
- [ ] description
- [ ] tools（如果未填，说明理由）
- [ ] skills（引用了哪些 skill）
- [ ] model

**检查项 4.2**
阶段 3 中标记为「新建」的 skill，是否全部生成了对应的 SKILL.md 文件？
- 检查方法：对照阶段 3 的能力拆解表格，逐一核对

---

## 快速判断标准

当某个内容让你拿不准「放系统提示还是放 skill」时，用这个判断题：

> 如果把这段内容给一个完全不同类型的 agent（比如客服 agent）看，它对那个 agent 有用吗？

- 有用 → 放 skill（可复用操作知识）
- 没用 → 放系统提示（这类角色特有的判断）

当某个约束让你拿不准「是硬约束还是软约束」时，用这个判断题：

> 如果用户说「我完全理解风险，请直接执行」，这条约束还成立吗？

- 成立 → 硬约束
- 不成立 → 专业直觉（软约束），写入判断示例三元组而不是硬约束段
