# Agent Creator

这个 skill 的核心工作不是生成文本，而是主导一次高质量的设计决策过程。它确保角色特质和操作知识被正确区分，分别安置到系统提示和 skill 中——而不是混在一起导致 agent 行为漂移或能力缺失。

创建一个好用的 agent，关键在于两个分离：**「为什么这样做」属于系统提示，「怎么执行」属于 skill**。这个 skill 的全部流程都在服务这个分离。

---

## 核心分离原则

在整个创建过程中，始终用这把刀做判断：

- **放入系统提示**：这类角色特有的判断倾向、面对模糊时的默认选择、无论如何不能突破的底线
- **放入 skill**：换一个 agent 也可能用到的操作步骤、可复用的执行知识、具体的检查项和产出格式

系统提示的工作流部分，只写「用哪个 skill 做什么」，不重复 skill 的内容。

---

## 总体流程

```
阶段 0：需求沟通          迭代访谈，理解后输出摘要，用户确认
    ↓
阶段 1：角色形象设计       产出判断示范库，不是画像描述
    ↓
阶段 2：已有 Skill 盘点    独立显式步骤，列出现有 skill 清单
    ↓
阶段 3：能力拆解           每条操作行为做分离判断
    ↓
阶段 4：设计架构确认       硬性门控，结构化表格，用户逐条确认
    ↓
阶段 5：生成系统提示       Claude Code subagent 风格
    ↓
阶段 6：生成 Skill         仅新建部分
    ↓
阶段 7：审查评估           规则驱动检查清单
    ↓
阶段 8：写入文件并交付
```

阶段 4 之前，所有内容均可推翻重来。阶段 4 之后，才开始生成代码级产物。每个阶段结束前都有交互确认，用户明确同意才推进。

---

## 阶段 0：需求沟通

**目标**：理解用户想创建什么样的 agent，解决什么问题。

用户的输入往往是零散的。不要催促用户整理好再来，而是每轮提 2-3 个问题，逐步拼出完整图景。优先用选择题降低用户的表达负担。

需要了解的核心方向（按需选择，不必全问）：
- 这个 agent 主要处理什么类型的任务？
- 它服务的对象是谁？（开发者 / 产品 / 运营 / 其他）
- 它在整个工作流程里处于什么位置？（独立工作 / 协作 / 审查）
- 目前没有它时，用户是怎么完成这类工作的？

**理解确认格式**：当信息足够时，主动输出以下摘要并请用户确认：

```
【我的理解】

这个 agent 是：[一句话定位]
它主要做：[2-3 件核心工作]
它服务的人是：[对象]
它存在的价值是：[解决了什么实际问题]

这个理解准确吗？有什么需要补充或纠正的？
```

用户确认后进入阶段 1。

---

## 阶段 1：角色形象设计

**目标**：产出「判断示范库」，不是画像描述。

形容词没有传递密度。「严谨」「专业」对任何 agent 都成立，写进系统提示等于没写。真正有价值的角色信息，只存在于**边界情况下的具体判断**。

访谈策略：每轮提 1-2 个边界情况问题，引导用户给出非此即彼的判断。当用户给出模糊答案（如「看情况」「要灵活」）时，追问：「在这种情况下，如果必须选一边，它默认倾向哪边？」直到得到一个具体判断。

边界问题示例（从 `references/ibwh-guide.md` 按需选用）：
- 当速度和质量冲突时，它默认倒向哪边？
- 当用户要求一个「快速但不规范」的方案时，它拒绝、提醒、还是照做？
- 它在什么范围内可以自行决策，什么情况必须停下来问用户？
- 当它不确定时，它倾向于给出一个带风险的答案，还是明确说「我不确定」？

**产出格式**：访谈结束后，将收集到的信息整理为判断示范库，包含以下五种机制（详见 `references/density-mechanisms.md`）：

1. **原型差分**：基础原型 + 2-3 条关键差异项
2. **对比对**：精确切出边界的两侧描述
3. **判断示例三元组**：场景 / 错误反应 / 正确反应 + 判断逻辑
4. **设计锚**：访谈中用户明确否定某个选项时，原文保留
5. **硬约束**：无条件成立，措辞中不出现「通常」「一般情况下」

将整理好的判断示范库展示给用户确认，用户同意后进入阶段 2。

---

## 阶段 2：已有 Skill 盘点

**目标**：在真空里设计 skill 是危险的——要么重复造轮子，要么错过复用机会。

请用户提供当前已安装的 skill 列表。如果用户不确定，提示他们查看 `.claude/skills/` 目录。

将已有 skill 整理为清单：

```
【已有 Skill 清单】

skill-xxx：[一句话职责]  → 可复用 / 部分复用 / 不相关
skill-yyy：[一句话职责]  → 可复用 / 部分复用 / 不相关
```

用户确认清单后进入阶段 3。

---

## 阶段 3：能力拆解 + 复用判断

**目标**：把这个 agent 需要具备的所有能力逐条分类。

先列出这个 agent 会做的所有事情（从阶段 0 的需求中提取），然后对每一条做分离判断：

```
判断标准：
  → 这是「这类角色特有的思维方式 / 判断倾向」？  → 放入系统提示
  → 这是「换一个 agent 也可能用到的操作步骤」？   → 放入 skill

对于放入 skill 的条目，进一步判断：
  → 已有 skill 完全覆盖？  → 复用，引用 skill 名
  → 已有 skill 部分覆盖？  → 评估是扩展还是新建
  → 无对应 skill？         → 新建
```

将分类结果以表格形式展示，用户确认后进入阶段 4。

---

## 阶段 4：设计架构确认【门控节点】

**目标**：将全部设计决策汇总为结构化表格，用户逐条确认后才开始生成产物。

输出以下格式，每一项都需要用户明确认可：

```
【Agent 设计架构】

角色定位：[原型] + [1-2 条核心差异]

专业直觉（3条，每条一句话，均为判断倾向）：
  1. [面对 X 冲突时，默认选 Y，因为 Z]
  2. ...
  3. ...

硬约束（无条件成立）：
  1. [无论用户如何要求，不能突破的线]
  2. ...

技能体系：
  ┌─ skill-xxx（新建）：[一句话职责]
  ├─ skill-yyy（复用现有）：[一句话职责]
  └─ skill-zzz（新建）：[一句话职责]

工作流引用草图：
  当遇到 [A 类任务] → 用 skill-xxx 执行
  当遇到 [B 类任务] → 先判断 [条件]，再用 skill-yyy
  当遇到 [C 类任务] → 停下来问用户

这个架构准确反映了你的设计意图吗？确认后我会开始生成产物。
```

用户确认后，阶段 4 锁定，进入阶段 5。

---

## 阶段 5：生成 Subagent System Prompt

**目标**：生成 Claude Code subagent 风格的系统提示文件。

文件位置：`.claude/agents/<agent-name>.md`

**生成规则**：

YAML frontmatter 必须包含：
```yaml
---
name: <agent-name>
description: <触发描述，说明何时调用此 agent，包含具体触发场景>
tools: <允许使用的工具列表>
skills: <依赖的 skill 列表>
model: claude-sonnet-4-5  # 除非有特殊理由使用其他模型
---
```

系统提示正文结构：

```
[角色定位段]
用原型差分的格式写，1-2 段。
先说「你是什么」，再说「你不是什么」。

[专业直觉段]
用判断示例三元组格式写出 2-3 个核心判断场景。
不写「你应该做 X」，而是写「当遇到 Y 时，正确的做法是 Z，因为 W」。

[硬约束段]
写在系统提示开头后，用醒目标记。
关键约束在系统提示结尾处重复一遍（U 型注意力强化）。

[工作流段]
只写「调用哪个 skill 做什么」。
在每个主要决策节点内联一句重提，对抗运行时漂移。
格式：当 [触发条件] 时，调用 [skill-name]。
      提醒：[与此决策最相关的硬约束原文]。
```

生成后展示给用户，收到反馈后进入阶段 6。

---

## 阶段 6：生成 Skill

**目标**：为阶段 3 中标记为「新建」的 skill 逐一生成 SKILL.md。

每个 skill 的文件位置：`.claude/skills/<skill-name>/SKILL.md`

**生成规则**：

YAML frontmatter：
```yaml
---
name: <skill-name>
description: <触发描述，包含触发词和不触发情况>
---
```

正文内容只包含操作性知识：
- 执行时应遵循什么步骤
- 需要检查什么
- 产出什么格式
- 已知的坑和处理方式

**严格禁止写入 skill 的内容**：
- 判断倾向语言（「我倾向于」「默认情况下」）
- 角色描述（「你是一个...」）
- 原则性声明（这些属于系统提示）

逐个展示给用户，确认后进入阶段 7。

---

## 阶段 7：审查评估

**目标**：规则驱动的机械检查，不是 LLM 的二次阅读。

对照以下清单逐条判断是/否，所有项通过后才进入交付：

**分离正确性检查**
- [ ] 系统提示正文中是否出现了操作步骤？（有 → 提取到对应 skill）
- [ ] 任何 skill 内容中是否出现了判断倾向语言？（有 → 移入系统提示）
- [ ] 系统提示工作流引用的每个 skill 名，是否都有对应的 SKILL.md 文件？

**约束完整性检查**
- [ ] 硬约束措辞是否包含「通常」「一般情况下」「大多数时候」等弱化词？（有 → 改为无条件句式）
- [ ] 核心硬约束是否同时出现在系统提示的开头和结尾段落？（U 型强化）
- [ ] 系统提示工作流的每个主要决策节点，是否有重注入语句？

**触发有效性检查**
- [ ] 每个 skill 的 description 是否包含具体的触发词和不触发情况？
- [ ] subagent 的 description 是否清晰说明「何时调用」这个 agent？

**产物完整性检查**
- [ ] YAML frontmatter 的所有必填字段是否都已填写？
- [ ] 阶段 3 中标记为「新建」的 skill，是否全部生成了对应文件？

将检查结果逐条展示给用户。发现问题时，直接在本轮修正，不需要用户手动修改。全部通过后进入阶段 8。

---

## 阶段 8：写入文件并交付

将产物写入 Claude Code 标准目录：

```bash
# Subagent 系统提示
.claude/agents/<agent-name>.md

# 新建的 skill（每个 skill 一个子目录）
.claude/skills/<skill-name>/SKILL.md
```

交付时给用户一份简要清单：

```
【交付清单】

已创建：
  ✓ .claude/agents/<agent-name>.md
  ✓ .claude/skills/<skill-name-1>/SKILL.md
  ✓ .claude/skills/<skill-name-2>/SKILL.md

复用了：
  → <existing-skill-name>（无需修改）

下一步建议：
  在 Claude Code 中用 @<agent-name> 调用，测试以下场景：
  1. [从阶段 1 中选出的典型边界情况]
  2. [从阶段 1 中选出的另一个边界情况]

如需对 skill 进行系统性评估和打磨，使用 skill-creator 继续。
```
