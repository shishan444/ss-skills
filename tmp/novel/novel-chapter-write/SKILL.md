# 小说单章写作执行器

这是整个创作流程中被执行最多的环节。每一章的质量决定了全书的成败。

**核心原则：写好一章。不管是单章精确写作还是多章串行批量写作，每一章的质量决定全书的成败。批量写作时，"串行"意味着一章接一章、串行推进，但每章仍然是独立的精确写作单元——不是流水线。**

## 职责边界

- 负责：把已经确认的剧情节点写成章节正文，并在写完后自检修正。
- 不负责：重新选题、重写世界观或大规模改大纲。发现节点不足或状态冲突时，先做节点覆盖审计；确实需要推演时交回 `novel-worldbuilding`。
- 不负责：阶段质量总审计。阶段写完后交给 `novel-stage-review`。
- 不负责：最终标点、合并和目录整理。全书完成且P0/P1清零后交给 `novel-polish`。

如果用户只要求“写”，不要先停下来做长篇方案汇报。读取必要上下文后直接进入章节执行；只有遇到时间线越界、节点重复、设定冲突这类会导致正文作废的问题，才暂停说明。

## 输入

- 本章在大纲中的剧情节点（来自outline.md）
- **architecture-status.md**（如存在）— 故事引擎/设定状态/角色状态/断裂点/伏笔清单/情绪曲线。比outline更详细的作战地图。
- **characters.md + character-ledger.md** — 角色稳定内核、语言动作识别、关系压力、当前状态账本。若旧项目缺 `character-ledger.md`，先用 characters.md + 最近2-3章生成最小账本，再写正文；不要静默跳过。
- **foreshadow-tracker.md**（如存在）— 伏笔埋设/回收状态跟踪。确认本章该推进或回收哪些伏笔。
- 上一章的结尾（确认衔接）
- 当前时间线/人物状态/空间状态
- 写作设定（tone-guide.md + emotion-style-map.md + writing-guide.md）
- **front-30-hook-map.md**（如存在）— 前三章冷启动和前30章追读钩子链。写Ch1-30时必读；缺失则回到 `novel-worldbuilding` 补齐，不要静默跳过。
- **已完成章节的正文风格**（续写时必读——匹配已有散文的句式/标点/标志性手法）
- **写作经验库专题**（先读 `../写作经验库/index.md` 定位，再按本章格调和情绪路由加载1-3个相关专题；路由规则见 `references/craft-experience-routing.md`）

## 输出

chapter-{NN}.md（单章正文）

如果写作过程中发现 `emotion-style-map.md` 缺失，正文仍可按临时最小路由完成，但最终任务反馈必须单独标记：

```markdown
情绪路由风险：
- emotion-style-map.md 缺失，本章使用临时路由。
- 建议回到 novel-worldbuilding 补齐 emotion-style-map.md，避免后续章节长期依赖临时判断。
```

**文件位置**：章节正文通常在 `drafts/` 子目录下（如 `stories/XX/drafts/chapter-{NN}.md`），不是故事根目录。如果根目录找不到，用 `rg --files` / `find` / `search_files` 搜 `chapter-1*.md` 定位实际路径。

## 执行步骤

### 第零步：生成情绪路由卡（强制）

先读本章在 outline.md / architecture-status.md 中的剧情功能、读者期待、目标体感、呈现模式、爽点阶段和章长策略。若项目存在 emotion-style-map.md，必须读取；若缺失，则按 `../创作方法论/emotion-style-routing.md` 临时生成最小路由，并在最终任务反馈中标记“emotion-style-map.md 缺失”。临时路由只能作为本章兜底，不能替代 `novel-worldbuilding` 的正式情绪路由表。

写正文前必须形成这张作战卡，不写入正文文件：

```markdown
本章情绪路由卡：
- 剧情功能：
- 读者当前期待：
- 目标体感：
- 呈现模式：
- 场景类型：
- 爽点阶段：
- 设定入场方式：本章设定如何在行动/冲突/收益中出现
- 追读钩子：章内即时收益/冲突 + 章末下一章牵引
- 章长策略：完整展开/中等推进/压缩过渡
- 本章文字策略：句式、段落、感官、独白、对话、标点如何调整
- 禁止写法：
- 写完验证：
```

判断原则：
- 稳定的是 tone-guide 的叙述声音，不是一成不变的情绪。
- 轻松、搞笑、烟火气、严肃、压抑、期待铺垫、爽点兑换、余波等模式必须跟剧情功能匹配。
- 关键爽点、关系定调、死亡告别、重大反转必须完整展开；低密度过渡必须压缩，不能为了字数注水。
- 如果大纲只给了“爽/紧张/温暖”标签，要把它翻译成可执行的呈现模式后再写。
- Ch1-30必须额外检查 front-30-hook-map.md：本章属于哪个短钩子/中钩子/长钩子；章末必须有明确的下一章牵引。前三章必须完成“设定入场→限制加压→小爽点兑换”。

### 第一步：加载经验库（强制！不能跳过！）

先读 `../写作经验库/index.md`，再根据本章的剧情类型、格调和情绪路由卡，加载对应的写作经验库专题。不要一次性吞完整经验库；每章只读最相关的1-3个文件，把经验转成可执行约束。

| 本章涉及 | 必须加载的经验库文件 |
|---------|-------------------|
| 搞笑/荒诞场景 | ../写作经验库/专题/writing-exp-1-1.md |
| 反差幽默/认知错位 | ../写作经验库/专题/writing-exp-1-2.md |
| 紧张/悬疑/危机 | ../写作经验库/专题/writing-exp-1-3.md |
| 温情/日常/烟火气 | ../写作经验库/专题/writing-exp-1-4.md |
| 商战/博弈/谈判 | ../写作经验库/专题/writing-exp-1-5.md |
| 爽点/赚钱/碾压 | ../写作经验库/专题/writing-exp-1-6.md |
| 群像/配角互动 | ../写作经验库/专题/writing-exp-2-1.md |
| 配角弧线推进 | ../写作经验库/专题/writing-exp-2-2.md |
| 反派/对手戏 | ../写作经验库/专题/writing-exp-2-3.md |
| 女性角色互动 | ../写作经验库/专题/writing-exp-2-4.md |
| 权力/认可/地位 | ../写作经验库/专题/writing-exp-3-2.md |
| 社会认知融入 | ../写作经验库/专题/writing-exp-4.md |

**加载后必须阅读"创作指引"层（老师傅口诀），不是只看概念框架。**

加载后按 `references/craft-experience-routing.md` 生成本章经验摘要：本章主要风险、已加载经验、3-5条执行约束、写完验证点。摘要只作为作战笔记，不写入正文。

### 第二步：确认本章定位

读大纲确认：
- 本章对应哪个剧情节点？
- 本章在时间线上的什么位置？
- 本章涉及哪些角色？他们的当前状态？
- 本章的情绪路由是什么？（剧情功能/读者期待/目标体感/呈现模式/爽点阶段/章长策略）

### 第三步：角色取样（防脸谱化）

若本章有2个以上角色出场，必须读取 characters.md/character-ledger.md 中相关角色条目，做极短取样。若 ledger 缺失，先补一个最小账本，覆盖本章出场角色的当前处境、核心利益线、最近关键选择、关系压力边和下一步最优选择：

```markdown
本章角色取样：
- [角色A] 本章想要什么？隐藏什么？说话识别点是什么？会因哪个缺点做错判断？
- [角色B] 他和A/主角的利益冲突在哪里？他会怎么误判别人？他不能说出口的私心是什么？
```

如果所有出场角色的目标都只是"帮助主角推进剧情"，先修本章设计，再写正文。角色必须带着自己的欲望、误判、私心和语言节奏进入场景。

### 第四步：读上一章结尾

确认衔接点——上一章的最后一个场景是什么？最后一个动作/对话/内心独白是什么？本章必须从那里接上，不能断裂。

### 第五步：写作（遵循六条铁律）

**铁律零：叙述声音稳定，情绪呈现动态**

tone-guide 决定这本书“谁在讲、怎么听起来像同一本书”；情绪路由卡决定这一章“要让读者感到什么”。不能把全书写成单一情绪，也不能让每章像不同作者。

- 轻松/搞笑章：反应快、认知错位清楚、笑点短促，不靠角色犯蠢。
- 烟火气/温情章：用物件、饭菜、声音、称呼、手上动作承载情绪，不抽象煽情。
- 严肃/压抑章：让选择变窄、声音减少、身体反应具体，不用破折号碎片假装沉重。
- 危机/博弈章：信息收紧、对话有假话和底牌、每个人都有不能退的线。
- 期待/爽点兑换章：先回收铺垫和误判，再写释放瞬间、见证者反应和余波，不能只写结果。
- 过渡章：压缩低密度过程，只交代状态变化和下一步钩子。

**铁律一：决策过程必须完整展示，绝不写"他决定了"**

主角的每一个选择都是戏。完整决策链：
1. 触发：什么让主角面临选择？
2. 本能反应：第一反应是什么？（通常是逃避/走捷径/装看不见）
3. 内心拉锯：为什么想做？为什么不敢做？
4. 身体先于意识：身体做了什么？（意识还没决定，手已经动了）
5. 决定后的反噬：决定之后是什么感觉？（后悔/不安/隐秘的满足）

**铁律二：氛围靠感官细节，绝不写"气氛很压抑"**

用具体的、可感的、多感官的细节建立氛围：
- 视觉：光线/颜色/形状/动作
- 听觉：声音/噪音/沉默/节奏
- 触觉：温度/质感/疼痛/风
- 嗅觉：气味/空气/食物/脏
- 味觉：（如适用）

抽象形容词（压抑/恐怖/温暖/孤独）一律删除，替换成具体细节。

**铁律三：内心独白占比30-50%**

主角的内心吐槽/感知/拉锯是全书的灵魂。嘴上说的和脑子里想的必须不一样。

**铁律四：搞笑来自认知错位，不来自角色犯蠢**

如果格调包含搞笑：
- 笑点来自2025年认知×1979年现实的碰撞（或任何两个世界的认知差）
- 不是让角色做蠢事来制造笑点
- 笑点用极短句收尾
- 每次碰撞都有标志性反应（如"好家伙"）

**铁律五：角色必须差异化，绝不脸谱化**

每个角色说话的方式/思考的角度/行为的模式必须不同。
- 角色A自私但讲义气 → 他的对话和决策必须体现这个矛盾
- 角色B精明但心软 → 他的行为必须有这个反差
- 角色C老实但不蠢 → 他不能被写成"老好人"
- **绝对不能出现"全员好人/全员睿智"——人性的自私、矛盾、灰色地带必须体现**
- 每个角色至少有2个人格缺点和自己的私心

**铁律六：物价/社会细节必须查设定库（era-database.md）**

如果小说涉及年代文/穿越/双穿，写作时涉及物价/商品/社会制度/通讯方式/交通方式等时代细节时，必须查era-database.md确认准确性。不能靠想象编价格或编社会规则。

特别是涉及2025年现代线时——2025年的大学生不用网购/不刷短视频/不用微信是严重不合理。主角在现代的生活方式必须真实。

### 第六步：写完后自检

写完每一章后必须检查：

- [ ] 衔接前一章（无断裂）
- [ ] 已生成并执行本章情绪路由卡（目标体感/呈现模式/章长策略清楚）
- [ ] 稳定叙述声音没有漂移，动态情绪呈现跟本章剧情功能匹配
- [ ] 决策过程完整展示（没有"他决定了"）
- [ ] 氛围有感官细节（没有抽象形容词）
- [ ] 内心独白占比30-50%
- [ ] 标志性手法符合本章情绪（该有时有，不该有时不硬塞）
- [ ] 如果本章是爽点兑换，是否有铺垫回收、释放瞬间、见证者反应和余波？
- [ ] 如果本章是烟火气/温情，是否用具体生活细节承载关系变化？
- [ ] 如果本章是压迫/危机，是否让选择变窄并写出具体代价？
- [ ] 所有角色智商在线（没有人犯蠢）
- [ ] 无元引用（Ch编号/concept/outline/设定文件名不出现在正文中）
- [ ] 物品合理性（如有倒卖/搬运——物品符合时代合理性）
- [ ] 数字精确（物价/时间/距离/温度）
- [ ] 本章出现的角色是否保持了性格区分度？
- [ ] 有没有角色表现得像"好人俱乐部"成员（全员睿智/全员大爱）？
- [ ] 角色的缺点/私心/矛盾是否体现了？
- [ ] 本章是否至少有一个角色做了不完全服务主角利益的选择？
- [ ] 如果遮住角色名，只看台词/动作，主要角色能否被分辨？
- [ ] 本章是否把写作经验摘要转化成了场景、冲突、对话或选择，而不是只在脑中“参考”？
- [ ] 涉及物价/商品/社会细节时是否查了era-database.md？
- [ ] 现代线的行为是否符合2025年真实状态（网购/微信/短视频等）？
- [ ] 本章推进了剧情（不是水章——读完后故事往前走了一步）
- [ ] 若本章在Ch1-30：本章是否兑现 front-30-hook-map.md 的短/中/长钩子？
- [ ] 若本章在Ch1-3：是否完成冷启动对应职责（Ch1设定入场、Ch2限制加压、Ch3小爽点兑换）？
- [ ] 设定是否在本章行动/冲突/收益中产生效果，而不是只被解释或渲染？
- [ ] 章末是否留下具体下一章牵引，而不是只有氛围或情绪收束？

## 串行批量写作（单agent多章连续写作）

当用户要求一次性写多章（如"写Ch107-160"），工作流与单章不同。核心原则不变（六条铁律、经验库加载），但执行策略调整：

### 批量写作第零步：上下文预加载（一次性，不重复）

开始写之前，一次性读取全部上下文文件：

1. **outline.md** — 确认本章范围对应的剧情节点（如N01-N51）
2. **architecture-status.md** — 如果存在，必读。包含：故事引擎状态、设定资产清单、断裂点与风险、角色当前状态、未回收伏笔清单、情绪曲线位置。这是比outline更重要的作战地图。
3. **foreshadow-tracker.md** — 如果存在，必读。跟踪每个伏笔的埋设/回收状态和计划回收章节。批量写作时必须确认每章该回收/推进哪些伏笔。
4. **tone-guide.md + emotion-style-map.md + writing-guide.md** — 写作设定、动态情绪呈现路由和经验执行规则
5. **front-30-hook-map.md** — 如果本批涉及Ch1-30，必读。确认每章短钩子、中钩子、长钩子和设定入场方式。
6. **characters.md + character-ledger.md** — 必读！确认本批出场角色的稳定内核、当前处境、关系压力和偏离警报。批量写作最容易把配角写成同一种“理性好人”，必须用角色账本纠偏。
7. **最后2-3章已完成的正文** — 必读！吸收已有的散文风格（句式节奏、标点习惯、标志性手法）。批量续写时，风格一致性比单章完美更重要。
   - ⚠️ **退化检测检查点（必做）**：读完后立刻判断——这些章节是否退化？指标：①"好家伙"≥5次/章 ②大量单句成段+词间"——" ③内心独白全是喃喃自语无反差 ④章节字节量骤降（<5KB） ⑤搞笑元素归零。如果命中≥2条——**不要吸收这些章节的风格！** 立即回退到全书早期章节（Ch1-3或act-review中被赞扬的章节）作为风格基准。退化章节是\"重写的对象\"，不是\"续写的参考\"。
8. **情绪路由总表** — 从 outline/architecture-status/emotion-style-map 中抽取本批每章的剧情功能、读者期待、目标体感、呈现模式、爽点阶段和章长策略。批量写作时每章开写前仍要生成该章的情绪路由卡。
9. **经验库专题** — 根据批量范围内的主要格调和情绪路由，一次加载多个经验库文件（如温情+商战+女性角色+紧张），不要一章一章加载。按 `references/craft-experience-routing.md` 把经验转成批量执行约束。
10. **architecture-status.md 的 P0 断裂点** — 如果存在，检查是否有"文风退化"或"情绪兑现失败"相关的P0标记。有则意味着前序章节确实退化，必须用早期章节做基准（与第7步的退化检测联动）。
11. **⚠️ 节点覆盖审计（CRITICAL — 不做这步会写重复内容！）**：大纲的节点→章节映射（如"N162-N184在Ch242-280"）是**规划时的预估**，不是进度跟踪。实际写作中，前序session可能已经把远超映射的节点写完了。**必须做的事**：
   - 读取用户指定的起始章（如Ch242）之前的**最近5-8章正文**（不是2-3章——2-3章只够看风格，5-8章才能看到剧情进展到哪里了）
   - 对照outline逐一标记：哪些节点已经写了？写到哪了？时间线推进到了哪一年？
   - 用可用搜索工具搜关键词（如角色名/事件名/年份）快速确认关键节点是否已被覆盖；Codex 环境优先用 `rg -n` / `rg -c`，有 `search_files` 时再用 `search_files`
   - **输出一个"实际剩余未写节点"清单**——这才是你真正要写的东西。用户说的"写N162-N184"可能实际上只剩N172+N178-N179+N181-N184需要写
   - 如果发现大部分节点已写完：**立即调整策略**——基于剩余节点重新规划章节数和三幕结构。不要盲目按原指令写39章——可能20章就够了，或者需要用outline中标注但前序session跳过的"间隙剧情"来填充（如角色coda、世界观回响、烟火气日常）

### 批量写作中的逐章执行

每章仍然遵循"确认定位→读前章结尾→写作→自检"的核心流程，但：

- **读前章结尾**：上一章刚写完就在同一个session里，不需要重新读文件——你的上下文里已经有。
- **情绪路由逐章执行**：不要因为批量写作就只按一个总格调写到底。每章必须根据本章剧情功能选择呈现模式：该压迫就压迫，该爽点兑换就给足篇幅，该过渡就压缩。
- **前30章追读逐章执行**：Ch1-30每章必须明确“本章即时收益/冲突”和“章末下一章牵引”。设定必须在行动中产生效果，不能只作为氛围或说明。
- **节点映射灵活**：有些剧情节点对应完整一章，有些半个节点就够，有些两个节点合一章。由剧情密度决定，不强求1:1。
- **章长自然变化**：批量写作的后期章节通常比前期短——因为大纲节点在收敛，角色弧线在汇合，有些场景只需要一个片段而非完整章节。不要为了字数注水。
- **伏笔跟踪**：每写完一章，在心里（或笔记里）标记推进了哪些伏笔。确认没有遗漏outline中标注的关键伏笔回收节点。

### 超大批量写作（用户请求100+章 / 节点数远超可写章数）

当用户请求的章节数远超单次session可完成的量（如"写Ch161-280共120章"），且对应的outline节点数量庞大（如68个节点）时，必须采用**节点压缩策略**：

**核心原则：覆盖所有节点 > 严格1:1节点映射。** 用户要的是"故事推进到N119"，不是"恰好120章"。如果40章能覆盖全部68个节点且质量达标，比120章注水章节有价值得多。

**节点压缩的具体策略：**

1. **按密度分组**：将节点按剧情密度分为三档：
   - **高密度节点**（伏笔回收/高潮/转折/关键角色弧线）→ 独立成章，完整展开。如F001核心回收、大低谷→大翻盘、感情线定调。
   - **中密度节点**（日常推进/配角互动/信息传递）→ 2-3个节点合并为一章，用过渡段连接。
   - **低密度节点**（背景铺垫/时间过渡/状态更新）→ 压缩为半章甚至几段，嵌入其他章节的开头或结尾。

2. **三幕结构压缩**：将全部节点分为三幕（对应outline的子阶段），每幕约占总章节的1/3。幕与幕之间用"标志性章节"做分界（如年终盘点/大事件转折）。

3. **每幕的节奏**：开头2-3章正常密度（建立场景/衔接前章），中间章节加速压缩（多节点合并），结尾1-2章回到正常密度（本幕的高潮/收束）。

4. **关键章节不可压缩**：以下类型的章节必须独立成章、完整展开：
   - 伏笔回收高潮章（如F001核心回收 = 老张看到自己的记忆画面）
   - 大低谷→大翻盘（如两界同时收紧→转型）
   - 感情线定调章（如沈父深谈/沈知秋做选择）
   - 存在主义/主题升华章（如蝴蝶效应自我认知危机→救赎）

5. **进度报告**：每写完10-15章，向用户报告：已覆盖的节点清单、未覆盖的节点、当前时间线位置。如果session接近工具调用上限，提前总结已完成和未完成的部分。

**超大批量写作的自检重点：**
- 每个幕结尾做一次全盘检查（不是每5-8章），确认：本幕覆盖了哪些节点？遗漏了哪些？伏笔是否按计划推进？
- 关键章节（高潮/回收/转折）仍然逐章自检。
- 时间线边界扫描更频繁——超大批量写作容易在中后期"滑出"用户给的硬约束。

### 批量写作自检频率

单章模式要求每章写完都自检。批量模式下改为：

- **每5-8章做一次批次检查**：回头扫一遍这批章节，确认衔接无断裂、伏笔按计划推进、角色状态一致、时间线没有跳。
- **每5-8章更新一次角色账本**：记录出场角色的关系变化、利益变化、最近关键选择和偏离警报。不要等阶段 review 才发现人物已经偏了。
- **关键章节（伏笔回收章/高潮章/转折章）仍然逐章自检**。
- **批次结束时做一次全盘检查**：覆盖的节点清单、未写到的节点、伏笔回收状态。用可用搜索工具批量验证；Codex 环境优先用 `rg -n` / `rg -c`：
  - **元引用扫描**：搜 `outline|concept|worldbuilding|设定|第.阶段|chapter-[0-9]|foreshadow` — 命中即为违规
  - **时间线边界扫描**：搜 `^198[0-9]年|^202[0-9]年` 确认年份不超出用户给的硬约束
  - **标志性手法密度**：统计标志性短语（如"好家伙"）出现次数。长期归零说明格调可能丢失；平均每章>5次且无新信息，通常是机械填充式退化。关键看它是否在制造新的认知错位，而不是只看数字。

### 批量写作的陷阱（Pitfalls）

1. **散文风格漂移**：写到第20章时，可能已经忘了第1章的"——"风格。解决：每10章回去翻一眼早期章节的格式。**这是最常见的批量质量问题。**
2. **内心独白稀释**：批量写作容易越写越"快"，跳过内心独白直接写事件。但内心独白是全书的灵魂——后期章节也不能低于30%。
3. **伏笔遗忘**：architecture-status.md里标注的"急等回收"伏笔，在批量写作中容易被后续剧情冲走。解决：每章开写前快速扫一眼foreshadow-tracker.md。
4. **绝对边界遗忘**：用户给的硬约束（"只写到1985夏"、"不能写结局"）在写到第30章时容易忘记。解决：把约束写在每章的开头注释里或持续默念。
5. **角色智商下降**：批量写到后期，配角容易变成工具人。每个配角出场时提醒自己——这个人有自己的利益线和判断逻辑。
6. **关系回到原点**：角色经历重大事件后，下一章关系仍像没发生过一样。解决：每批更新 character-ledger.md 的关系变化和下一次必然摩擦。

## 批量重写（修正文风退化）

当用户要求"重写Ch107-160"或"修正文风退化"时，工作流与批量新写完全不同。核心区别：不是从大纲生成新章节，而是**从已退化的章节中提取剧情事件，然后用正确的文风重新执行文字**。

**详细诊断清单、风格基准对比、验证命令见 `references/batch-rewrite-protocol.md`。**

### 第零步：风格基准校准（一次性）

批量新写时读"最后2-3章已完成正文"吸收已有风格。但批量重写时——**最后几章恰恰是退化的**。读它们只会吸收坏习惯。必须读**退化之前的早期章节**作为风格基准：

1. **Ch1-3**：全书最初的文风。吸收句式节奏、标点习惯、标志性手法、内心独白密度
2. **act-review 中被赞扬的章节**：这些是风格巅峰期的代表作
3. **用户指定的基准章节**：如"先读Ch47/Ch55作为文风基准"——用户比review更知道哪些章节是"对的"
4. **双重基准校准（推荐）**：读退化范围**两侧**的好章节（前一阶段末尾 + 后一阶段开头），形成"夹击"基准。两个基准的共性（正常段落长度、标点习惯、内心独白密度）比单一基准更可信，且能防止你被退化章节"同化"。例如重写Ch61-106时读Ch60（前一阶段末尾）+Ch107（已修正的后一阶段开头）。

### 重写工作流（逐章）

每章遵循以下流程：

1. **读取退化章节**：只提取剧情事件（发生了什么、谁说了什么、角色状态变化、伏笔推进）。**完全忽略文字执行**——那些碎片化断句、破折号、碎片化独白不是"风格"，是"退化"，不是重写的参考而是重写的对象
2. **写作**：用风格基准的句式节奏完全重新写。剧情事件保留，文字全部重写。不是"编辑"或"润色"——是**从零重写**
3. **自检**（见下方重写专用自检）

### 重写专用自检（比新写更严格）

除第六步的常规自检外，批量重写必须额外检查：

- [ ] **碎片化断奏体检查**：连续3段不应出现词与词之间的"——"（破折号只用于正常的语句内补充说明或场景分隔）。注意碎片化有多种形态：①**词间"——"**（最严重）②**单句成段泛滥**（每句话单独一段，无段落整合）③**句内"——"过度**（一句话内插入多个"——"做停顿）。三种都需消除。但正常文学用法的"——"（对话引入、句内补充、场景分隔线）应保留——目标是恢复正常散文节奏，不是消灭破折号。
- [ ] **标志性短语密度**：如"好家伙"每章≤1-2次（退化版本可能5-20次/章）
- [ ] **内心独白质量**：不是纯喃喃自语，恢复"嘴上一套心里一套"的反差（退化版本内心独白膨胀到70-80%但全是自言自语）
- [ ] **认知错位搞笑**：两个世界的认知碰撞笑点是否恢复（退化版本搞笑元素归零）
- [ ] **公式化收尾**：检查是否每章都以"好家伙。/一步一步。"结尾——变换收尾方式
- [ ] **笔记本反思**：每3-4章最多1次，不是每章都有（退化版本每章结尾都写笔记本总结）
- [ ] **空壳章节**：如果原章节字节量骤降（如从8KB降到3-4KB），重写后必须恢复到正常篇幅（6-10KB），增加场景细节和内心独白

### 批量重写的token管理

大范围批量重写（如50+章）无法在一次session中完成。必须：
- 分批写入（每5-8章一批），避免单次输出token超限
- 用户提示"分步写入避免token超限"时——严格遵守，每批完成后报告进度
- 如果session被截断，总结已完成章节和未完成章节，方便下次继续

### 何时使用批量重写 vs 让用户从下一章开始改

- **批量重写**：退化覆盖一个连续区间（如Ch107-160），且用户明确要求"保留剧情事件，完全重写文字执行"
- **从下一章开始改**：退化是渐进的，review建议"下一阶段强制回归正常散文风格"——此时不需要回头改已写章节，只需要下一阶段注意

## 如果在批量写作中执行（dispatch给subagent）

dispatch时必须在context中明确：
1. 本章的剧情节点
2. 上一章的结尾内容（贴最后一段）
3. 当前状态（时间线/空间大小/人物关系状态）
4. 必须加载的经验库文件路径
5. 本章出场角色的角色取样（来自 characters.md/character-ledger.md）
6. 本章情绪路由卡（剧情功能/读者期待/目标体感/呈现模式/爽点阶段/章长策略）
7. 本章经验路由摘要（来自 `references/craft-experience-routing.md`）
8. 本章字数由剧情需要决定——不设上限下限；关键情绪兑换章必须完整展开，低密度过渡章必须压缩

**subagent每写完一章后必须执行第六步自检，自检不通过的章节当场修。**

## Pitfalls（实战教训）

### Pitfall 1：元引用侵入正文（每部小说都犯过）

问题：subagent在正文中写出"他在concept.md里读到过""在worldbuilding里看到过""这个设定"等元引用。这破坏了第四面墙。

**常见元引用向量**（写完用可用搜索工具逐一排查）：
- **文件名**：`concept`、`outline`、`worldbuilding`、`设定里`、`tone-guide`、`writing-guide`
- **大纲阶段名称**：如"第五阶段""第三阶段前奏""第N幕"——这些来自outline的结构标签，绝不能出现在正文末尾作为"阶段性收束"的标注
- **伏笔编号**：如"F003回收了""N44节点"——大纲/foreshadow-tracker的内部编号
- **章节自指**：如"第一百四十八章"出现在正文中间（标题除外），或"下一章"。**变体**：使用缩写形式如"林北在Ch101说过"——这种更隐蔽，grep用 `Ch[0-9]+` 搜

根因：subagent读了设定文件后，设定文件的概念"渗透"进了叙事。批量写作末期特别容易在大纲节点交汇处留下阶段标签（如"（第一阶段·完）"）。

修复：
- 写完每章后用 `rg -n`、脚本或 `search_files` 搜"concept""worldbuilding""outline""设定里""这个设定"等关键词
- 发现即替换为正文内合理的来源（"他在某本商业书里读到过"/"他这些年摸出来的"）
- 这不是个例——第8/9/10部小说都犯过这个错，必须在自检环节强制检查
- **编辑性脚手架泄漏（变种）**：agent在写章节文件时，偶尔会在标题下面插入内部编辑注释（如 `> 注：以下为正文，承接前章` 或 `> 本卷章节按时间线连续排列`）。这些不是小说内容——它们是agent的内部"脚手架"泄漏到了文件里。写完后检查每章的前5行——如果标题下面有 `>` 引用块或"注：""说明："等编辑标记，立即删除。标题（`# 第N章 标题`）后面应该直接是空行+正文。

### Pitfall 2：时代不合物品（致命逻辑bug）

问题：带去过去/异世界的东西明显不属于那个时代。例如方便面在1979年完全不存在（康师傅1992年才进大陆），带过去立刻露馅。

规则：带入过去时代的物品必须经过严格筛选：
- 绝对不能带：任何带现代品牌logo/条形码/二维码/生产日期的包装、电子设备、一次性塑料制品、现代食品
- 处理后可以带：药品拆掉包装换装、布料剪掉标签、金属零件磨掉标识、种子（外观差异不大）
- 每次倒卖什么、怎么处理"时代痕迹"本身就是策略博弈和看点

### Pitfall 3：文风退化到极端——不是"漂移"而是"崩溃"

Pitfall 1 的"散文风格漂移"指的是轻度偏差（忘了一开始的标点习惯等）。但实战中发现一种更严重的退化：**碎片化断奏体崩溃**——几乎每个词/短语之间都插入"——"，配合大量单句成段、空行和标志性短语（如"好家伙"）机械填充，章节字节数从正常8KB骤降到3-4KB。

这种退化的特征：
- "好家伙"从标志性口头禅（正常0.2-3次/章）变成机械填充（8-20次/章）
- 内心独白膨胀到70-80%但全是喃喃自语，丧失"嘴上一套心里一套"的反差质量
- 搞笑元素归零——认知错位笑点完全消失
- 公式化收尾——25+章以相同模式结尾
- 信息密度极低——碎片化排版占满了篇幅，实际剧情内容只有字节数的1/3

**批量新写时的预防**已经不够了——Pitfall 1 的"每10章翻一眼早期章节"无法阻止这种崩溃，因为一旦开始碎片化就会自我强化（越写越碎越容易继续碎）。

**修复**：一旦发现这种崩溃（通过review或用户反馈），必须执行**批量重写**（见上方"批量重写"章节），而非"下一阶段注意"。

**关键教训**：碎片化断奏体本质上是在"讲述"而非"展示"——大量内心独白碎片化铺陈，读者看到的是作者在解释角色在想什么，而非通过场景自己感受。这与铁律一（决策过程必须完整展示）、铁律三（内心独白30-50%）直接冲突——碎片化让独白占比膨胀到70-80%但信息密度反而下降。

### Pitfall 4：经验库"被提到但没被读"

问题：dispatch context里写了"参考writing-guide.md和写作经验库"，但subagent实际上没有真正深入读经验库的专题文件。结果是后续章节退化成"叙述事件"而不是"写小说"。

修复：
- dispatch时必须用skill_view(name='novel-chapter-write')让subagent加载SKILL
- SKILL中强制要求"加载后必须阅读创作指引层（老师傅口诀），不是只看概念框架"
- 在自检环节检查"内心独白占比是否30-50%""决策过程是否完整展示"——这些是经验库的核心指引，不读就不会做
- 每章开写前输出经验路由摘要；如果没有摘要，等同于没把经验库转化为执行约束

### Pitfall 5：subagent的"叙事完成驱动力"——忽略边界一口气写到结局

问题：给subagent的指令是"写Ch61-140（第二阶段，1981-1983）"，subagent却写了Ch61-140并一口气把整部小说从1981写到2031年+主角老死+结局+传承。这不是"忘记边界"——这是subagent被"完成叙事弧线"的驱动力压过了边界指令。

根因分析：
1. LLM天生倾向于"把故事讲完"——给了大纲全览后，它会想走到终点
2. "只写一个阶段"跟"完成故事"在LLM的注意力里是冲突的——前者是约束，后者是本能
3. 给了subagent全部6个阶段的outline后，它看到了"终局长什么样"，就想直接写过去

预防：
- **dispatch时只给本阶段的剧情节点**——不要给后续阶段的outline内容。subagent看不到"终局"就不会想写过去
- **在指令中重复边界至少3次**：开头一次、中间一次、结尾一次。"绝对不能写结局""时间线不能超过1983年底""只写Ch61-140"
- **用"留白"替代"禁止"**：不要说"不能写到1984年"，而是说"Ch140的最后一句应该停在1983年底的某个场景，后面的内容留给第三阶段"。给subagent一个明确的"停在哪里"的锚点
- **如果subagent的输出有"完结感"（主角老死/传承/终章标题），立即判定为违规**——不需要review，直接删除重写

### Pitfall 6：Session工具调用上限——超大批量写作的实际容量天花板

问题：用户请求"写Ch201-280共80章"，session在写完约40章后达到工具调用上限（tool-call iteration limit），被迫中断。剩余章节无法在同一session中完成。

根因：单章写作通常需要1次文件写入 + 预加载阶段需要多次读取/搜索。80章 × 1次/章 = 80次写入调用 + 预加载约10次 + 上下文加载约5次 = 约95次工具调用。加上自检扫描，总调用次数可达100+次，远超单session上限。

**实际容量经验值：单session约可完成25-35章高质量新写章节（视章节长度和预加载量而定）。批量重写容量更高——单章只需1次read+1次write，无outline/architecture咨询开销，实测单session可完成40-46章重写。超过46章建议分session。**

预防与处理策略：

1. **事前预估**：用户请求N章时，如果N > 30，立即告知用户"本session预计完成约25-30章，剩余章节需要下个session继续"。让用户有预期。

2. **分批写入策略**：
   - 将N章按outline的子阶段分为2-3批
   - 每批结束时（或接近上限时）向用户报告：已完成章节列表、最后覆盖的节点编号、当前时间线位置、未完成章节对应的大纲节点
   - 下一个session从"读取最后完成的章节结尾"开始续写

3. **跨session续写的关键动作**：
   - 读取最后完成的1-2章正文——确认衔接点和散文风格
   - 读取outline中剩余未写的节点
   - 读取architecture-status.md确认角色/设定当前状态
   - 读取foreshadow-tracker.md确认哪些伏笔已回收、哪些还未回收
   - 从上一session的最后节点继续，不需要重新推演

4. **节点压缩与session上限的关系**：超大批量写作的节点压缩策略（见上方"超大批量写作"节）本身就是应对session上限的方法——压缩后40章覆盖80个节点，比80章每个注水更有价值。但即使压缩后，如果压缩章数仍>30，仍需分session。

5. **不要牺牲质量换取章数**：如果接近session上限——宁可少写几章但保持质量，不要为了多写而降低章节篇幅或跳过自检。用户要的是高质量章节，不是数量。

### Pitfall 7：结局阶段写作的特殊陷阱——加速冲动与终极意象

**这是写全书最后一阶段（结局/最终阶段）时特有的问题。**

问题：当写到小说的最后几十章（如全书280章的最后40章），agent会被一种强烈的"收束冲动"驱使——急于把所有线收完、把所有人安排好、把故事讲完。这导致：

1. **加速陷阱**：结局章节越写越短、越写越快。重要场景（死亡/告别/传承/终章）被压缩成几段概述，而不是完整的场景。**结局应该比正文更慢，不是更快。** 老张的死、沈父的完全信任、空间的最终告别——这些场景需要比日常章节更多的篇幅和更慢的节奏。

2. **终极意象草率**：全书的最后一段（如"桃树在冻土下面根在生长……甜的"）是整部小说的回响。它不是随手写的——它要呼应第一章到最后一章的所有核心意象（桃子/根/甜/好）。**终极意象必须在推演阶段就设计好**，然后在写作时精确执行。

3. **角色收束不均**：结局阶段需要给每个重要角色一个"最终位置"（归位）。但agent倾向于只给主角和高潮角色篇幅，忽略配角。**每个在故事中存活到结局的角色都需要一个收束场景——哪怕只有几段。**

4. **伏笔系统性回收**：结局阶段是伏笔回收的最后机会。必须在写作开始前系统性地检查foreshadow-tracker.md——列出所有"推进中"和"未回收"的伏笔，为每一个设计回收方式。遗漏任何一个都是永久遗憾。

**结局阶段写作的执行要点：**

- **死亡/告别场景给足篇幅**：精神锚点角色（如老张）的死亡至少需要2-3章的完整展开——从病重到告别到葬礼到善后。不要压缩成一章。
- **"过去-现在-未来"三重收束**：结局不仅要收束"现在"的剧情线，还要回望"过去"（老张/钱老的传承）和展望"未来"（下一代/桃树苗/沈姓后人）。
- **全员归位清单**：写结局前列出所有存活角色的"最终位置"——在哪里、做什么、跟谁在一起。逐个确认都有收束。
- **终极意象预设**：在推演阶段就确定全书的最后一个画面、最后一句话、最后一个词。写作时所有结局章节都向这个终极意象推进。
- **结局不是"结束"是"继续"**：好的结局不是画句号——是让读者觉得"这些人会继续活下去"。利民店还开着、桃树还在长、马三还在算账——这些人"在"。

### Pitfall 8：大纲节点映射与实际进度脱节（批量续写必查！）

**这是批量续写（接着已有章节往后写）时最危险的盲区。**

问题：用户说"写Ch242-280，覆盖N162-N184"。你读了outline确认N162-N184的内容，然后开始写。但你**没读Ch228-241的正文**——只读了最后1章（Ch241）确认衔接。结果你从N162开始写——但N161-N180**已经在Ch228-241中被前序session写完了**。你写的内容跟已有的章节重复了。

根因：outline中的"节点→章节"映射（如"N162-N184在Ch242-280"）是**规划阶段**的预估。但实际执行中，前序session的写作速度可能远超预估——一个session可能写了40章覆盖了80个节点。如果你只读最后1-2章确认衔接，你只能看到"最后一章写了什么"，看不到"前序session已经覆盖了哪些节点"。

**真实案例**：本书Ch228-241（14章）已经覆盖了outline中N161-N180的全部内容（沈父深谈/老张去世/葬礼/空间告别/锚点松动/各人归位/孙主任退休/利民招牌），但outline的映射把这些节点放在"Ch242-280"区间。用户指令"写Ch242-280覆盖N162-N184"——如果不做节点审计，会从N162开始写重复内容。

预防（已编入第零步第8条）：
- 读最后**5-8章**（不是2-3章）——扫描内容，标记每个章节实际覆盖了哪些outline节点
- 用 `rg -n`、脚本或 `search_files` 搜关键事件/角色名快速确认：老张走了吗？沈父来了吗？空间告别发生了吗？
- 生成"实际剩余未写节点"清单——基于这个清单（不是基于outline的原始映射）规划本次写作
- 如果剩余节点远少于章节数：用**角色coda章节**（每个配角一个收束场景）、**世界观回响章节**（蝴蝶效应的美丽一面）、**烟火气日常**（邻居串门/过年）来填充。这些不是"水章"——结局阶段的它们是"呼吸章"，让读者在高潮之间喘口气
- 向用户报告发现：前序session已覆盖N161-N180，实际剩余约7个节点，我将基于这些+角色coda重新规划39章的三幕结构

### Pitfall 9：角色同质化——"全员好人俱乐部"（用户最痛的反馈之一）

问题：280章写完后，用户反馈"人物的设定过于脸谱化，所有角色风格都一样，全员睿智/全员大爱"。角色没有缺点/没有私心/没有矛盾——每个人都是"好人"，面对利益冲突时都选择了"大义"。

根因：
1. 设定阶段没有写角色小传——角色只有功能性信息（利益线/弧线），没有"他是一个什么样的人"的深度
2. 写作时AI倾向于写"好人"——让所有角色都表现得理性/善良/识大体。这跟"角色智商在线"是两回事——智商在线不等于没有缺点
3. 没有强制检查角色差异化——不同角色说了差不多的话/做了差不多的事/有差不多的反应

修复：
- 写每个角色前先翻characters.md中的角色小传——这个人说话什么风格？遇到利益冲突他会怎么选？他的缺点是什么？
- **同一场景中如果有2+个角色**，检查：他们说话的方式一样吗？如果互换台词读者能分辨吗？不能=脸谱化
- 每个角色出场时提醒自己：这个人至少有2个缺点。他可能聪明但自私，可能善良但懦弱，可能讲义气但冲动
- 反派/对手不能只有"坏"——但也不能只有"苦衷"。真实的人是复杂的：他可能做了坏事但认为自己是对的，也可能做了好事但动机是自私的
- **检查"全员好人"信号**：如果一章里所有角色都表现得理性/善良/识大体，说明你写成了"好人俱乐部"——加缺点、加私心、加矛盾

### Pitfall 10：2025年现代线不真实（用户反馈）

问题：主角是2025年的大学生，但现代线中他不用网购/不刷短视频/不用微信——完全不符合2025年年轻人的真实生活方式。

根因：写作时只关注了过去线的时代细节（1979年的物价/制度），忽略了现代线的真实性。认为"现代线是主角的日常生活不用特别处理"——但恰恰是日常生活的细节最容易让读者出戏。

修复：
- 现代线必须融入2025年的真实生活方式：手机/网购/外卖/微信/短视频/支付宝
- 写到现代线时查era-database.md确认现代物价和行为方式
- 不要只写"主角回到出租屋"——他回来后刷不刷手机？看不看外卖？微信有没有消息？这些细节让现代线真实



### Pitfall 11：编排者在中途停下来报告进度（用户明确说"不需要确认"时）

问题：用户明确说了"中途不需要任何人工确认"。但编排者在dispatch后台写作任务后，立即停下来呈现进度表格，像是在等用户确认。用户两次指出这个问题：
- 第一次："你为什么要停下来，为什么不准寻写作流程"
- 第二次：在编排者又一次停下来做状态报告后继续

根因：编排者把"完成任务后的交付"跟"中途的阶段转换"混为一谈。dispatch后台任务后，编排者应该：
1. 告知用户"任务已dispatch，完成后自动推进下一步"
2. **不要**停下来等用户确认
3. 后台任务返回结果后，**立即**推进下一步（review/修复/下一批推演），不呈现进度表格等用户说话

**正确行为**：
```
用户说"中途不需要任何确认" →
  dispatch后台写作 →
  一句话说明"完成后自动推进" →
  任务返回结果 →
  立即推进下一步（不呈现进度表格等确认）→
  继续直到全流程完成
```

**例外**：只有在遇到真正需要决策的分叉点时才停下问用户——例如"发现subagent写了全书结局需要删除重写，确认吗？"这种不可逆操作。常规的review→修复→推演→写作循环不需要确认。
