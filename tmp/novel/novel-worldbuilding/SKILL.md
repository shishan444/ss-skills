# 剧情架构系统

这个Skill解决一个根本问题：**AI创作小说时不再孤立生成剧情或设定，而是始终检查双向关系——当前剧情由哪些设定支撑？当前设定会逼出哪些剧情？剧情推进后哪些人物、地图、关系、背景和规则会发生变化？**

它接在 `novel-concept-design` 之后。概念阶段已经确定的“第一章追读画面、核心机制、进攻性限制、长写变量X”是本 skill 的输入，不要回头重做选题。只有当 concept 的发动机明显不成立时，才做最小修正，并把修正理由写清楚。

## 核心原则

**好设定不是"丰富"，而是"有压力"。**

它会：
- 逼人物做选择（因为设定创造了必须回应的处境）
- 逼剧情转弯（因为设定产生了不可预期的连锁反应）
- 逼关系变形（因为设定改变了各方的利益格局）
- 逼秘密浮出水面（因为设定施加的压力让隐瞒变得不可持续）

**坏设定的特征：删掉它剧情照样跑。说明它是装饰不是结构。**

## 职责边界

- 负责：把 concept.md 扩成可执行的世界观、角色圣经、关系图、角色状态账本、时代设定库、剧情节点、伏笔系统、叙述腔调、动态情绪呈现路由、前三章冷启动设计、前30章追读钩子链和 architecture-status.md。
- 不负责：写章节正文。正文执行交给 `novel-chapter-write`。
- 不负责：在没有必要时重新筛选题材。选题入口交给 `novel-concept-design`。
- 不负责：阶段完成后的质量门禁。审计和修复循环交给 `novel-stage-review`。

如果发现前一环节的输入有缺口，先判断缺口是否会导致后续崩塌。会崩塌的，标为P0并修；只是信息不够细的，在本环节补齐，不把流程退回给用户。

## 触发场景

1. **新小说设定阶段**：从concept.md构建完整剧情架构
2. **阶段写作前**：检查本阶段的剧情节点是否由设定自然逼出
3. **写作中发现断裂**：当前剧情跟设定脱节时，用压力测试诊断
4. **新角色/新设定引入时**：检查它跟现有架构的压力关系
5. **阶段review时**：验证剧情-设定映射是否完整
6. **长篇中期状态盘点**：已完成50+章后，对全局做完整的状态盘点——确认所有设定/角色/伏笔/时间线/情绪曲线的当前状态，产出"状态快照"作为后续推演的准确输入。这是长篇小说最关键的维护动作。

## 接入原则

读取 concept.md 后，先提取四个承重输入：

```markdown
- 第一章追读画面：
- 核心机制：
- 进攻性限制：
- 长写变量X：
```

所有世界观、人物和剧情节点都必须围绕这四个输入展开。不要为了显得宏大而增加无法产出剧情的设定。若需要时代、职业、商品、制度等事实支撑，先调研并生成 era-database.md；查到的事实和基于事实做出的推断要分开写。

## 默认诊断输出（五块）

做架构诊断、状态盘点或阶段推演时，输出至少包含以下五块。新小说完整构建还要额外产出 worldbuilding.md、tone-guide.md、emotion-style-map.md、characters.md、character-ledger.md、outline.md、front-30-hook-map.md、writing-guide.md、era-database.md、foreshadow-tracker.md。

---

### 输出一：故事引擎判断

回答一个问题：**是什么在持续驱动这个故事往前走？**

```
故事引擎 = 核心机制 × 限制 × 主角处境 × 外部压力

判断内容：
1. 引擎类型：
   - 资源差驱动（两界物资落差/信息差/能力差）
   - 限制驱动（必须解决限制才能推进）
   - 压力驱动（外部威胁逼近/倒计时）
   - 欲望驱动（主角想得到什么但得不到）

2. 引擎是否可持续：
   - 这个引擎能跑400+章吗？
   - 引擎会不会中途熄火？（信息差用完了怎么办？）
   - 引擎熄火后有没有第二引擎接力？

3. 引擎产生的"自然剧情"清单：
   - 不是作者想写什么，是引擎会逼出什么
   - 例：空间1m³+日月光充能 → 逼出"晒太阳攒能量""选什么带""能量不够怎么办"
```

---

### 输出二：设定资产清单

每个设定不是一条"信息"，是一个"会产出剧情的资产"。

```
对每个设定标注：
| 设定 | 类型 | 当前状态 | 它逼出了什么剧情 | 它还在产压力吗 | 删掉会怎样 |
```

类型分类：
- **机制设定**：空间规则/充能规则/穿越规则/成长规则
- **人物设定**：角色的利益线/秘密/关系/能力边界
- **世界设定**：时代背景/社会规则/权力结构/经济系统
- **限制设定**：使用代价/时间限制/地理限制/社会限制

关键标注——"删掉会怎样"：
- 如果删掉这个设定剧情还能跑=这个设定是装饰，考虑去掉或升级
- 如果删掉剧情立刻崩=这个设定是承重墙，必须在前期牢牢建立

---

### 输出三：剧情-设定映射

**这是最核心的输出。** 每一条剧情必须能追溯到支撑它的设定。

```
| 剧情节点 | 由哪个设定逼出 | 该设定还支撑了哪些其他剧情 | 推进后什么变了 |
```

示例：
```
| 林北第一次倒卖被马三挤压 | 1979年投机倒把罪+集市黑市规矩+马三的利益线 | 同一设定还逼出了：被举报危机/学会给人好处/理解关系网 | 推进后：林北理解了1979年办事要靠关系 |
| 沈知秋发现空间秘密 | 沈知秋的观察力+她的独立判断+空间的可见性限制 | 同一设定还逼出了：信任危机/合作升级/沈知秋成为合伙人 | 推进后：两人的关系从观察变成利益绑定 |
```

**如果一条剧情找不到支撑它的设定——这条剧情是"作者想让它发生"而不是"设定逼它发生"。需要重新设计。**

---

### 输出四：断裂点与风险

主动寻找架构中的裂缝：

1. **设定空转**：有设定但没产出剧情。比如设计了"空间内有上一个主人的痕迹"但从来没用到
2. **剧情悬空**：有剧情但找不到支撑设定。比如"林北突然变聪明了"——凭什么？
3. **压力衰减**：某个设定前期有压力但后期没压力了。比如"投机倒把罪"到1985年后就不构成威胁了——需要新压力源接力
4. **角色脱离利益线**：角色做了不符合他利益线的事。比如马三突然无条件帮林北——凭什么？
5. **时间线冲突**：历史事件跟剧情推进对不上。比如1983年严打的时间点跟大纲标注的时间

每个断裂点标注严重程度：
- 🔴 P0：不修会导致后续崩塌
- 🟡 P1：影响质量但不致命
- 🔵 P2：可以后续处理

---

### 输出五：设定状态变更追踪

**剧情推进后，设定不是静态的——它会变。** 每个阶段结束后更新：

```
| 设定 | 阶段前状态 | 阶段后状态 | 变化原因 | 下一阶段它会产生什么新压力 |
```

示例：
```
| 空间大小 | 2m³ | 4m³ | 完成了20次往返 | 4m³能放更多货→但启动能量也多了→穿越频率降低→每次穿越必须做更大的事 |
| 马三跟林北的关系 | 敌对→七三分账 | 合作伙伴(但暗中收集信息) | 林北的货确实比马三好+马三判断合作比对抗更有利 | 马三掌握的信息越来越多→未来可能成为泄密源 |
| 1979年投机倒把环境 | 高压 | 严打期间更高压→严打后松动 | 1983年严打+后续政策放宽 | 松动后竞争者增多→林北的先发优势被削弱 |
```

## 剧情压力测试（核心模块）

**每当设计一个情节，必须回答以下五个问题：**

### 问题1：这个情节是由设定自然逼出来的，还是作者想让它发生？

判断方法：
- 从当前所有设定出发推理——这些设定组合在一起，"必然"会发生什么？
- 如果答案是"主角会遇到X困境"——✓ 设定逼出的
- 如果答案是"主角会做X选择"——需要追问"他凭什么这么选？"
- 如果答不上来——✗ 这是作者硬塞的

**不合格的处理：重新设计，让设定来逼出这个情节。**

### 问题2：如果角色不按作者意愿行动，按他的动机和资源，他最可能怎么做？

对每个关键角色追问：
- 他的核心利益是什么？
- 在这个处境下，他的最优选择是什么？
- 如果他选了最优选择，主角的计划会不会被打乱？

**如果所有角色的最优选择都恰好让主角成功——说明你在操控角色。**

真正的张力来自：角色的最优选择跟主角的利益冲突。

### 问题3：如果世界规则是真的，这个事件会造成哪些二级后果？

每个事件不是孤立发生的。它会引发连锁反应。

示例：
- 一级后果：林北在集市上卖出了好东西，赚了钱
- 二级后果：马三注意到了→调查林北的货源
- 三级后果：马三找不到货源→跟派出所的人提起"来了个来历不明的人"
- 四级后果：派出所开始留意林北

**设计情节时必须追到至少三级后果。只看一级后果=偷懒。**

### 问题4：如果这个设定删掉，剧情是否还能成立？

这是最锋利的一刀。

- 删掉空间的大小限制→剧情还能跑吗？如果能，说明1m³这个限制没产生足够的压力
- 删掉1979年投机倒把罪→剧情还能跑吗？如果能，说明这个历史背景是装饰不是结构
- 删掉沈知秋的观察力→剧情还能跑吗？如果能，说明这个角色特质没有承重

**如果还能成立，说明这个设定不够关键。要么让它产生更多压力，要么承认它是装饰。**

### 问题5：这个设定/情节在前期埋了伏笔吗？后面会收束吗？

- 前面有没有铺垫？
- 后面有没有回收？
- 如果这个情节凭空出现→不合格
- 如果这个情节出现了但后面再也没用到→不合格

## 创造性剧情推演（核心模块）

**大纲不是一次写死的——它是活的。随着写作推进，设定在变、人物在变、世界在变。这个模块的作用是：基于当前状态，推演出"接下来必然会发生什么"，让写作永远有剧情可写。**

这不是"编故事"——是"让设定说话"。

详细的输入清单、超大范围推演和结局推演协议见 `references/pressure-line-derivation.md`；主流程只保留必要骨架。

### 触发时机

- 当前阶段的剧情节点快用完时（还剩不到5个节点）
- 写作中发现"接下来不知道写什么了"
- 进入新阶段前需要扩展剧情节点
- 已完成章节产生了大纲中没有预料到的新状态

### 推演输入

推演前必须读取以下当前状态：

1. **已完成章节的最后5-7章**——确认当前剧情走到了哪里。如果是100+章的长篇，还需抽读关键里程碑章节（重大转折/新角色出场/设定状态变化点），仅读最后3章会遗漏关键状态变化
2. **设定状态变更追踪**——每个设定现在的状态是什么
3. **人物当前状态**——每个角色的处境/利益/关系网现在的状态
4. **时间线当前位置**——故事推进到了哪个时间点
5. **未回收的伏笔**——哪些伏笔还在悬着
6. **情绪路由当前位置**——上一个路由状态是什么（剧情功能/读者期待/目标体感/呈现模式/爽点阶段），接下来读者需要什么体感；必须判断要用哪种文字呈现模式兑现，不只写“爽/紧张/温暖”标签
7. **最新的阶段review文件**（如act1-review.md/act3-review.md等）——如果存在review文件，**必须读取**。它包含：写作质量问题（碎片化文体/口头禅泛滥/搞笑元素归零等）、P0/P1级风险项、伏笔回收状态核查、章节长度数据。这些直接影响推演节点的设计——如果review说"搞笑元素归零"，推演节点必须增加“轻松搞笑/爽点兑换”等路由，并标出目标体感、呈现模式、爽点阶段和章长策略；如果review说"碎片化文体是P0"，必须在architecture-status.md的断裂点中标注为最高优先级，并在推演质量自检中新增对应检查项。

### 推演方法（六条压力线交叉法）

从六条独立的压力线出发，分别推导"它们各自会逼出什么"，然后找交叉点——交叉点就是新剧情。

#### 压力线1：设定压力线

每个活跃设定现在在产生什么压力？

```
对每个设定问：它的当前状态会逼出什么？
```

#### 压力线2：角色利益线

每个角色的当前处境+利益，他下一步最优选择是什么？

```
对每个角色问：如果他按自己的利益行动，他接下来会做什么？
```

**关键：角色的最优选择跟主角的利益冲突的地方，就是新剧情的爆发点。**

#### 压力线3：后果链

已发生的重大事件，它的二三四级后果是什么？

```
对最近3个重大事件追问：这件事发生后，接下来会连锁发生什么？
```

#### 压力线4：历史事件线

接下来这个时间段，历史上会发生什么大事？

```
查当前时间点之后的真实历史事件，问：这个事件会对主角和所有角色产生什么影响？
```

#### 压力线5：情绪曲线线

当前情绪状态是什么？读者需要什么？

```
问：上一个阶段/章节的情绪是什么？读者现在需要爽还是需要紧张？需要新鲜感还是需要熟悉感？这一章应该用什么文字呈现模式让读者真的感到它？
```

规则：
- 如果刚爽完→需要一个挫折或换地图
- 如果刚挫折完→需要快速翻盘
- 如果模式重复了→需要引入新元素/新角色/新场景
- 如果太紧张了→需要一个日常/温情/搞笑的缓冲
- 如果读者等待已久→需要完整的期待兑换，不得压缩成结果说明
- 如果只是低密度过渡→需要压缩，不要把转场硬写成完整章

#### 压力线6：伏笔回收线

哪些伏笔该回收了？

```
查foreshadow-tracker.md：哪些伏笔已经"埋了超过2个阶段但还没回收"？
```

规则：
- 埋了2个阶段以上的伏笔必须在近期回收
- 回收的方式可以跟新剧情结合——"原来这件事的真相是X"成为新剧情的爆发点

### 交叉——找到爆发点

把六条压力线的产出放在一起，找交叉点。交叉点就是新剧情：它不是凭空编出来的，而是设定、人物利益、后果链、历史事件、情绪需求和伏笔回收自然交汇的结果。

### 推演产出

每次推演产出20-30个新剧情节点（覆盖单一阶段）。如果跨阶段推演（如从第四阶段末推到第五阶段中），节点数可扩展到40-50个——但必须标注阶段边界，且跨阶段的节点要有"承上启下"的过渡设计。格式：

```
| 节点编号 | 剧情事件 | 由哪条压力线逼出 | 涉及角色 | 时间 | 冲突/转折 | 剧情功能 | 读者期待 | 目标体感 | 呈现模式 | 爽点阶段 | 章长策略 | 伏笔状态 |
```

每个节点必须通过压力测试5个问题后才能加入大纲。

**超大范围推演（80+章/2+故事年）的处理：**

当推演范围超过80章或覆盖2+个故事年时（如用户要求"Ch161-280"或"写完整个第五阶段"），读取 `references/pressure-line-derivation.md` 和 `references/large-range-derivation-example.md`，拆成命名子阶段，每个子阶段有时间边界、情绪弧线和阶段收束节点。

**结局阶段推演的特殊处理：**

当推演范围包含全书结局（最终阶段/最后一阶段）时，必须额外读取 `references/ending-stage-derivation.md`。结局推演与中后期推演有本质差异——中后期是"展开新线"，结局是"收束所有线"。结局推演必须完成三份前置清单（伏笔清算表/角色最终位置表/终极意象设计）后才能开始节点推演。

**Review文件的P0/P1问题必须在推演产出中体现：**

读取的review文件中标记的P0/P1问题不能被忽略。它们必须在以下位置反映：
1. architecture-status.md的"断裂点与风险"部分（写作质量问题也算断裂点）
2. 推演质量自检的新增检查项（如"搞笑元素比重是否符合review要求"）
3. 推演节点的情绪路由设计（如果review说"连续N章沉重"，推演必须有意识地增加"轻松搞笑/爽点兑换/烟火气"等路由节点，并标出读者期待、目标体感、呈现模式、爽点阶段和章长策略）

### 推演质量自检

- [ ] 每个推演出的剧情都有至少一条压力线支撑（不是凭空编的）
- [ ] 至少有3个剧情节点来自多条压力线的交叉（这些通常是最精彩的）
- [ ] 推演出的剧情推进了时间线（不是原地踏步）
- [ ] 推演出的剧情改变了至少一个角色/设定的状态
- [ ] 伏笔回收线中"超期未回收"的伏笔被处理了
- [ ] 情绪曲线符合原则（爽多于压抑/挫折后快速翻盘）
- [ ] 每个关键节点都有剧情功能、读者期待、目标体感、呈现模式、爽点阶段和章长策略
- [ ] 核心爽点/关系定调/死亡告别/重大反转被标为完整展开，而不是被固定章数压缩
- [ ] 跟已完成章节衔接（不断裂）
- [ ] 所有角色智商在线（没有人为了剧情需要犯蠢）

## 工作流程

### 场景A：新小说完整构建

1. 读concept.md；若是新书完整构建，同时读 `../创作方法论/creation-workflow.md` 建立流程边界
2. 做故事引擎判断——确认引擎类型和可持续性
3. 构建设定资产清单——每个设定标注类型/状态/压力产出/删掉后果
4. 构建 tone-guide.md——读取 `../创作方法论/tone-model.md`，定义稳定叙述声音、幽默发动机、语言质感、情感节奏和读者体感基调。不要把 tone 写成单一情绪标签。
5. 构建 emotion-style-map.md——读取 `../创作方法论/emotion-style-routing.md` + `../创作方法论/style-model.md` + `../创作方法论/scene-model.md`，定义本书的稳定叙述基准、动态呈现模式、模式切换规则和章长策略。
6. 构建人物系统——读取 `references/character-bible-and-continuity.md` + `../写作经验库/index.md`，必要时读取 `../写作经验库/专题/writing-exp-2-1.md` ~ `2-4.md`，产出 characters.md + character-ledger.md。每个重要角色必须有一句话钉子、欲望与恐惧、人格矛盾、行为算法、语言动作识别、关系压力。
7. 构建世界观——读取 `../创作方法论/worldbuilding-model.md`，落成六层结构+五条二阶闭环
8. 构建写作经验路由——读取 `../写作经验库/index.md`，在 writing-guide.md 中标注本书高频场景类型应调用哪些写作经验，以及如何转化为章节执行约束；这些经验必须服务 emotion-style-map.md 的呈现模式。
9. 构建时间轴——历史事件+成长链+阶段划分
10. **拆剧情节点**——读取 `../创作方法论/pacing-model.md` + `../创作方法论/payoff-model.md`，每条剧情必须通过压力测试五个问题，并标注剧情功能、读者期待、目标体感、呈现模式、爽点阶段和章长策略
11. **设计前三章冷启动和前30章追读链**——读取 `../创作方法论/webnovel-opening-hook-model.md`，产出 front-30-hook-map.md；前三章必须形成“设定入场→限制加压→小爽点兑换”，前30章必须有短钩子/中钩子/长钩子
12. 建立伏笔与预期清单——读取 `../创作方法论/foreshadow-model.md` + `../创作方法论/expectation-model.md`，产出 foreshadow-tracker.md 的初始状态
13. 做剧情-设定映射——确认每条剧情有设定支撑
14. 做断裂点检查——找空转/悬空/衰减/脱离/冲突/情绪路由缺失/冷启动弱/前30章钩子断裂
15. 输出文件

### 场景B：阶段写作前的架构检查

1. 读outline.md中本阶段的剧情节点
2. 对每个节点跑压力测试
3. 检查每个节点是否有情绪路由：剧情功能、读者期待、目标体感、呈现模式、爽点阶段、章长策略
4. 如果检查范围在前30章内，额外读取 front-30-hook-map.md，确认本阶段短钩子/中钩子/长钩子不断裂
5. 更新设定状态变更追踪
6. 补充断裂点
7. 确认本阶段有足够的"设定逼出的剧情"、"读者情绪曲线"和"追读牵引"
8. 如果节点不够→执行**创造性剧情推演**扩展

### 场景C：写作中发现断裂

1. 定位断裂类型（空转/悬空/衰减/脱离/冲突）
2. 跑压力测试诊断根因
3. 修正设定或剧情
4. 更新映射

### 场景D：剧情枯竭时推演扩展

1. 读取当前状态（已完成最后3章/设定状态/人物状态/时间线/伏笔/情绪曲线）
2. 执行六条压力线交叉推演
3. 产出20-30个新剧情节点
4. 每个节点跑压力测试
5. 通过的节点加入outline.md
6. 更新剧情-设定映射和设定状态变更追踪
7. 输出更新后的大纲段落

### 长篇小说的可持续创作循环（推演→写作→review→推演→写作→...）

400章以上的长篇小说不可能一次推演完所有剧情节点。正确的模式是**循环推进**——每完成一批章节后重新推演下一批。这个循环在实战中支撑了280章的持续创作。

```
循环单元（约40-80章/循环）：
  1. 推演：读当前状态→六线交叉→产出40-80个剧情节点→更新outline/architecture-status/foreshadow
  2. 写作：按剧情节点串行写作（每批≤30章受session上限约束）
  3. （跨session续写：如果上一步没写完，新session续写剩余章节）
  4. Review：八维度检查→P0/P1修复
  5. 回到第1步，推演下一批
```

**循环的关键约束**：
- 每次推演只覆盖1-2个阶段的时间跨度（不能一次推演全书）
- 每次推演必须基于"已完成章节的实际状态"，不能基于上一次推演时的预估
- 写作阶段的边界（时间线/不能写结局）必须在dispatch指令中重复强调至少3次
- 推演和写作可以由不同的subagent执行，但推演的输出（outline节点+状态更新）是写作的唯一输入

## Pitfalls（速记）

详细案例和修复理由见 `references/worldbuilding-pitfalls.md`。主流程执行时至少记住这五条：

1. 不做时代调研，后续物价、商品、制度和现代线生活方式会系统性失真。
2. dispatch 写作时边界不清，模型会受"把故事讲完"的冲动驱使，擅自写到结局。
3. 只有阶段概述没有节点表，写作中段必然停滞。
4. 固定章节数会污染结构判断，让所有小说长成同一种形状。
5. 推演必须读最新 review；写作质量P0/P1也是剧情架构的断裂点。

### 场景E：长篇中期状态盘点（50+章）

触发：已完成50+章 / 进入新大阶段 / 感觉剧情跟设定脱节但不确定断在哪。

1. 读取全部设定文件（concept/worldbuilding/characters/character-ledger/outline/foreshadow-tracker/tone-guide/emotion-style-map/writing-guide）+ **最新的阶段review文件**（act*-review.md，如果存在）
2. 读取最后5-7章确认当前剧情位置 + 抽读关键里程碑章节（重大转折点/新角色出场/重大事件）
3. **产出"当前状态快照"**——这是后续所有推演的基础输入：
   - 时间线位置（两界各到了什么时间点）
   - 空间状态（大小/穿越次数/充能情况/限制状态）
   - 角色当前状态表（每个角色的处境/核心利益线/下一步最优选择）
   - 关系压力变化（哪些关系边已经变形，哪些仍停在旧状态）
   - 未回收伏笔清单（哪些超期了）
   - 情绪曲线当前位置（上几章是什么情绪，读者下一步需要什么）
4. 对照设定资产清单，逐项检查"这个设定现在还在产出压力吗"→标注压力衰减项
5. 用五块输出做全局诊断（故事引擎/设定资产/剧情-设定映射/断裂点/设定状态变更）
6. 追溯更新foreshadow-tracker.md——用可用搜索工具按关键词搜索实际章节，把"未埋设"改为实际状态；Codex 环境优先用 `rg -n` / `rg -c`，有 `search_files` 时再用 `search_files`
7. 执行六条压力线交叉推演，产出后续剧情节点（跨阶段时40-50个）
8. 输出：architecture-status.md（新建或重写）+ outline.md（追加节点表）+ foreshadow-tracker.md（重写状态）+ character-ledger.md（写回角色当前状态/关系变化/偏离警报）；若写作经验路由或情绪呈现策略发生变化，同时更新 writing-guide.md / emotion-style-map.md

> 详细的状态快照格式和断裂点分级示例见 `references/status-audit-example.md`

## 质量门槛与时代设定库

构建新小说架构、做大范围推演或修正设定时，读取 `references/worldbuilding-quality-gates.md`。它定义：

- 引擎、设定、映射、压力、角色、时间线六类质量门槛
- era-database.md 的调研范围和事实/推断标注规则
- 调研后如何反向修正 worldbuilding.md、characters.md、outline.md

只做小范围断裂诊断时，可以不读完整参考文件，但必须保留两个底线：剧情有设定支撑；真实年代/职业/物价/商品不能靠想象编。

## 输出文件

1. **worldbuilding.md** — 世界观（六层+五闭环+12问）+ 设定资产清单
2. **characters.md** — 人物（每个角色的利益线+秘密+压力关系+缺点+私心+矛盾点）
3. **character-ledger.md** — 角色状态账本（当前处境/关系变化/利益变化/最近关键选择/下一步最优选择/偏离警报）
4. **outline.md** — 详细剧情大纲（每个节点通过压力测试）+ 剧情-设定映射
5. **architecture-status.md** — 故事引擎判断 + 断裂点与风险 + 设定状态变更追踪
6. **tone-guide.md** — 本书稳定叙述声音、幽默发动机、语言质感、情感节奏和读者体感基调
7. **emotion-style-map.md** — 情绪-文字呈现路由（稳定叙述基准、动态呈现模式、模式切换规则、章长策略）
8. **front-30-hook-map.md** — 前三章冷启动设计 + 前30章短/中/长钩子链
9. **writing-guide.md** — 本书写作执行指南（高频场景类型、经验库路由、角色语言规则、场景执行约束）
10. **era-database.md** — 时代设定库（物价/商品/社会细节/物品验证清单）
11. **foreshadow-tracker.md** — 伏笔追踪表 + 预期清单（每次推演后必须更新状态）

> **常见情况**：写作过程中foreshadow-tracker.md经常被忽略，大量伏笔标记为"未埋设"但实际已在章节中埋设。每次状态盘点时必须根据实际章节内容追溯更新伏笔状态——用 `rg -n`、脚本或 `search_files` 搜索关键词定位实际埋设/回收的章节，然后更新状态列。
