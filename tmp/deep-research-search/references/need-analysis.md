# 需求理解

需求理解决定 6 轮搜索的质量上限。理解偏了，后面所有轮次都在错误的方向上浪费力气。

这一步只做一次，在进入研究循环之前完成。产出 `research-brief.md`。

---

## 三层分析

### A. 意图理解

以用户视角推测四个问题，不需要向用户确认：

- **做什么**：用户想了解什么（不是字面意思，是背后的目的）
- **做到什么程度**：概览级 / 决策级 / 专家级
- **什么算完成**：能回答什么问题、能做出什么判断
- **产生什么价值**：解决什么困惑、支撑什么决策

### B. 领域全景（→ 横向搜索方向）

目标：确保不遗漏重要视角。

1. 识别领域归属（技术 / 商业 / 学术 / 社会 / 产品 / ...）
2. 应用场景：这个主题出现在哪些场景中？谁在用？为什么用？
3. 产业链/生态：上下游是谁？竞争者是谁？依赖什么？
4. 普通人视角：不在这个领域的人如何看待这个主题？
5. 每个场景转化为一个搜索方向（"主题 × 场景关键词"）

### C. 推理路径（→ 纵向搜索方向）

目标：确保关键维度有数据支撑。

**不要问"这个领域有哪些维度"——这个问题会让你退回模板思维。**

要问的是："这个领域里最核心的矛盾或张力是什么？"
找到矛盾，从矛盾的两端展开，维度就自然浮现了。
维度不是你"设计"的，而是从主题本身"长出来"的。

以下是这个思考方法在不同领域的体现：

- 技术领域 → 核心矛盾通常是"想要什么"和"技术上能做到什么"之间的差距。从两端展开：需求侧的真实痛点是什么？供给侧的技术极限在哪里？中间的鸿沟怎么填？
- 商业领域 → 核心矛盾通常是"价值创造的方式"和"价值捕获的方式"之间的错位。从两端展开：谁在创造价值？谁在拿走利润？为什么？
- 学术领域 → 核心矛盾通常是"两个都合理的理论给出相反预测"。从两端展开：各自的证据是什么？分歧的焦点在哪？

如果主题不显然属于以上任何一类，就回到根本方法：找到这个主题里最让人纠结、最说不清楚的那一点，从那一点展开。

## 不离题规则

**所有搜索方向必须包含研究主题本身。**

搜的是"主题 × X"的交叉点，不是 X 本身。例如研究"Rust 异步编程"：
- ✅ "Rust 异步编程 × tokio 框架"
- ✅ "Rust 异步编程 × 嵌入式场景"
- ❌ "tokio 框架"（脱离主题，无限扩展）
- ❌ "嵌入式开发"（脱离主题，无限扩展）

脱离主题的搜索方向会带来两个问题：一是搜出来的内容和你的研究目标不相关，浪费搜索资源；二是容易越搜越远，偏离核心问题。

## 受众推断

从需求推断，不额外提问。不确定则默认普通人视角。

| 受众 | 报告焦点 |
|------|---------|
| 技术决策者 | 对比数据、选型建议、风险评估 |
| 业务决策者 | 市场格局、趋势判断、投资回报 |
| 学习者 | 概念脉络、入门路径、关键里程碑 |
| 不确定 | 面向普通人的通用报告 |

数据采集和报告剪裁是两回事——先全面采集，最后根据受众组织报告。受众只影响报告的焦点和详略，不影响数据采集范围。

## 搜索语言策略

默认**中英文双语搜索**——每个搜索方向产出中文 + 英文两组搜索词。中文源给本土视角，英文源（BBC / ESPN / The Athletic / Guardian 等）通常深度更高、噪声更少。

**例外**：主题纯中国本地、无国际语境（如某地政策、纯国内社会事件），只搜中文。

判断标准：主题是否含国际实体或全球性议题？是 → 双语；否（纯本地）→ 单中文。此判断写入 `research-brief.md`，循环阶段据此设计搜索词。

## 产出格式

写入 `research-brief.md`：

```markdown
# 研究简报：{主题}

## 意图理解
（A 层分析结果）

## 领域全景
（B 层分析结果 + 横向搜索方向）

## 推理路径
（C 层分析结果 + 纵向搜索方向）

## 研究问题清单
（横向 + 纵向的综合问题列表。每轮缺口分析时逐条检查覆盖状态。）

## 第 1 轮搜索方向
（横向 + 纵向的搜索词组合，遵循不离题规则）

## 受众判断
（暂存，报告阶段才使用）

## 意外追踪
（研究过程中发现的、不在初始问题清单中但值得关注的信息。
 前 3 轮消化时把"有意思但和当前方向不直接相关"的信息记到这里。
 后 3 轮枚举搜索方向时，"异常信号"来源从这里取。）
```
