| # | 失败现象 | 根因 |
|---|---------|------|
| 1 | 任务恢复逻辑复杂，初始化耗时 | 旧任务检测 + 进度推断增加不必要的状态管理 |

## 设计决策记录

### 初始化简化
- **去掉**：check_env.py、任务恢复逻辑、空模板预创建
- **保留**：check-deps.mjs（Node + Chrome CDP 连通性）、init_workspace.py
- **为什么**：初始化职责是"确保环境就绪"，不是"管理历史状态"

### 需求理解独立为一次性步骤
- 从初始化阶段剥离，循环前执行一次
- A(意图)/B(领域全景)/C(知识覆盖) 三层平行分析
- B 和 C 是平行独立视角（横向 vs 纵向），不是串行依赖
- 产出 research-brief.md，不向用户确认
- **锚点规则**：搜索方向必须包含主题本身，防止无限扩展

### 6轮分两阶段
- 轮次1-3覆盖（缺口驱动）+ 轮次4-6深挖（图谱驱动）
- 前3轮轻量消化：概念提取 + 图谱增量 + 缺口标注
- 后3轮深度消化：关联/矛盾/暗线 + 图谱深化 + 薄弱区域识别
- 固定3:3分界，避免复杂判断逻辑
- 每轮多主题并行搜索（所有缺口/薄弱节点）

### 搜索采集自动化
- 新增 serp_collect.py：搜索词列表输入 → URL文件 + 报告输出
- LLM 介入点：设计搜索词 → 跑脚本 → 读报告 → 判断
- 脚本内部：搜索词 × 3引擎，每引擎最多6页
- **CDP Proxy 保留**：操作模式优势（bash 脚本化 vs LLM 逐次调用）

### 数据获取工具分工
> V3.1 更新：WebFetch 已替换为 `url_fetch.py`（trafilatura+BS4 两级提取，并发+质量评分），CDP 补漏改为 `cdp_fetch.py`。下方为 V3 原始决策记录。

- ~~WebFetch 首选（并行 ≤10）~~ → **`url_fetch.py` 首选**（并发 + 断点续传）
- ~~CDP Proxy 补漏（WebFetch 失败/内容不完整时）~~ → **`cdp_fetch.py`**（反爬/JS渲染/登录态兜底，读 `_failed_urls.txt`）
- 不做数据分级，完整存储

### 报告锻造简化
- 6轮消化已产出知识图谱（图谱即熔炼结果）
- 不再单独做"全量熔炼"
- 流程：读图谱 → 受众×领域 → 按需回查原始数据 → 撰写

### Reference 文件重组
- 旧5个文件 → 新3个文件，按执行阶段组织
- need-analysis.md（需求理解阶段加载）
- research-loop.md（循环阶段加载）
- report-forge.md（报告阶段加载）

### 受众推断（不提问）
- 从需求推断，不确定则默认普通人视角
- 受众只影响报告剪裁，不影响数据采集范围
- 数据采集与报告剪裁解耦

## V3 → V3.1 重构：驱动力从"不犯错"到"发现可能性"（2026-05-15）

### 根因问题

V3 的整套机制（缺口分析、图谱薄弱节点、铁律）本质上是"确认式"的——确认清单里的东西都覆盖了，确认每个节点的数据够用。擅长"把已知未知搞清楚"，不擅长"发现未知未知"。

### 重构决策

| # | 决策 | 理由 |
|---|------|------|
| D1 | 驱动力从"不犯错"转"发现可能性" | 研究价值在于发现而非确认 |
| D2 | 前 3 轮保持原有机制 | 建框架阶段机制成熟 |
| D3 | 后 3 轮增加全局审视 | 暗线只能在全局视角下发现 |
| D4 | 新增熔炼文件 smelting.md | 知识图谱是节点级理解，缺少全局级理解的载体 |
| D5 | 后 3 轮搜索方向从枚举排序选取 | 替代简单的缺口分析，确保方向多样性和战略价值 |
| D6 | 枚举来源 6 类、评分三维度 | 来源多样性比数量重要；发现潜力和认知杠杆优先 |
| D7 | 铁律三档重构 | 操作安全/结构保障/研究质量性质不同 |
| D8 | 节点模板自适应 | 不同知识类型需要不同的组织方式 |
| D9 | Agent/Skill 分离但显式桥接 | 防止两套指令脱节 |
| D10 | 后 3 轮 subagent 读相邻节点 | 暗线出现在节点之间 |
| D11 | 全局审视用反思性问题逼迫深度 | LLM 容易"读过就过" |
| D12 | 熔炼文件同时服务报告锻造 | 避免重新从图谱综合 |
| D13 | C 层方法论显式化 | 给方法论而不只给示例 |
| D14 | 新增意外追踪字段 | 为后 3 轮枚举提供异常信号来源 |

### 新增失败模式

| # | 失败现象 | 根因 |
|---|---------|------|
| F16 | 后 3 轮仍在填缺口 | 全局审视流于形式，没有真正退后看全貌 |
| F17 | 熔炼文件和图谱重复 | 没有理解两者是不同层面的东西（节点级 vs 全局级） |
| F18 | 枚举候选全来自缺口 | 没有从 6 个来源分别思考 |
| F19 | 节点模板选择机械 | 不读数据就预设类型 |

## V3.1 工程修复（2026-06）：storage/events 接入 + 死代码清理

### 根因
V3→V3.1 重构聚焦方法论层（research-loop / graph / report），新增的工程基础设施模块 `storage.py`（manifest 溯源）与 `events.py`（领域事件）只建了骨架，**未回填到早期写的 `url_fetch.py` / `cdp_fetch.py` / `serp_collect.py`**，导致：
- `_manifest.json` 永不生成，`health.manifest_summary` 恒返回 0
- `_events.log` 长期只有 `HEALTH_CHANGED` 一类，`ZERO_RESULTS` / `FETCH_COMPLETED` 从未写入
- `test_storage.py` 沦为「绿色谎言」——测试全过，但测的是从未在生产路径运行的代码

### 修复
- `url_fetch.py` / `cdp_fetch.py`：写文件后调 `storage.record_url`，manifest 真正生长（幂等：同 URL 覆盖更新，不重复计数）
- `cdp_fetch.py` 接 `events.emit(FETCH_COMPLETED)`、`serp_collect.py` 加 `--workdir` 接 `emit(ZERO_RESULTS)`——events 三接入点全兑现
- 清理死代码：`locale_service` 两函数、`cdp_client.key()`、`cdp-proxy.mjs` 9 个无调用方端点

### 教训
重构新增「基础设施模块」时，必须同步回填到既有调用方；否则模块空转、测试变绿色谎言，且 health 健康度指标长期失真。

## V3.2 设计转变（2026-06）：获取只搬运 + 图谱认知分层 + 双语搜索

### 驱动力
实战研究暴露两个结构性问题：
1. 获取层做质量判断（quality_score 打分、content_filter 过滤）脆弱——噪声词库打地鼠、相关性规则判不准。
2. 图谱按「数据形态」分模板（观点辨析/数据核对/机制解析/关系映射），是表面格式非认知本质；缺事实层、观点无归属、关联非边。

### 决策
- **获取层只搬运**：去掉 content_filter/quality_score 判断；只抓 + 提取（trafilatura favor_precision）+ 记客观元信息（来源域名/正文长度）。判断全交识别层 LLM。
- **图谱认知分层**：节点 = 产出层（核心判断）+ 结构层（事实 F / 观点 O 节点 + 关联 R 边）。事实/观点是节点，关联是边（[[双链]]）。观点强制带归属+信度。template-catalog 的 4 模板 → 融合节点结构。
- **三阶段消化**：逐页填结构（subagent）→ 节点级综合判断（subagent）→ 全局熔炼（主 LLM，smelting）。
- **双语搜索**：默认中英双语（引英文深度源），除非主题纯中国本地。
- **提取层 precision**：trafilatura favor_precision=True 结构化去噪（实测：噪声页丢弃、页脚去噪、英文无损），替代手写噪声词库。

### 教训
- 便宜的判定不准（规则），准的判定贵（LLM）——把判断移到 LLM 当场做（识别时），获取层只做客观能做的（提取 + 记元信息）。
- 图谱的价值在「连」（边）不在「分桶」——关联是边，不是第三类节点。
- 认知层次（事实/观点/关联）比数据形态（观点密集/数据密集）更本质。
