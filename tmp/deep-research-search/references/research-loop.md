# 研究循环

研究循环是深度研究的主体。每轮严格按"搜索采集 → 数据获取 → 数据消化"的顺序执行，一轮完成才进下一轮。

前 3 轮的任务是建框架——把主题的基本轮廓摸清楚，找到还没回答的问题。后 3 轮的任务是发现可能性——站在已有理解的全局视角上，看到框架之外还有什么值得追的东西。

---

## 前半程和后半程

| 维度 | 轮次 1-3（建框架） | 轮次 4-6（发现可能性） |
|------|-----------------|---------------------|
| 搜索词来源 | 缺口清单 | 全局审视后枚举排序选取 |
| 搜索词风格 | "XX 是什么""XX vs YY""XX 概述" | "XX 对 YY 的影响机制""XX 踩坑案例""XX 最新进展" |
| 搜索目的 | 确保每个维度都有数据 | 发现新的关联、暗线和可能性 |
| 消化方式 | 洞察提炼 + 图谱构建 + 缺口标注 | 综合式消化 + 全局审视 + 熔炼更新 |
| 消化输入 | 当轮新数据 | 当轮新数据 + 已有图谱 + 熔炼文件 |
| 方向决策 | 缺口分析直接推导 | 枚举 ≥20 个候选 → 评分排序 → 选 Top 10 |

---

## 阶段一：搜索采集

搜索引擎是数据的源头。每个方向至少 50 个有效 URL，是数据源质量的基础保障。

### 工具分工

| 职责 | 执行者 | 说明 |
|------|--------|------|
| 设计搜索词 | LLM | 需要判断力：遵循不离题规则 + 两阶段策略 |
| 写入搜索词列表文件 | LLM | 每行一个搜索词，保存到工作目录 |
| 执行搜索采集 | **serp_collect.py** | 机械执行，不阻塞 LLM |
| 检查结果 | LLM | 读 `_report.md`，决定是否补充采集 |

### 操作流程

**1. LLM 设计搜索词并写入文件**

根据当前轮次阶段选择搜索词来源：
- 轮次 1：从 `research-brief.md` 的第 1 轮搜索方向取
- 轮次 2-3：从上一轮 `round-N-plan.md` 的缺口清单取
- 轮次 4-6：从上一轮 `round-N-plan.md` 的"选中方向（Top 10）"取

将搜索词写入 `{workdir}/urls/round-N/search-words.txt`，每行一个。

**搜索词语言（默认双语）**：每个方向产出中文词 + 英文词两组（除非 `research-brief.md` 判定主题为纯中国本地、单中文）。英文词覆盖深度源（BBC/ESPN/Athletic/Guardian），中文词覆盖本土视角。例如方向「夺冠热门」：
- 中文：`2026世界杯 夺冠热门 球队实力`
- 英文：`2026 World Cup favorites team power ranking`

**2. 执行搜索采集脚本**

```bash
python ${CLAUDE_SKILL_DIR}/scripts/serp_collect.py \
  --search-words {workdir}/urls/round-N/search-words.txt \
  --output-dir {workdir}/urls/round-N \
  --workdir {workdir} \
  --cdp-proxy http://localhost:3456 \
  --min-urls 50
```

脚本内部流程：
- 逐个搜索词 × 三个引擎（Google / Bing / Baidu）
- 每引擎：`/new`（build_search_url 构造搜索 URL）→ 人类滚动 `/input/scroll` → `/eval` 注入 `serp_extract.js` → 人类翻页 `/input/click` → 采集 URL
- 滚动翻页：每引擎同时最多 6 个搜索结果页，超过先关最早的
- 每个搜索词×引擎独立 try-catch，失败不阻塞
- 搜索前 `/targets` 快照已有 tabs，防止误关用户页面
- 调用 `url_dedup.py` 去重过滤
- 输出：`{搜索词}.raw.txt`（原始）+ `{搜索词}.clean.txt`（去重）+ `_report.md`（执行报告）

**3. LLM 检查结果**

读取 `_report.md`，判断：
- **全部达标**（每方向 ≥ 50）→ 进入数据获取
- **某些方向失败** → 重试 / 换词 / 跳过
- **某些方向不足 50** → 调整搜索词，再次跑脚本（只跑不足的）

抽样检查：从每个方向取 5-10 个 URL，看和主题是否相关。如果不到一半相关，问题可能不是数量不够，而是搜索方向本身需要调整。

### CDP Proxy 搜索相关 API（serp_collect.py 实际调用）

| 端点 | 用途 |
|------|------|
| `GET /targets` | 搜索前快照已有 tab，防误关用户页面 |
| `GET /new?url=` | 开后台 tab（搜索 URL 由 build_search_url 构造） |
| `POST /eval?target=` | 注入 serp_extract.js 提取 SERP（body 为纯文本 JS） |
| `GET /input/scroll?target=&direction=&y=` | 协议级滚动（isTrusted，触发懒加载） |
| `POST /input/click?target=`（json={x,y}） | 人类翻页点击下一页（贝塞尔轨迹 + 随机落点） |
| `GET /close?target=` | 关闭本任务 tab（TabLease 自动调用） |

> 注：`/search`、`/scroll`、`/click`（JS 版）等端点已 `@deprecated`，serp_collect 不再使用。

> 完整 API 文档见 `scripts/cdp-proxy.mjs` 顶部注释。脚本已封装这些调用，LLM 通常不需要直接操作。

## 阶段二：数据获取

URL 的内容是消化的原材料。先批量获取大部分，再用浏览器补漏。

### 工具分工（主备分工，不可改为全浏览器）

> **固化原则（D9）**：内容获取 = **并发直接 HTTP（url_fetch，默认）** + **浏览器兜底（cdp_fetch，仅：登录墙 / 需点击交互 / 反爬403 / JS渲染）**。
> 不要改成"全走浏览器"——那样既慢又浪费登录态优势。浏览器只补漏。

| 职责 | 执行者 | 说明 |
|------|--------|------|
| 批量 URL 获取（主） | **url_fetch.py** | 并发直接 HTTP + trafilatura/BS4 两级提取，快速 |
| 失败 URL 兜底（备） | **cdp_fetch.py** | CDP 浏览器渲染 + 人类交互（展开/滚动）+ 登录态，**并发**兜底（--concurrency 5-8） |

### 操作流程

**第一步：url_fetch.py 批量获取**

```bash
python ${CLAUDE_SKILL_DIR}/scripts/url_fetch.py {workdir} \
  --round N --max-per-dir 15 --concurrency 5 --delay 0.3 --retries 2
```

脚本内部流程：
- 读取 `urls/round-N/*.clean.txt` 的 URL 列表
- 预过滤：跳过二进制文件、视频平台、搜索引擎页面
- 并发下载（requests + Chrome UA 伪装）+ HTTP 状态码分类
- 两级提取：trafilatura → BS4 降级 → 质量评分（满分100，≥30分通过）
- 跳过已有文件（断点续传）
- 可重试错误（429/502/503）自动重试，反爬（403）标记为 CDP 候选
- 输出：`data/round-N/{方向}-{序号}.md`（每个页面一个文件）
- 生成：`_report.md`（获取统计 + 错误分类）+ `_failed_urls.txt`（CDP候选列表）

存储格式：
```markdown
URL: {url}
搜索方向: {方向}
工具: url_fetch (trafilatura|bs4_fallback)
质量分: {score}/100
质量判断: 通过/低质量
标题: {title}

# {title}

{正文内容}
```

**第二步：cdp_fetch.py 兜底补获取**

url_fetch.py 完成后，对反爬（403）和 JS 渲染失败的 URL 用 CDP Proxy 补抓。

```bash
python ${CLAUDE_SKILL_DIR}/scripts/cdp_fetch.py {workdir} \
  --round N --max 20 --concurrency 5
```

脚本内部流程：
- 读取 `_failed_urls.txt`，筛选 CDP 候选（anticrawl / extract_failed / retryable）
- 逐个 URL：CDP `/new` 打开 → 等待加载 → `/eval` 注入 DOM 提取 JS → 获取正文 → `/close` 关闭
- DOM 提取不依赖外部 CDN，纯浏览器原生 API
- 输出格式与 url_fetch.py 完全一致（下游消化无需区分来源）
- 从 url_fetch.py 产出之后接续编号，不覆盖已有文件
- 生成 `_cdp_report.md`（兜底统计）

所有内容完整存储，不做分级。提前筛选可能遗漏关键信息，筛选留给消化阶段。

## 阶段三：数据消化

消化是研究的心脏。不是搬运信息，而是从数据中提炼认知。

### 消化机制

- 一个节点簇 = 一个 subagent（簇 = 直接相连的节点归入同一组）
- subagent 逐页读取数据文件，综合提炼后直接用 Write/Edit 工具写入图谱节点
- subagent 并发 ≤ 2
- 提示词模板见 `references/graph-refine-prompt.md`

---

### 前 3 轮：建框架

目标：快速搭起知识骨架，标注缺口。

**第 1 步：为每个节点簇启动 subagent（分析模式）**

将图谱节点按关联关系分组为簇，每个簇启动一个 subagent。
subagent 读数据，提炼洞察，构建节点。思考方式从"提取事实"转为"提炼认知"。

**第 2 步：LLM 汇总 subagent 产出**

所有 subagent 完成后，LLM：
1. 读取各节点文件，检查洞察质量和覆盖度
2. 对照 `research-brief.md` 研究问题清单做覆盖度评估
3. 检查各 subagent 发现的缺口
4. 把 subagent 发现的"有意思但和当前方向不直接相关"的信息记入 `research-brief.md` 的"意外追踪"字段

**第 3 步：缺口分析**

对照 `research-brief.md` 研究问题清单：
- ✅ 已覆盖 / ⚠️ 部分覆盖 / ❌ 未覆盖
- 从 ⚠️/❌ 推导下一轮搜索方向（含具体搜索词，遵循不离题规则）

**产出** `round-N-plan.md`：
```markdown
# 第 N 轮消化产出

## 图谱状态
（各节点的核心判断概要）

## 缺口分析
| 研究问题 | 状态 | 缺口说明 |
|---------|------|---------|

## 下一轮搜索方向
（缺口推导的搜索词，遵循不离题规则）
```

---

### 第 3 轮收尾：首次熔炼

第 3 轮消化完成后、写 round-3-plan.md 之前，主 LLM 做一件事：**站在所有节点之上，看一遍全局。**

读取所有图谱节点（graph/ 下全部文件），不逐个看细节，而是看整体——节点之间有什么关联、哪里有跨节点的模式、所有信源共同回避的是什么。

产出 `{workdir}/smelting.md`，这是研究熔炼文件，结构如下：

```markdown
# 研究熔炼：{主题}

## 全景判断
（3-5 句话。关于这个主题，最核心的理解是什么。
 不是某个节点的判断，是站在所有节点之上的顶层判断。）

## 跨节点暗线
（在多个节点之间浮现的关联和模式。
 不属于任何单一节点，只有从全局视角才能看到的联系。
 每条暗线说明：涉及哪些节点、关联是什么、为什么值得注意。）

## 核心张力
（贯穿整个研究的最关键争议。
 不是某个节点的未解张力，而是全局性的根本分歧。
 说明：对立的双方各持什么立场、各自的证据、为什么这个分歧重要。）

## 共享沉默
（所有信源都不讨论但你认为可能很重要的东西。
 只有在遍历了大量来源之后才能发现。
 说明：沉默的是什么、为什么你觉得它可能重要、有什么蛛丝马迹。）

## 理解地图
（当前理解的全貌。不是节点列表，而是"我们知道了什么"的结构化表述。
 用文字描绘这幅地图：核心板块有哪些、板块之间的关系是什么、
 哪里是高地（理解扎实）、哪里是洼地（理解薄弱）、哪里是空白（还没碰过）。）

## 理解演进
| 轮次 | 关键变化 | 变化来源 |
|------|---------|---------|
| 3 | 初始熔炼 | 首次全局综合 |
```

写完 smelting.md 之后，再按前 3 轮的正常流程写 round-3-plan.md（缺口分析 + 第 4 轮搜索方向）。第 4 轮的搜索方向仍然走缺口机制，因为这是第一次做全局审视，还没有到枚举排序的时机。

---

### 后 3 轮：发现可能性

后 3 轮和前 3 轮的根本区别：前 3 轮只看当轮数据，后 3 轮看全局。每次消化完之后，不是直接看缺口就决定下一轮搜什么，而是退后一步，看看全貌，再决定最值得追的方向是什么。

后 3 轮的完整流程：**消化 → 全局审视 → 熔炼更新 → 搜索方向枚举排序选取**

#### 第 1 步：综合模式消化（subagent）

和前 3 轮不同，后 3 轮的 subagent 不是只看自己的数据——它要带着上下文来消化。

每个 subagent 启动时，除了传给它负责的节点簇和当轮数据文件，还要传给它相邻节点的文件路径。subagent 先读自己的已有节点和相邻节点，再读新数据，然后问自己：

- 新数据是否改变了已有洞察？怎么变的？
- 新数据和相邻节点的洞察之间，有没有之前没看到的关联？
- 有没有信息在多个地方以不同形式反复出现？这种重复意味着什么？
- 有没有一条信息不属于任何现有节点，但它"好像很重要"？

如果新数据推翻了已有洞察，不要修补它——推翻它，标注"已被推翻"，记录推翻的证据。推翻比修补更有价值。

subagent 的产出：更新后的图谱节点 + 本轮消化发现的暗线和异常。

#### 第 2 步：全局审视（主 LLM）

所有 subagent 完成后，主 LLM 退后一步看全貌。

**读什么：**
1. `smelting.md`（上一轮的全局理解）
2. 本轮新增或更新的图谱节点（不是所有节点，只读本轮 subagent 改动过的）
3. subagent 的消化产出（本轮发现的暗线和异常）

**不读什么：** 原始数据文件、之前轮次的节点（它们已经压缩在熔炼文件中）。

**想什么：** 读完后，必须逐一回答以下六个问题。回答的过程就是深度思考的过程，不能跳过：

1. **不安**：看着更新后的全局理解，哪个判断最让你不安？为什么？
2. **冲突**：本轮新数据和已有全局理解之间，最大的冲突是什么？
3. **唯一方向**：如果你只能再追一个方向，你会追哪个？为什么是这个而不是别的？
4. **被搁置的线索**：有没有一个你一直想追但总是排在后面的线索？它为什么重要但总是被排后面？
5. **意外**：本轮发现的最让你意外的信息是什么？它有没有指向一个你还没考虑过的维度？
6. **重复**：有没有什么信息在多个节点以不同形式反复出现？这种重复意味着什么？

这些回答写入 `round-N-plan.md` 的"全局审视"章节。

#### 第 3 步：更新熔炼文件

根据全局审视的发现，更新 `smelting.md`：

- 全景判断是否需要修正？→ 更新
- 发现了新的跨节点关联？→ 追加到"跨节点暗线"
- 冲突触及了核心争议？→ 更新"核心张力"
- 注意到了新的共享沉默？→ 追加
- 理解地图发生了变化？→ 更新
- 在"理解演进"表中增加本轮记录

用 Edit 工具增量更新，不覆盖已有内容。

#### 第 4 步：搜索方向枚举 + 排序 + 选取

基于全局审视的发现（特别是第 3、4 题的回答），枚举下一轮的搜索方向。

**枚举：从六个来源分别出方向，总量 ≥ 20 个**

| 来源 | 说明 | 举例 |
|------|------|------|
| 缺口清单 | 研究问题清单中仍未回答的 | 前 3 轮标注的 ❌ 条目 |
| 图谱薄弱 | 数据不够或洞察不够深的节点 | 只有 1-2 条证据的洞察 |
| 跨节点暗线 | 熔炼文件中发现的关联，需进一步验证 | "节点 A 的趋势似乎和节点 B 有关" |
| 异常信号 | 前几轮中"无处安放但有意思"的信息 | research-brief.md 意外追踪中的条目 |
| 相邻领域 | 与主题相关但未被纳入框架的领域 | 从未搜索过的关联领域 |
| 理解演进 | 理解的变化引出了什么新问题 | 某个判断被推翻后的连锁反应 |

来源多样性比数量更重要。20 个全来自缺口的候选，不如 15 个来自 5 个不同来源的候选。

**排序：三个维度评分（1-10 分）**

| 维度 | 权重 | 怎么判 |
|------|------|--------|
| 发现潜力 | 40% | 从来没搜索过的领域 → 高；和之前高度重叠 → 低 |
| 认知杠杆 | 40% | 影响多个节点、触及核心张力 → 高；只影响单一节点 → 低 |
| 信息可获取性 | 20% | 信息丰富的领域 → 高；冷门难找 → 低。作为软约束，不硬卡 |

综合得分 = 发现潜力 × 0.4 + 认知杠杆 × 0.4 + 可获取性 × 0.2

**选取：按综合得分降序，取 Top 10**

**产出** `round-N-plan.md`：
```markdown
# 第 N 轮消化产出

## 全局审视
（六个反思性问题的回答）

## 熔炼更新要点
（本轮更新了 smelting.md 的哪些部分，变化是什么）

## 搜索方向枚举

| # | 方向 | 来源 | 发现潜力 | 认知杠杆 | 可获取性 | 综合得分 |
|---|------|------|---------|---------|---------|---------|
| 1 | ... | 缺口 | 8 | 7 | 9 | 7.8 |
| 2 | ... | 暗线 | 9 | 8 | 6 | 8.0 |
| ... | ... | ... | ... | ... | ... | ... |

## 选中方向（Top 10）
1. #{序号}: {方向描述}（得分 {X.X}）
2. ...
```

## 轮次进度记录

每轮消化完成后更新 `scores.md`：

```markdown
# 研究进度

## 第 N 轮
- 状态：已完成
- 搜索方向数：X 个
- 采集 URL 数：Y 个（去重后）
- 获取页面数：Z 个
- 图谱节点数：M 个，洞察数：K 个
```
