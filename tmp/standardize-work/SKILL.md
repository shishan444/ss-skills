# 正确工作范式

## 核心目标

与用户共同判断一项工作是否具备五个成立条件：

1. **局面成立**：真正需要改变的现实明确，并且值得现在处理。
2. **结果可验**：目标结果、完成证据和必要边界明确。
3. **路径可信**：行动能够通过可解释的因果路径带来目标结果。
4. **风险可控**：关键不确定性能够被提前发现、验证和处理。
5. **行动闭环**：责任、决策、当前行动、检查和调整机制明确。

不要把框架当成固定问卷。先判断闸门是否成立；只有闸门答案存在缺口时，才调用对应展开项的引导问题。

开始工作时，完整读取 [references/work-framework.md](references/work-framework.md)。创建主题记录时，使用 [assets/topic-record-template.md](assets/topic-record-template.md) 作为结构基线。

## 方法结构

| 判断闸门 | 核心问题 | 不清楚时才展开 |
| --- | --- | --- |
| 局面成立 | 真正需要改变的现实是什么，为什么值得现在处理？ | 表面任务、真实问题、对象、目的、价值 |
| 结果可验 | 做成什么样，看到什么才算真的发生？ | 交付物、成功标准、价值证据、范围、取舍 |
| 路径可信 | 凭什么认为这些行动能够带来目标结果？ | 工作路径、里程碑、依赖、使用场景 |
| 风险可控 | 哪个关键判断最可能错，如何尽早发现？ | 假设、最小验证、风险、升级条件 |
| 行动闭环 | 现在谁决定、谁行动，何时根据什么调整？ | 角色、状态、待决策事项、检查点、验收、结果回看 |

“可复用经验”不属于当前工作成立的必要条件。只在工作结束、关键判断被修正或出现明显得失时，作为可选复盘项调用。

## 不可跳过的约束

1. 先问或复述当前闸门的核心问题，不从展开项开始机械提问。
2. 用户已有信息足以回答时，直接整理并校验，不重复询问。
3. 只有出现具体逻辑缺口时，才选择能够解释缺口的展开项。
4. 每轮通常提出 1—3 个紧密相关的问题；不得一次抛出全部展开项。
5. 收到回答后先检查清晰度，不因用户已经作答就自动进入下一闸门。
6. 当前回答存在指代不明、表述空泛、因果断裂、口径冲突、无法验证、权责不清或关键假设未标明时，继续追问当前缺口。
7. 达到“足以支持后续判断或行动”的程度后立即停止，不追求穷尽所有细节。
8. 明确区分已确认事实、用户判断或假设、模型推断、尚未消除的不确定性。
9. 未经用户确认的内容不得记录为确定事实。模型推断必须写明依据并等待确认。
10. 新判断推翻旧判断时，保留修正轨迹，不静默覆盖。
11. 不把原始对话逐字堆入记录，只保留有承重作用的信息、判断变化和决策依据。
12. 澄清阶段可以指出矛盾、因果缺口并要求证据；完整方案分析应在用户确认总体理解后给出。

## 闸门判定标准

一个闸门只有同时满足以下条件才能通过：

- **对象明确**：谁、什么事情、什么范围和什么时点没有关键歧义。
- **内容具体**：没有依赖“做好、提升、全面、尽快”等无法行动的表述。
- **关系成立**：关键因果、依赖、证据或权责关系能够解释。
- **状态可辨**：事实、假设、推断和未知得到区分。
- **足以行动**：答案已经能够支持下一步判断、验证、决策或执行。

结果、里程碑、价值证据、验证条件和闭环条件还必须可观察或可检查。

## 如何选择展开项

先指出核心问题中的具体缺口，再选择展开项：

- 不知道任务要求与真实问题是否一致：展开“表面任务、真实问题”。
- 不知道为谁、为何现在做、做完有何改变：展开“对象、目的、价值”。
- 不知道交付什么或何时算完成：展开“交付物、成功标准”。
- 不知道完成是否产生实际效果：展开“价值证据”。
- 内容不断扩大或要求发生冲突：展开“范围、取舍”。
- 只有待办清单，无法解释为何有效：展开“工作路径、使用场景”。
- 不知道阶段是否结束或被外部条件阻断：展开“里程碑、依赖”。
- 方案依赖未经证实的判断：展开“假设、最小验证”。
- 存在可能阻止目标实现的事件或超出执行者权限的变化：展开“风险、升级条件”。
- 不知道谁负责、现在卡在哪里或需要谁决定：展开“角色、状态、待决策事项”。
- 不知道何时检查、由谁结束或是否产生效果：展开“检查点、验收、结果回看”。

具体问法和各展开项的澄清标准见参考文件。不要直接向用户询问“你的范围是什么”“有哪些依赖”等抽象问题。

## 运行流程

### 第一步：识别主题并建立记录

从用户描述中提取简洁、稳定的主题名称。若无法命名，将主题边界作为首轮问题；一旦能够命名，立即建立临时记录。

默认记录位置：

```text
<项目根目录>/01inbox/正确工作范式-<主题名称>-临时记录.md
```

处理文件名时移除 `/\:*?"<>|` 等不安全字符。若 `01inbox/` 不存在，在当前工作目录创建记录；不要擅自创建知识库分类。

创建前搜索相同或高度相似主题：

- 找到未完成记录：读取并继续，不重复创建。
- 找到已确认记录但用户明确开始新一轮：创建带日期或序号的新记录。
- 无法判断是否同一主题：在当轮问题中最小确认。

若环境无法写入文件，明确告知用户，并在对话中维护同等结构的记录。

### 第二步：定位当前闸门

读取用户信息和已有记录，判断最早尚未成立、且会影响后续大面积判断的闸门。

遵循以下原则：

- 可以直接吸收用户已经主动说明的后续信息，先标记为“待确认”。
- 不为了固定顺序重复询问。
- 如果后续讨论暴露前一闸门的逻辑断点，返回修正。
- 用户只要求解决某个局部问题时，可以只处理相关闸门，但要说明未覆盖的范围。

### 第三步：先作闸门判断

围绕当前闸门的核心问题：

1. 复述已经知道的答案。
2. 指出尚不能成立的具体位置。
3. 判断是否需要展开。
4. 若无需展开，将闸门标为“已通过”。
5. 若需要展开，只选择当前缺口对应的 1—2 个展开项。

不要因为展开项存在，就默认逐项询问。

### 第四步：按需展开并收敛

从参考文件中选择现实语言的入口问题，每轮提出 1—3 个。

收到回答后：

1. 将内容分类为事实、用户判断或假设、模型推断和不确定性。
2. 检查回答之间及与既有记录之间是否矛盾。
3. 简短复述当前理解。
4. 对照展开项的澄清标准判断是否足够。
5. 若已足够，停止该展开项；若仍有缺口，只追问缺失部分。
6. 重新判断当前闸门能否通过。

若用户暂时无法提供关键信息，记录为未决项，并明确：

- 缺少什么；
- 由谁补充；
- 最晚何时需要；
- 缺失会影响什么；
- 是否存在临时替代口径。

### 第五步：维护记录

每轮对话后更新主题记录，至少维护：

- 当前闸门、整体状态和下一步；
- 各闸门的当前答案、状态与逻辑缺口；
- 已调用的展开项及结论；
- 已确认事实；
- 用户判断与假设；
- 模型推断及依据；
- 未决问题、歧义和冲突；
- 决策、备选方案、取舍理由和确认人；
- 被否定或修正的判断及原因；
- 下一轮计划处理的缺口。

展开项状态只用于后台诊断，不把 24 项完整列表展示给用户，除非用户明确要求查看。

### 第六步：总体理解确认

五个闸门全部通过，或剩余不确定性不妨碍推进时，生成“总体理解确认稿”，至少包括：

- 当前现实、真实问题、对象、目的和价值；
- 目标结果、完成证据、效果证据与边界；
- 行动到结果的因果路径、阶段标志和依赖；
- 关键假设、验证办法、主要风险与升级条件；
- 当前责任、状态、待决策事项、下一步和反馈闭环；
- 已确认事实；
- 用户判断与假设；
- 模型推断；
- 剩余但不妨碍推进的不确定性。

请用户确认理解是否准确、有无遗漏或错误。将记录状态改为“等待总体确认”。

若用户修正：

1. 更新相关闸门和展开项；
2. 保留修正前后变化及原因；
3. 检查是否影响其他闸门；
4. 重新提交受影响部分确认。

### 第七步：确认后分析

只有用户明确确认总体理解后，才输出完整分析和建议。内容包括：

1. 五个闸门构成的工作全局图；
2. 从当前现实到最终价值的主因果链；
3. 关键逻辑断点、冲突和薄弱假设；
4. 最可能导致返工或失败的风险；
5. 最值得优先验证的事项及理由；
6. 可能被忽略的利益相关者、约束、替代方案和二阶影响；
7. 具体下一步行动、负责人、建议时点和完成标志。

明确区分基于用户信息的结论与模型分析建议，并说明推理链条。

完成后：

- 将记录状态改为“已确认并完成分析”；
- 写入用户确认时间；
- 写入最终分析摘要和下一步行动；
- 保留文件，不自动删除或移动。

工作结束、关键判断被修正或出现明显得失时，可以调用“可复用经验”复盘，但不得阻碍当前工作闭环。

## 每轮对话格式

优先使用以下结构：

```markdown
我目前的理解：
<复述当前闸门已经明确的内容>

还不能成立的地方：
<指出一个具体缺口；没有缺口时说明当前闸门已通过>

这一轮需要弄清：
1. <现实语言问题>
2. <可选问题>
3. <可选问题>
```

不要把内部完整检查表整段展示给用户。用户询问进度时，按“已通过、正在判断、尚未处理、未决项”汇报五个闸门。

## 质量底线

最终理解必须让一个具备基本背景、但没有参与前期讨论的人回答：

- 为什么这件事值得做；
- 要改变谁的什么现实；
- 最终结果是什么，凭什么判断完成和产生价值；
- 行动为什么可能有效；
- 最大的不确定性和风险是什么；
- 谁负责、谁决定、谁执行、谁验收；
- 当前在哪里，下一步是什么；
- 根据什么证据继续、调整、停止或升级。

若其中任何一项无法回答，不得声称工作全局已经清楚。
