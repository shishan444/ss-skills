# 入口点识别与流程筛选策略

## 入口点类型与信号

### HTTP Handler

#### Python 后端

| 框架 | 信号模式 | 提取信息 |
|------|----------|----------|
| FastAPI | `@router.get/post/put/delete/patch("path")` | HTTP 方法 + 路径 + handler 函数名 |
| Flask | `@app.route("path")` 或 `@bp.route("path")` | HTTP 方法 + 路径 + view 函数名 |
| Django | `urls.py` 中的 `path("url", view)` 或 `re_path` | URL 模式 + view 名称 |
| Tornado | `Application([(path_pattern, Handler)])` | URL 模式 + Handler 类名 |

#### 前端 Web

| 框架 | 信号模式 | 提取信息 |
|------|----------|----------|
| React | `<Route path="..." element={...}/>` 或 `useRoutes([...])` | 路由路径 + 页面组件名 |
| Vue | `routes: [{ path, component }]` | 路由路径 + 组件名 |
| Next.js | `app/**/page.tsx` 或 `pages/*.tsx` | 路由路径 + 页面名 |
| Angular | `RouterModule.forRoot([...])` | 路由路径 + 组件名 |

### 定时任务（Python）

| 信号模式 | 提取信息 |
|----------|----------|
| `@celery.task` / `@app.task` | 任务名 + 调度配置 |
| `@scheduler.scheduled` / `APScheduler` | 任务名 + cron 表达式 |
| Celery Beat 配置 | 任务列表 + 调度间隔 |

### 消息消费（Python）

| 信号模式 | 提取信息 |
|----------|----------|
| `@celery.task`（消息模式） | 消费者名 + 队列名 |
| `async def handle_message` / `@consumer` | handler 名 + topic |

### 用户交互触发（前端）

| 信号模式 | 提取信息 |
|----------|----------|
| 表单提交：`onSubmit`, `handleSubmit` | 表单名 + 提交动作 |
| 核心操作按钮：需人工判断哪些按钮触发核心业务流程 | 操作描述 |

## 流程筛选标准

不是所有入口点都值得生成流程文档。按以下标准筛选：

### 纳入标准

| 优先级 | 类型 | 说明 |
|--------|------|------|
| **P1** | 核心业务流程 | 直接产生业务价值的端到端路径。如：下单、支付、注册、数据导入 |
| **P2** | 异常/降级流程 | 错误恢复、重试、降级处理。如：支付失败重试、服务降级 |
| **P3** | 管理/运维流程 | 后台管理操作、数据同步。如：数据导出、缓存刷新 |

### 排除标准

- 纯 CRUD 的简单列表/详情查看（无业务逻辑）
- 静态页面渲染
- 仅做数据透传的代理接口

### 筛选方法

对每个入口点做快速判断，回答三个问题：

1. 该流程是否跨模块？（跨模块 → 值得记录）
2. 该流程是否包含数据转换/状态变迁？（包含 → 值得记录）
3. 该流程是否有异常处理分支？（有 → 值得记录）

**判定规则**：三个问题中有 2 个以上"是" → 生成流程文档。

**数量控制**：P1 全部纳入，P2/P3 按重要性选择，流程文档总数通常不超过 10 个。

## 流程追踪方法

从入口点开始，追踪数据在系统中的流转：

1. **标记入口**：确定触发入口的 handler 函数/组件
2. **追踪调用链**：从 handler 沿函数调用链向下追踪
   - Python：函数调用 → service → repository → model
   - 前端：组件 → hook/API 调用 → 后端 API
3. **记录数据转换**：每个节点的输入数据 → 处理逻辑 → 输出数据
4. **识别状态变迁**：是否有数据状态发生变化
5. **标注边界**：追踪到外部系统（数据库、第三方 API）时停止

### 追踪深度限制

- 最深追踪 5 层调用（避免无限追踪）
- 跨系统边界时停止（如前端调用后端 API → 在 API 边界记录"调用后端接口 X"，不在前端流程中展开后端内部实现）
