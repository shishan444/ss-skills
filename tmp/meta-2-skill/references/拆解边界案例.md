# 落位边界案例

## 案例一：内容明显属于 references/，但骨架阶段不知道具体内容

> **情况**：骨架中某个步骤需要引用一份检查清单（如代码审查的检查项），清单内容需要领域知识才能编写，骨架阶段无法确定。

处理方式：在 SKILL.md 的步骤中标注引用位置和预期用途，同时在完善指引中标注"需要创建 references/checklist.md"。

```
步骤 3：按 references/checklist.md 中的检查项逐项执行
        [待完善：创建 references/checklist.md——定义每个维度的检查项清单]
```

## 案例二：内容可内联也可抽文件，看信息量决定

> **情况**：一段判断逻辑，既可以直接写在 SKILL.md 的步骤旁边（3-4 行），也可以抽成独立的 reference 文件。

判断依据：信息量。如果 3 行以内能说清 → 内联在 SKILL.md。如果超过 3 行或未来可能独立变更 → 抽成 references/ 文件。

## 案例三：识别出需要 scripts/ 或 assets/，但骨架阶段不知道具体实现

> **情况**：目标 Skill 的流程中某个步骤是一个机械操作（如目录初始化、文件格式转换），适合写成脚本。但骨架阶段还没设计脚本的具体实现。

处理方式：在完善指引中标注"需要创建 scripts/xxx.sh——功能是 Y，输入是 Z，输出是 W"。骨架阶段只识别需求，不实现。

## 案例四：不同类型骨架的 references 模式

> **流程型**倾向于需要：检查清单、判断标准文件、输入输出格式说明
> **创作型**倾向于需要：模板、范例、风格指南、评分标准
> **对话引导型**倾向于需要：提问库、对话策略集、阶段切换条件说明

---

## 大方向落位速查口诀

1. 先问：这块内容是跟着流程走的还是被流程引用的？（跟着走 → SKILL.md；被引用 → references/）
2. 再问：跟着走的内容，3 行以内能说清吗？（能 → 内联；不能 → 抽文件）
3. 最后问：是不是可自动化的机械操作？（是 → scripts/）
