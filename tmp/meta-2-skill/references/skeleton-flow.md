# 流程型骨架构建指南

## 识别特征

- 有明确的输入和输出
- 中间步骤可枚举
- 从输入到输出单向推进，不存在迭代回环
- 完成标准可客观验证

## 骨架构建方法：主路径 + 分支点 + 约束锚定

### 第一步：提取主路径

从方向声明中提取从输入到输出的主干步骤序列。每个步骤用一句话说明做什么。此时不考虑分支和异常，只走最顺的那条线。

### 第二步：标注分支点

遍历主路径，识别"走到这里可能需要分岔"的位置：
- 分岔条件是什么？
- 每个分支走向哪里？
- 如果分岔条件的判断依据尚未明确，标注 `[待完善：...]`

### 第三步：锚定约束

将方向声明中的约束标记到它生效的步骤旁边。约束写在执行者恰好需要知道它的位置——不是收在文末。

### 产出形态

一个带分支条件的步骤序列，可能包含 `[待完善：...]` 标注。

## 完整推演示例：代码审查 Skill

### 方向声明

> **触发**：用户指定一个或多个源代码文件路径时调用。不触发：测试文件、配置文件。
> **输入**：文件路径（必须）+ 审查范围（可选，缺省全部）
> **输出**：按严重程度分级的结构化审查意见
> **约束**：不能修改源代码；审查维度至少包含命名、结构、安全三类

### 骨架产出

```markdown
## 做什么

当用户指定源代码文件路径时，产出一份按严重程度分级的结构化审查意见。
不触发：测试文件、配置文件、构建脚本。

## 怎么做

1. 接收文件路径，读取源代码内容
2. 确定审查范围
   - 如果用户指定了范围 → 按指定范围执行
   - 如果未指定 → 对全部代码执行审查
3. 对每个审查维度执行检查
   - 检查维度：命名、结构、安全（至少覆盖这三类）
   - 每个维度的检查项清单 `[待完善：定义每个维度的具体检查项——命名维度查什么、结构维度查什么、安全维度查什么]`
   - 检查时的判断标准 `[待完善：判断代码是否符合标准时，依据是什么——是项目 lint 配置还是语言社区规范，如果用户提供了 lint 配置则优先使用]`
4. 对每条审查意见判定严重程度
   - 严重程度分级 `[待完善：定义分级标准——什么算阻塞、什么算警告、什么算提示]`
5. 为每条意见生成修改建议（包含行号和具体修改方案）
6. 汇总输出审查报告，确认三个维度均已覆盖

## 完善指引

1. [高] 步骤 3 的检查项清单 —— 需要为命名/结构/安全三个维度各列出具体检查项，可参考 ESLint/SonarQube 的规则分类，补全后可验证每个维度至少有 5 个检查项
2. [高] 步骤 3 的判断标准 —— 需要明确"有 lint 配置时用配置、没有时用什么默认标准"，补全后执行者可在步骤 3 直接做判断
3. [高] 步骤 4 的严重程度分级标准 —— 需要定义阻塞/警告/提示的具体判断依据，补全后每条意见的分级可客观判定
4. [中] 步骤 1 的文件类型判断 —— 需要补充如何识别"测试文件/配置文件"不触发场景的具体规则
5. [低] 审查报告的输出格式 —— 需要定义报告的模板结构，补全后输出物格式统一

## 不可破

- 不能修改源代码
- 审查维度至少包含命名、结构、安全三类
- 每条审查意见必须标注行号和修改建议
```

## 常见留白位置

流程型骨架中最常出现留白的典型位置：

- **分支条件的细化**：主路径上的分岔点识别出来了，但"什么条件下走哪条分支"的判断依据不够具体
- **输入格式的验证规则**：知道要接收输入，但输入的格式、边界条件未定义
- **输出质量标准**：知道要产出什么，但"做到什么程度算合格"的标准缺失
- **异常处理策略**：主路径走通了，但异常情况的回退或兜底动作未标注
