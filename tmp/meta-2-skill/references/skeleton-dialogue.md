# 对话引导型骨架构建指南

## 识别特征

- 核心是多轮交互——Skill 的执行过程就是一系列对话
- 每轮根据用户的回应调整方向，路径不可完全预设
- 没有固定的步骤序列，但有固定的对话结构
- 完成标准通常是"用户的需求/想法已经被充分澄清或推进"

## 骨架构建方法：循环结构 + 推进条件 + 收敛标准

### 第一步：定义循环结构

识别每轮对话的基本模式——每轮要完成什么：
- 收集什么信息或理解什么状态
- 做什么判断或分析
- 推进什么——输出什么给用户、引导用户往哪个方向
- 循环入口和出口分别在哪里

### 第二步：识别推进条件

定义对话阶段的切换规则：
- 什么信号表明可以从当前轮推进到下一阶段
- 什么信号表明需要回退或调整方向
- 推进条件如果不够明确，标注 `[待完善：...]`

### 第三步：定义收敛标准

定义对话何时结束：
- 什么状态下产出最终结果
- 收敛的信号是什么
- 产出物是什么形态

### 产出形态

一个带推进条件和收敛标准的对话循环结构，可能包含 `[待完善：...]` 标注。

## 完整推演示例：需求澄清 Skill

### 方向声明

> **触发**：用户提出一个模糊的需求想法，需要通过对话澄清具体目标时
> **输入**：用户的初始表述（自由文本）
> **输出**：一份经用户确认的需求方向声明
> **约束**：对话不超过 5 轮；每轮只问 1-2 个问题

### 骨架产出

```markdown
## 做什么

当用户提出模糊需求时，通过多轮对话澄清需求方向，产出一份经用户确认的需求方向声明。
不触发：需求已经足够具体、可直接执行的情况。

## 怎么做

### 初始轮：理解入口

1. 接收用户的初始表述
2. 识别表述中的核心意图和含混点
3. 输出：用自己的话复述理解，提出 1-2 个最关键的澄清问题

### 循环体：澄清与收敛

每轮对话执行以下步骤：

1. **分析回应**：理解用户回答中明确的部分和仍然模糊的部分
2. **判断状态** `[待完善：定义"澄清程度"的判断标准——哪些维度必须明确才算澄清到位]`
   - 已充分澄清 → 进入收敛轮
   - 仍有模糊点 → 继续
3. **选择追问方向**：从仍模糊的维度中选择最关键的 1-2 个继续追问
4. **输出回应**：确认已明确的部分 + 提出新的追问

### 收敛轮：产出方向声明

1. 汇总所有已澄清的信息
2. 组织为方向声明格式（触发/输入/输出/约束）
3. 交给用户确认
   - 用户确认 → 完成
   - 用户修正 → 回到循环体继续澄清

### 推进条件

- 从初始轮到循环体：用户对第一个复述做出了回应
- 从循环体到收敛轮：核心维度的澄清程度达标 `[待完善：核心维度有哪些，达标标准是什么]`
- 触发回退：用户的回答与之前的表述矛盾

### 收敛标准

- 所有核心维度已澄清
- 用户对方向声明逐条确认

## 完善指引

1. [高] 循环体步骤 2 的澄清程度判断标准 —— 需要定义哪些维度是必须澄清的（场景、目标、边界、约束等），以及每个维度"足够清晰"的判定线，补全后可客观判断何时进入收敛
2. [高] 推进条件中的核心维度定义 —— 与上一条关联，需要明确需求澄清必须覆盖的最小维度集合
3. [中] 初始轮的澄清问题生成策略 —— 需要定义如何从模糊表述中识别"最关键的澄清点"，可参考需求工程中的提问框架
4. [低] 对话风格/语调定义 —— 需要定义追问时的语调（引导式 vs 挑战式 vs 探索式），影响用户体验

## 不可破

- 对话不超过 5 轮
- 每轮只问 1-2 个问题
- 方向声明必须经用户逐条确认
```

## 常见留白位置

对话引导型骨架中最常出现留白的典型位置：

- **每轮对话的提问策略**：知道要问，但"问什么、怎么问"的策略不够具体
- **推进条件的量化标准**：知道什么信号推进，但信号本身的判断不够可操作
- **收敛的可度量标准**：知道"充分澄清"是目标，但什么程度算"充分"未定义
- **对话风格的边界**：知道要对话，但追问的力度、语气、深度的边界未设定
