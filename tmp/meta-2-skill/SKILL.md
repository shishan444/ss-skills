# Meta-2-Skill

## 做什么

当用户说"我想做一个能完成 X 的 skill"，且 X 不能通过修改已有 skill 的局部指令解决时，用三个阶段引导用户从模糊意图走到一个结构完整的 skill 骨架目录。

**不触发的情况**：X 可以通过修改已有 skill 的参数或局部指令解决；X 是对已有 skill 的单次使用而非新建。

完成后，目标目录下存在一个 SKILL.md——其中包含骨架流程、`[待完善：...]` 留白标注、完善指引——以及所需的 references/、scripts/、assets/ 资源文件的占位。

## 怎么做

### 阶段一：理解校准

本阶段的唯一目标：把用户的模糊需求翻译为一份经用户确认的"方向声明"。未经确认不准进入阶段二。

方法是**需求收拢四问**。在对话中逐一追问，直到四个维度的方向明确：

| 问 | 你要追出什么 | 追问方向 |
|---|---|---|
| 工作问 | 这项"工作"何时变得必须做？ | "什么信号出现时，这个 skill 被触发？有没有看起来像但**不该**触发的场景？" |
| 手持问 | 执行开始时，手里有什么？ | "执行者能拿到什么信息？是自由文本还是固定格式？有什么可选但重要的附赠信息？" |
| 变化问 | 完成后世界发生了什么变化？ | "这个 skill 执行完，世界多了/少了/变了什么？" |
| 不变问 | 什么条件在执行中不可改变？ | "有没有硬性的技术限制、环境限制、时间限制？有没有绝对不能违反的规则？" |

四问走完后，将答案浓缩为**方向声明**（方向明确即可，不要求每个维度完全闭合）：

> **触发**：[什么情况下调用 / 什么情况下不调用]
> **输入**：[必须提供什么 + 可选什么]
> **输出**：[完成后世界的变化]
> **约束**：[不可违反的限制]

追问卡住时，查阅 `references/需求追问范例.md`。

**门控**：将方向声明交给用户逐条确认。确认通过则进入阶段二；用户修正则回到追问环节。

---

### 阶段二：骨架构建

本阶段的目标：从已确认的方向声明推导出一个方向正确、逻辑闭合的骨架——关键决策点和分支逻辑都有了，但细节允许留白。

#### Step 1：识别 Skill 类型

根据方向声明判断这个 Skill 的内在逻辑结构属于哪一类：

- **流程型**：有明确的输入和输出，中间步骤可枚举，从输入到输出单向推进，不存在迭代回环
- **创作型**：有阶段性，但阶段之间存在迭代回环——写完初稿后可能回到前面的阶段修改。质量标准比步骤顺序更重要
- **对话引导型**：核心是多轮交互——Skill 的执行过程就是一系列对话，每轮根据对方的回应调整方向，路径不可完全预设
- **无法归入以上类型** → 默认使用流程型框架（最通用的默认）

#### Step 2：按类型构建骨架

根据 Step 1 的类型判定，使用对应的骨架框架构建骨架。详细的推演示例见 `references/skeleton-{type}.md`。

**[流程型] 主路径 + 分支点 + 约束锚定**
1. 从方向声明提取从输入到输出的主路径步骤序列，每步一句话
2. 标注分支点：哪些步骤存在条件判断，分岔条件是什么
3. 将约束锚定到它生效的步骤旁边
产出：带分支条件的步骤序列

**[创作型] 阶段划分 + 质量门控 + 迭代回路**
1. 识别从初始素材到最终交付经历的关键阶段，定义每阶段的产出物
2. 为阶段之间的衔接点定义通过标准和不达标时的回退
3. 标注哪些阶段之间存在迭代回环，回退条件是什么
产出：带质量门控和迭代回路的阶段图

**[对话引导型] 循环结构 + 推进条件 + 收敛标准**
1. 定义每轮对话的基本模式：收集什么、判断什么、推进什么
2. 识别推进条件：什么信号触发阶段切换或方向调整
3. 定义收敛标准：什么状态下对话结束，最终产出什么
产出：带推进条件的对话循环结构

#### Step 3：标注留白

遍历骨架中的每个节点，对"逻辑必须闭合但内容尚未明确"的位置标注留白。

格式：`[待完善：具体说明需要补什么]`

要求："具体说明"必须指明缺的是什么（判断标准、分支条件、输入格式、覆盖维度等），不能只写"待完善"。

**底线**：骨架的逻辑必须可走通——从输入到输出不存在断路。但步骤内部的具体判断标准、分支条件的细化允许留白。

---

### 阶段三：落位与指引

本阶段的目标：把骨架映射到 skill 目录结构，并生成完善指引。

#### Step 1：大方向落位

对骨架内容做粗粒度归类（不逐块判定）：

- 骨架主体流程、决策点、约束 → 归入 SKILL.md
- 被流程引用的独立知识（检查清单、标准文件等）→ 标注为 references/ 文件
- 可自动化的机械操作 → 标注为 scripts/ 文件
- 被引用的静态资源 → 标注为 assets/ 文件

骨架阶段只标注文件的存在和预期用途，不必写出完整内容。落位拿不准时查阅 `references/拆解边界案例.md`。

#### Step 2：生成完善指引

紧跟在骨架末尾，按优先级列出需要完善的项目：

- `[高]`：没有它 Skill 跑不起来
- `[中]`：跑起来但质量差
- `[低]`：锦上添花

每条指引必须对应一个具体的 `[待完善]` 标注，说明：需要补什么、从哪里获取补全信息。

#### Step 3：轻量自检

以第一次执行的视角走一遍骨架：

1. frontmatter 是否完整？（文件以 `---` 开头，包含 name 和 description 字段。迭代已有 skill 时，原有 frontmatter 字段必须全部保留）
2. 骨架逻辑是否闭合？（步骤之间无断路）
3. 留白标注是否清楚？（每处都说了缺什么）
4. 指引是否可操作？（每条对应具体标注，说明补全方式）

任何一处不通过，回到对应步骤修正。全部通过才交付。

产出物格式规范见 `assets/产出框架指南.md`。

---

## 不可破

- **阶段不可跳过。** 理解校准 → 骨架构建 → 落位与指引，必须依序执行。
- **方向声明必须经用户确认。** 未经确认不准进入阶段二。
- **骨架产出必须包含留白标注和完善指引。** 没有留白说明过度填充，没有指引说明没说清怎么完善。
- **不自检不交付。** 骨架必须通过轻量自检的三项检查。
- **每种类型的骨架框架内联在本文件中。** 类型判断逻辑和三种框架方法不走 references，确保执行连贯。
