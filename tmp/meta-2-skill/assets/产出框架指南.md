# 骨架产出规范

此文件定义 meta-2-skill 产出的目标 Skill 的格式规范。产出物是骨架，不是完整可执行的 Skill——骨架的逻辑必须可走通，但细节允许留白。

---

## SKILL.md 文件结构

SKILL.md 由两部分组成：YAML frontmatter（文件头部）和 Markdown 正文（四个部分）。

### 零、YAML Frontmatter

SKILL.md 文件必须以 YAML frontmatter 开头：

```yaml
---
name: skill-name
description: 触发场景 + 核心功能。这个描述决定 Skill 何时被匹配触发。
---
```

**必填字段**：

| 字段 | 约束 | 说明 |
|------|------|------|
| `name` | kebab-case，最长 64 字符 | Skill 标识符，与目录名一致 |
| `description` | 最长 1024 字符 | 触发机制核心，决定 Claude 何时调用此 Skill |

**常用可选字段**：`disable-model-invocation: true`（仅手动触发）、`allowed-tools`（预批准工具列表）。

**写入规则**：
- **新建 skill**：根据方向声明中的触发条件编写 name 和 description
- **迭代已有 skill**：必须保留原有 frontmatter 的所有字段，仅在触发机制有明确问题时修改 description
- 无论新建还是迭代，写入 SKILL.md 时 frontmatter 必须出现在文件最开头

---

## SKILL.md 正文的四个部分

### 一、做什么

用 1-2 句话说清：在什么情况下调用这个 Skill，它最终让世界发生了什么变化。

写法：
- 前半句说触发条件（"当用户……时"），后半句说完成结果（"产出……"）
- 如果存在明显的误判场景，用"不触发"排除
- 避免使用"帮助"、"辅助"这类不可验证的词——用"生成"、"完成"、"产出"

### 二、怎么做

描述从输入到输出的骨架流程。

组织原则：
- 流程的步骤划分和分支结构由阶段二的骨架决定
- 每个步骤是一句话：做什么，不是怎么想
- 决策点附带分岔条件和判断依据。如果判断依据尚未明确，用 `[待完善：...]` 标注
- 约束写在它生效的步骤旁边，不收在文末
- 需要查阅外部文件时，写明文件路径

#### 留白标注

格式：`[待完善：具体说明需要补什么]`

要求：
- "具体说明"必须指明缺的是什么——判断标准、分支条件、输入格式、覆盖维度等
- 不能只写 `[待完善]` 或 `[待完善：补充细节]` 这种无法操作的标注
- 示例：`[待完善：定义"严重程度"的分级标准——阻塞/警告/提示的具体判断依据]`

#### 骨架的逻辑底线

骨架中的步骤必须逻辑可走通——从输入到输出不存在断路。但步骤内部的具体判断标准、分支条件的细化、输入输出的格式细节允许留白。

### 三、完善指引

紧跟在"怎么做"后面，按优先级排列。

格式：
```
完善指引
1. [高] 对应步骤 X 的 [待完善：...] —— 需要补充 ...，可从 ... 获取参考信息，补全后可验证 ...
2. [中] ...
3. [低] ...
```

要求：
- 每条指引必须对应一个具体的 `[待完善]` 标注
- 必须说明：需要补什么、从哪里获取补全信息
- 优先级判断依据：没有它 Skill 跑不起来 → [高]；跑起来但质量差 → [中]；锦上添花 → [低]

### 四、不可破

列出不可违反的硬约束。

要求：
- 每条约束是可判断的——不需要"理解上下文"就知道是遵守还是违反
- 如果某条约束只在某个步骤生效，写在那个步骤旁边，不放在这里

## 资源文件

### references/
骨架阶段只标注文件的存在和预期用途，不必写出完整内容。
示例：`references/checklist.md — 代码检查项清单，阶段二 Step 2 按需查阅`

### scripts/ 和 assets/
骨架阶段识别出需要什么类型的文件，在指引中说明。具体实现留给后续完善。

### 文件头部的查阅条件
每个 reference 文件在 frontmatter 中标注 `查阅条件`：写明在 SKILL.md 的哪个步骤或决策点下需要查阅。
