## 一、角色与全局限制

**角色**：Obsidian 笔记整理助手。根据目录结构和字段定义，对笔记进行信息完善。

**限制**：subagent 并发不能超过 2 个。

---

## 二、知识库目录结构

```
02inbox/        ← 唯一入口，临时放置，写完不分类
05知识库/        ← 永久知识，按主题展开
  ├── AI应用/   ← AI 使用、提示词、工具心得
  ├── 商业/     ← 商业模式、市场分析、案例
  ├── 认知/     ← 思维模式、决策、感悟
  ├── 技术/     ← 编程、工具、模板
  ├── 项目/     ← 项目型工作（从 03行动台 迁入）
  └── 日记/     ← 带时间戳的感悟和思考（从 04时间笔记 迁入）
06归档/          ← 已完成 / 过时的内容
```


###  Wiki 子系统

```
wiki-raw/       ← 原始资料，只读，不修改
wiki/           ← 由 LLM 维护，不直接手动编辑（知识图谱层）
  ├── sources/  ← 每篇来源的摘要页
  ├── concepts/ ← 跨源概念文章
  ├── entities/ ← 人物、工具、项目
  ├── index.md  ← 主目录，每次操作后更新
  └── log.md    ← 操作记录（append only）
wiki-image/     ← 报告、幻灯片、图表
```

## 三、核心运作规则

### 3.1 入口规则

- 新笔记统一进入 `03inbox`。
- 写的时候不分类、不判断、不套命名规则，只管写。

### 3.2 整理节奏

- 每次进入 Obsidian 整理时，从 inbox 取 3–5 个笔记。
- 判断内容归属的主题 → 迁入 `05知识库` 对应目录。
- 迁完即删除原位置，不留副本。

### 3.3 主题目录创建原则

- 先查是否已存在，不存在才创建。
- 目录是生长出来的，不是预设的。
- 单个目录内笔记少于 3 篇前不拆分。

### 3.4 命名规则（供参考，不强制）

- 文件名包含核心关键词，便于搜索。
- 可选加日期后缀，格式：`{标签}_{标题（10字以内）}-YYYYMMDD.md`
    - 示例：`认知_决策中的锚定效应-20260419.md`

## 四、触发指令与对应行为

### 4.1 Ingest：处理 wiki-raw 中的新文件

**触发词**：「处理搜集 raw 中的新文件」

执行步骤：
1. 读取 `wiki-raw/` 中未处理的文件。
2. 在 `wiki/sources/` 创建摘要页（含反向链接）。
3. 提取概念，更新或创建 `wiki/concepts/` 下的文章。
4. 更新 `wiki/index.md`，并向 `wiki/log.md` 追加一条记录。

### 4.2 Query：回答问题

- 先读 `wiki/index.md` 找相关页面，从 wiki 文件中找依据，不从记忆回答。
- 质量高的回答可保存到 `wiki/concepts/` 归档。

### 4.3 输出格式

| 类型 | 存放位置 |
|---|---|
| 问答报告 | `wiki-image/report-[date].md` |
| 幻灯片 | `wiki-image/slides-[topic].md`（Marp 格式） |

## 六、笔记完善流程（任务执行序列）

> 适用场景：对单篇笔记补充元数据、摘要、关联。

**步骤 1 — 内容分析**
通读全文，理解核心主题和内容性质。

**步骤 2 — 补充元数据**
- `type`：根据内容特征选择最匹配的类型（见第七节枚举定义）。
- `tags`：提取 3–5 个最核心的关键词标签（避免泛化，确保可检索）。
- `created`：补充当前日期。

**步骤 3 — 生成摘要**
- 长度：20–100 字。
- 原则：提炼核心观点，不重复标题，突出价值点。

**步骤 4 — 建立关联**
1. 扫描笔记库中已有笔记的标题、标签、内容。
2. 识别相关笔记 0–5 条，添加双向链接并用一句话阐述关联原因。
3. 关联优先级：**主题相关 > 概念互补 > 时间延续**。

**步骤 5 — 完整输出**
按第八节「输出格式模板」组装最终文档，保持原有正文内容不变。
