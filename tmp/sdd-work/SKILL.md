# 编程工作

所有编程任务的统一入口——修 bug、加功能、重构、写测试，都从这里开始。

## 行为哲学
所有任务都走同一条路，**先研究分析清楚工程代码，然后进行分析、设计plan 步骤序列规划，做后按步骤进行编程开发**。
先在阶段 A 研究清楚工程现状和任务目标，产出一条从现状到方案的完整推理链；等用户确认后，再在阶段 B 分析设计 plan 步骤序列规划，逐步实现。

## 你怎么工作

你的工作分成4个阶段，所有任务都必须经历这4个阶段顺序执行：

```
阶段 A（研究循环）→ 门控（用户确认推理链）→ 阶段 B（实现循环）→ 最终校验
```

阶段 A 是探索性的——收集信息、深度分析、验证假设、构建任务推理链。阶段 B 是承诺性的——基于用户确认的推理链构建task plan 去写代码。门控是两者的分界线：用户确认的不是"方案看起来不错"，而是"这条推理链我认可，可以基于它去实现"。


## 任务启动

拿到需求后，先做三件事：

**扫描项目上下文。** 加载 `references/context-scanning.md`，按指导扫描。扫描的入口是 CLAUDE.md——从中找到工程目录和工程设计文档目录。然后到 docs/ 目录下阅读工程设计文档，了解工程的内容和架构。这是理解项目的基础。如果工程已存在但没有 docs/ 目录，直接停止服务，告知用户需要先添加工程文档描述。

**理解用户意图。** 确认你理解了用户要做什么。如果任务描述有歧义，向用户提问澄清——不要带着不确定进入分析。

**创建 task 文件。** 记录任务目标和初始理解。写入文件到 `工程目录/task/{年月日}-{10个字以内容任务描述}/task.md`这个文件会在整个流程中持续更新，最终成为推理链的载体和阶段 B 的实施基准。

## 阶段 A：研究循环

这个阶段的目标是：通过结构性研究理解工程现状，产出一条完整的、经过验证的推理链。

阶段 A 分两步走：先通过研究循环建立对系统的结构性理解，再基于这份理解构建推理链。推理链有七个环节，环环相扣——任务定义 → 现状定位 → 解决策略 → 范围边界 → 行为规格 → 风险披露 → 实施顺序。每一环依赖上一环，现状定位错了后面全部建立在错误基础上。所以研究循环的质量直接决定推理链的质量。

### A1. 研究循环

加载 `references/engineering-research.md`

以循环方式深入工程代码，建立结构性理解。每轮循环包含三步：

1. **架构理解**——派出 2 个 subagent 并行研究：一个从架构层分析模块职责边界，一个从链路层追踪数据流路径。汇总两路结果，综合判断系统的骨架和流动。后续轮次改为定向 subagent，针对上一轮发现的不确定性做针对性研究。

2. **根因分析与工程验证**——基于架构理解的结构地图，定位任务的关键点。每个分析判断都必须经过工程验证（证伪优先）。验证的具体方法加载 `references/verification-guide.md`。

3. **思考引导**——回答三个问题做出决策：我现在对系统的结构性理解是什么？这个理解如何改变了我对任务的看法？还有什么不确定的东西，如果知道了答案会改变我的处理方式？第三问的答案决定循环继续（封装 subagent 定向研究）还是结束。

每轮产出写入 task 文件，追加保留，不覆盖。研究循环结束后，task 文件中包含完整的理解演化轨迹和经过验证的分析判断。

研究过程中产生的每一个判断都必须有代码证据（文件:行号）支撑——不能用"我觉得"或"大概是这样"来代替证据。

### A2. 构建推理链

加载 `references/reasoning-chain-guide.md`

研究循环完成后，把结构性理解和分析判断组织为推理链，写入 task 文件。推理链七环的含义：

1. **任务定义**——要做什么。一句话说清楚目标。让不了解项目的人读完就知道这次任务要达成什么。
2. **现状定位**——工程当前的状态是什么、关键点在哪里。从研究循环的结构性理解中提取，包含模块边界分析和数据流分析的综合判断。每一步都要有代码证据，要明确是架构设计问题、代码设计问题还是业务逻辑设计问题。
3. **解决策略**——怎么修或怎么建。策略必须直接针对现状定位的关键点，不是绕过它。说清楚选这个策略的原因，以及排除了哪些替代方案。
4. **范围边界**——改什么文件，明确不改什么。列出改动文件清单和排除项，每项说清为什么在范围内或范围外。
5. **行为规格**——系统应该表现出什么行为。不是"怎么判断完成了"，而是"系统的行为契约是什么"。包含输入输出契约、不变量、边界和异常处理、前置后置条件。每条规格标注验证方式：`[测试验证]`（核心逻辑）、`[代码审查]`（简单变更）、`[集成测试]`（多模块协作）。模糊的规格不是规格。
6. **风险披露**——可能出什么问题。区分"确定有风险"和"不确定是否有风险"，对每个风险说出影响和缓解方式，对每个不确定性说出如何消除。
7. **实施顺序**——按依赖关系和风险排定的任务序列。从行为规格提取实现单元，按依赖约束排序（被依赖的在前），依赖不约束时高风险优先。每个任务独立可验证、在一个实现循环内可完成。

每一环不允许空洞陈述。"代码写得不对"不是现状定位，"系统正常工作"不是行为规格，"可能出问题"不是风险披露。

### A3. 任务研究计策判断分析

构建完推理链后，评估它是否闭合：

- 推理链某环与研究循环的结构性理解矛盾 → 回到研究循环补充研究，修正后重新构建
- 发现了新问题 → 评估是否纳入本次任务。直接影响目标 → 纳入；独立问题 → 记录但不纳入；不确定 → 问用户
- 连续 2 轮推理链无实质变化 → 当前方向已挖到底，向用户报告已确认项和不确定项

**出口条件**：七环推理链全部有证据支撑，无未解决的矛盾。达到后才进入门控。

## 门控：用户确认推理链

加载 `references/gate-presentation.md`

把推理链呈现给用户，**结论先行**——先给七环摘要，再给支撑每环的证据，细节按需展开。对每环标注置信度（确认 / 高置信 / 待确认），让用户知道哪些环节还有不确定性。

等待用户确认。用户的反馈有三种处理方式：

- **要求修改** → 回到阶段 A 的对应环节迭代，修改后重新走门控。不能只改一处就跳过其他环节的检查——修改可能引起连锁影响。
- **确认** → 冻结 task 文件，标记推理链为冻结状态。冻结意味着阶段 B 将以此为基准，偏离必须报告。
- **用户明显没有仔细看就确认** → 简要复述最关键的风险和范围边界，确保用户知情后再冻结。

## 阶段 B：实现循环

### B1. 准备
理解 **推理链**，使用subagent 研究工程已有的测试框架和用例设计情况，评估是否需要新增或者删除用例，对测试用例框架进行迭代

### B2. 实施规划细化设计

加载 `references/implementation-guide.md`

基于冻结的推理链执行。根据推理链规划一个实施方案步骤序列。

在进入实现之前，把任务级的行为规格分解为实现级规格。对实施顺序中的每个条目：

**行为规格细化。** 推理链的行为规格是任务级的——描述整个任务的行为要求。细化到实现级：精确到函数签名、数据结构、算法描述。比如行为规格说"对输入数据做归一化处理"，细化后是"normalize 函数接收 float 数组，返回等长 float 数组，每个值在 [0,1] 区间，全零输入返回全零数组，NaN 输入抛出 ValueError"。

**测试用例设计。** 为标注了 `[测试验证]` 的行为规格条目设计具体测试用例——测什么行为、输入什么、期望什么输出、覆盖什么路径（正常/边界/异常）。设计完成后，B3 实现时直接翻译为测试代码。

### B3. 任务实施循环

加载 `references/spec-implementation.md`

对实施方案"实施顺序"中的每个条目，执行规格驱动实现循环：

**对规格实现。** 读该条目的实现级行为规格，写代码让系统表现出规格描述的行为。

**自查合规。** 逐条对照行为规格检查实现——每条契约是否满足、每个不变量是否成立、每个边界是否处理。可以用 agent 做自查。不满足的立即修正。

**按需测试验证。** 为标注了 `[测试验证]` 的行为规格写测试（用例已在 B2 设计好），运行确认通过。测试未通过 → 修正实现或测试。

**范围检查。** 确认改动没有超出推理链的"范围边界"。超范围 → 停下来报告，不继续推进。

循环直到该条目的所有行为规格满足、所有测试通过，然后进入下一个条目。

### B4. 任务实施合规确认

所有条目完成后，做全面合规确认。从三个维度检查，确认实施方案的完整性和准确性：

**覆盖完整。** 行为规格的每条都有对应的实现和验证。运行所有测试确认通过。对标注 `[代码审查]` 的行为规格条目，通过审查确认满足。没有遗漏的规格条目，也没有实现了但规格中没有的额外行为。

**意图正确。** 实现不仅"能跑通"，而且表达了规格描述的行为意图。测试通过不代表意图正确——有时候测试本身写错了，或者测试覆盖了错误的行为。对关键行为规格，审查实现代码是否真正表达了规格意图。

**策略一致。** 代码的实现方式和推理链的策略一致。推理链选了策略 X 是有原因的，如果实现悄悄用了策略 Y，推理链的风险评估和范围分析可能失效。发现策略偏移 → 评估是否需要更新推理链，不能悄悄偏移不加说明。

测试多次尝试无法通过，且排除了测试错误和实现错误 → 推理链可能有缺陷，回到阶段 A 对应的断裂环节，通知用户重新确认。


## 最终校验：推理链 vs 实际对比

加载 `references/comparison-guide.md`

阶段 B 完成后，将冻结的推理链和实际实现逐环对比。七环各有对比重点：

- **任务定义**：实际完成的工作是否是原始定义的任务
- **现状定位**：实现是否针对了推理链认定的现状关键点
- **解决策略**：实际策略是否和推理链一致
- **范围边界**：是否有超出范围的改动
- **行为规格**：所有行为规格条目是否都被满足
- **风险**：推理链披露的风险是否出现，是否有未预见的风险
- **实施顺序**：是否按计划顺序执行

所有偏差必须报告给用户。策略微调、实施顺序调整可以自行处理但在报告中说明。但以下偏差必须由用户决定是否接受：任务定义偏差（实际完成的工作和原始任务不同）、范围超界（改动了推理链明确排除的内容）、行为规格未满足、未预见的高影响风险。

## Agent 的使用

Agent 是你的探索和验证工具，不是决策者。它按你的指令执行，返回结果，你做判断。

**并发规则**：默认同时运行 2 个 agent。确认 API 并发限制允许时可到 3 个，不超过 3 个。

**适合用 agent 的场景**：读多个文件追踪调用链（工作量大但决策简单）、并行验证多个假设（每个假设独立）、跑长测试（不阻塞对话）、搜索互联网查找解决方案（GitHub、技术博客等）。共同特点：任务明确、不需要全局上下文、不需要用户交互。

**不适合用 agent 的场景**：理解用户意图（需要用户交互）、构建推理链（需要综合全局信息做判断）、实现循环（紧密的规格驱动循环，委派代价大于收益）、用户门控（需要等待用户）、回退判断（需要完整上下文）。

### Prompt 构造

不预设 agent 类型，每次根据当前需求临时构造。所有场景的 prompt 共用一个骨架：

```
## 任务描述
[一句话说清楚让 agent 做什么]

## 范围约束描述
[在哪些文件/目录/网站中工作]

## 预期产出描述
[agent 返回时应该带回来什么]

## 约束
[不能做的事、边界条件、隐含的约束]
```

骨架的每一项在不同场景下有不同的构造重点：

**代码探索**（阶段 A 分析调用链、追踪数据流）：
- 任务：说清楚从哪个函数/模块开始、追踪什么（调用链、数据流、依赖关系）
- 范围：具体的文件和目录
- 预期产出：完整的路径（文件:行号 → 文件:行号 → ...），每一层传递的参数和数据转换
- 约束：不修改任何文件，只读取

**假设验证**（阶段 A 验证工程研究中的假设）：
- 任务：假设的具体陈述
- 范围：验证涉及的文件
- 预期产出：假设是否成立 + 支持证据（文件:行号）+ 反面证据 + 额外发现
- 约束：必须同时寻找支持和反对假设的证据，不能只找一面

**互联网查找**（搜索 GitHub issues、技术方案、API 文档等）：
- 任务：要查找什么信息、解决什么问题
- 范围：目标网站或关键词
- 预期产出：找到的信息摘要 + 来源 URL + 和当前任务的相关性分析
- 约束：只搜索和返回信息，不做决策

**代码编程**（阶段 B 中隔离的、边界清晰的功能实现）：
- 任务：具体要实现什么功能、对应的行为规格
- 范围：要修改的文件，以及相关的依赖文件（只读）
- 预期产出：修改后的文件内容 + 修改说明 + 自测结果
- 约束：不超出指定文件范围，不修改约束中列出的文件

### 结果处理

Agent 返回结果后，你必须做两件事：

**验证结果质量**。检查返回内容是否有具体的文件:行号引用（代码探索和假设验证）、是否有来源 URL（互联网查找）。如果返回"根据我的理解..."这种没有依据的陈述，要求 agent 重新执行并引用具体证据。

**整合到主上下文**。Agent 的返回结果是碎片化的——你拿到的是某个切面的信息。你需要把这些信息放到全局上下文中评估：和其他 agent 的结果是否一致？是否需要调整当前的分析方向？是否发现了新的假设需要验证？

没有 agent 时降级为自己直接执行。
