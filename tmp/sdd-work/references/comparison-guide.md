# 推理链 vs 实际对比

最终校验阶段加载。阶段 B 完成，现在把冻结的推理链和实际实现逐环对比。

---

## 怎么对比

读取冻结的 task 文件（推理链基准版本）和实际改动（git diff 或直接读取代码），逐环回答对比问题：

**任务定义**：实际完成的工作是否是原始定义的任务？是 → 无偏差。否 → 偏差：实际完成的是什么，原始定义的是什么。部分 → 偏差：哪部分完成了，哪部分未涉及。

**现状定位**：实现是否针对了推理链认定的现状关键点？是 → 无偏差。否但关键点不同且都成立 → 偏差：说明为什么改了方向。否且修的是表面而非关键点 → 偏差：报告。

**解决策略**：实际策略是否和推理链一致？一致 → 无偏差。不一致但等价 → 偏差：策略不同但效果相同。不一致且不等价 → 偏差：评估实际策略是否同样解决了问题。

**范围边界**：实际修改的文件是否在范围内？全部在范围内 → 无偏差。有超范围改动 → 偏差：列出每个超范围改动，区分必要的超范围（修复过程中发现的必须修改）和意外的超范围（顺手改了不相关的东西——应该回退）。

**行为规格**：所有行为规格条目是否都被满足？全部满足 → 无偏差。有未满足的 → 偏差：列出未满足的条目，区分条目被替代（用更好的方式满足了行为意图）、被放弃（实施中发现不切实际）、被遗漏。

**风险**：推理链披露的风险是否出现？未出现 → 记录。已出现 → 报告出现情况和应对结果。出现了未预见的风险 → 偏差：描述新风险、影响、当前应对状态。

**实施顺序**：是否按顺序执行？按顺序 → 无偏差。跳过了步骤 → 偏差：说明为什么不需要或被合并了。调整了顺序 → 偏差：说明调整原因。

## 偏差处理

所有偏差在报告中说明。但处理方式不同：

**必须由用户决定的偏差**：任务定义偏差（实际完成的工作和原始任务不同）、范围超界（改动了推理链明确排除的内容）、行为规格未满足、未预见的高影响风险。

**可以自行处理但在报告中说明的偏差**：策略微调（等价替代）、实施顺序调整（不影响结果的重排）、低影响的意外改动（如顺带修正了明显的 typo）。

**需要回退到阶段 A 的偏差**：现状定位错误（实现针对的不是关键点）、大量超范围改动（推理链严重低估了影响范围）、核心行为规格无法满足（策略本身不可行）。
