# 验证

研究循环中按需加载。在根因分析阶段，你对工程现状形成了一组判断，现在需要验证这些判断是否站得住。验证的目标是证伪——尝试推翻判断，而不是确认它。

---

## 证伪优先

验证必须设计成能证伪假设。只做确认性验证是浪费——你总能找到支持假设的证据。

关键区别：假设说"X 函数总是返回 null"，证伪性验证去找"X 返回非 null 的所有情况"；确认性验证去找"X 返回 null 的一个例子"。前者能推翻假设，后者只能确认你知道的那个情况存在。

---

## Agent prompt 构造

验证多个假设时，为每个假设创建一个 agent 并行执行。按 SKILL.md 中"Agent 的使用"章节的"假设验证"场景构造 prompt——关键是约束中必须要求同时寻找支持和反对假设的证据。

并发规则和 prompt 骨架见 SKILL.md。假设超过并发上限时分批执行。

验证范围只有 1-2 个文件中的明确位置时，自己直接读取验证，不需要 agent。验证只需要一个命令（如搜索某个模式）时，直接执行。

---

## 验证结果处理

**假设被推翻**——回到架构理解步骤重新分析。推翻的证据指向什么新方向？之前的结构性理解在哪里断裂？

**假设部分成立**——收窄推理链的范围。明确假设成立的条件范围，不成立的范围是否有独立的原因需要追踪。考虑是否需要拆分为更精确的假设。

**假设经受住证伪**——作为推理链的证据支撑。但注意"没找到反例"和"假设成立"是不同的结论——没找到可能只是因为验证范围不够广。

**验证发现新问题**——评估是否纳入。直接影响当前任务目标 → 纳入；独立问题 → 记录但不纳入；不确定 → 问用户。

---

## 容易犯的错

- 验证范围太窄，只在假设指向的文件中验证——应该扩展到调用方和被调用方
- 验证粒度太粗，"整个模块看起来是对的"——缩小到具体的函数、行号、值
- 不同假设的验证互相影响——每个假设的验证应该独立设计，这样才能发现假设之间的矛盾
