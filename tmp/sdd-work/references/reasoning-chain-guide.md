# 推理链

阶段 A 的 A2 步骤加载。你刚完成研究循环，手里有完整的结构性理解演化轨迹和经过验证的分析判断，现在要把这些结果组织为一条完整的推理链。

---

## 七环之间的关系

推理链不是七个独立的部分，而是一条因果链——每一环依赖上一环：

```
任务定义（要做什么）→ 基于 → 现状定位（工程当前状态和关键点）→ 所以 → 解决策略（怎么做）→ 限于 → 范围边界（改什么不改什么）→ 达到 → 行为规格（系统应该表现出什么行为）→ 考虑到 → 风险披露（可能出什么问题）→ 按照 → 实施顺序（按什么原则拆分和排序）
```

现状定位错了，后面的策略、范围、行为规格全都建立在错误基础上。推理链的质量取决于最薄弱的环节。

---

## 逐环的质量标准

**任务定义**——一句话说清要做什么。修什么 bug、加什么功能、重构什么结构、补什么测试、建什么工程。自检：一个不了解项目的人读完能否知道这次任务要达成什么？

"预测系统有问题"不合格——没有说清要做什么。"修复 runner.py 主循环中 observe 在 predict 之后调用的时序 bug"合格。"为价格区间预测系统添加离线进化器"合格。

**现状定位**——工程当前的状态是什么、关键点在哪里。从研究循环的结构性理解中提取，包含模块边界分析和数据流分析的综合判断——不只是代码层面的观察，还要说清楚结构上意味着什么。每步有代码证据。自检：你的现状判断是否都有"证据：文件:行号"标注？是否包含了结构性的诊断而不只是事实的罗列？

修 bug 时是因果链——"测试 test_observe_timing 失败 → predict 使用的 sigma 不是最新的 → predict 在 observe 之前执行（runner.py:384）→ 主循环时序与设计方案 Section 5.4 矛盾"合格。"代码写得不对"不合格——没有因果链，没有证据。

加功能时是模块边界和数据流的综合判断——"当前信号系统只有 judgment.py 的 Rule 0-4，没有价格区间预测能力。信号通过 BarSignals 传递，account.py 的 process_bar_v2 接受 prediction_result 参数但未使用。新功能的职责属于 judgment 模块的边界内，但所需的价格数据当前未流经该模块，需要新增数据通路"合格。只列出"没有预测能力、参数未使用"而不说明结构含义，不合格。

重构时是结构缺陷描述——"account.py 同时负责仓位管理和挂单执行，_process_tranches 方法 120 行，与主流程耦合。模块边界模糊导致数据流穿越了不该穿越的职责边界"合格。

写测试时是覆盖情况——"judgment.py 的 evaluate 函数有 6 个分支规则，当前测试覆盖了 min_hold_bars 和 initial_entry，缺少 profit_add_only 和 max_hold_bars 的边界测试"合格。

**解决策略**——说清用什么方法达成任务目标。策略必须直接针对现状定位的关键点。自检：如果现状的关键点是 X，这个策略是否直接解决了 X？还是绕过了 X？同时说明选这个策略的原因，以及排除了哪些替代方案。

**范围边界**——列出要改的文件清单和明确不改的内容。自检：能否清楚说出为什么某个文件在范围内或范围外？

"改一下相关文件"不合格。"改动：runner.py（主循环时序调整）；不改：predictor.py（predictor 本身无 bug）、account.py（tranche 逻辑不受影响）"合格。

**行为规格**——系统应该表现出什么行为的完整描述。不是"怎么判断完成了"，而是"系统的行为契约是什么"。包含四个部分：输入输出契约（给什么输入、返回什么输出、类型和约束）、不变量（无论什么条件都必须成立的关系）、边界和异常（极端输入怎么处理、异常怎么响应）、前置后置条件（执行前必须满足什么、执行后保证什么）。每条规格标注验证方式：`[测试验证]`（核心逻辑、数据转换）、`[代码审查]`（简单赋值、配置）、`[集成测试]`（多模块协作）。自检：拿到这份规格的人，不看代码能否判断实现是否正确？是否有模糊到无法验证的条目？

"系统正常工作"不合格——没有行为契约，无法验证。"PriceNormalizer 接收 float 数组，返回等长 float 数组，每个值在 [0,1] 区间；全零输入返回全零数组；NaN 输入抛出 ValueError。[测试验证]"合格——输入输出契约明确，边界和异常有处理，验证方式已标注。

**判断标准**：如果换一种实现方式，外部行为不变，那这个描述就不是行为规格，是实现细节。"系统支持深色/浅色主题切换"是行为——换用 CSS 变量还是 CSS-in-JS，行为不变。"使用 React Context 管理主题状态"是实现细节——换用 Redux 行为不变，所以不属于行为规格。

**风险披露**——区分"确定有风险"和"不确定是否有风险"。对每个风险说出影响和缓解方式，对每个不确定性说出如何消除。

**实施顺序**——按依赖关系和风险排定的任务序列。不是随意列举，而是遵循拆分原则。自检：每个任务是否独立可验证？任务之间是否有明确的依赖说明？高风险任务是否排在前面（当依赖允许时）？

**拆分原则**：从行为规格中提取实现单元 → 识别单元之间的依赖关系（B 依赖 A 的输出则 A 必须在前） → 按依赖约束排序，依赖不约束时高风险优先 → 检查粒度（每个任务在一个实现循环内可完成、完成后可独立对照行为规格验证）。

差的实施顺序："1. 实现预测模块 2. 实现数据模块 3. 实现主循环"——粒度太粗、无依赖分析、不可独立验证。

好的实施顺序："1. PriceNormalizer 工具类（无依赖，可立即验证归一化行为）→ 2. DataPipeline 集成归一化（依赖1，验证数据流转）→ 3. PredictionModel 使用归一化数据（依赖2，验证预测输出）→ 4. Runner 接入 PredictionModel（依赖3，端到端验证）"——依赖递进、每步独立可验证、粒度适中。

## 整体自检

构建完成后做五项检查：环环相扣（每环自然推出下一环，无断裂）、证据完整（每环都有证据）、可操作性（拿到这条推理链的人能否独立实施）、无空洞（没有只有标题没有内容的环节）、无矛盾（不同环节之间没有互相矛盾的陈述）。

## 典型缺陷

- **任务定义和现状定位混淆**——"任务是代码写错了"把"要做什么"和"当前状态是什么"混在一起。任务定义说目标，现状定位说现状。
- **策略和范围混淆**——"策略是修改 runner.py"把方法（怎么做）和边界（改哪些文件）混在一起
- **行为规格不完整**——"系统正常工作"没有行为契约，无法验证实现是否满足
- **行为规格和测试混淆**——行为规格写成了测试用例（"assert result == 5"），规格应该描述行为而不是断言值
- **风险变成解决方案**——"风险是改错了所以要多测试"把风险描述成了行动建议
- **实施顺序拆分不当**——粒度太粗（"实现整个模块"一个循环做不完）、没有依赖分析（顺序随机）、不可独立验证（要等其他任务完成才能验证）
