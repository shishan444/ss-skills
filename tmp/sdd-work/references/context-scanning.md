# 上下文扫描

启动阶段加载。你刚拿到用户的任务描述，需要在开始分析之前了解项目是什么、怎么构建的。

---

## 第一步：找到工程目录

读取项目根目录的 CLAUDE.md，从中找到两个关键路径：**工程目录**（如 MyQuant/）和**工程设计文档目录**（如 MyQuant/docs/）。CLAUDE.md 是工程目录的唯一来源，不需要其他参考路径。

---

## 第二步：判断工程状态

到工程目录下检查 docs/ 目录是否存在。这里会遇到两种情况：

**老工程**：工程已存在一段时间，有代码、有历史。docs/ 目录应该存在，里面有工程设计文档（架构描述、模块说明、流程描述等）。如果 docs/ 不存在——直接停止服务，告知用户："当前工程缺少 docs/ 工程文档目录，无法理解工程内容。请先添加工程文档描述后再使用编程工作 skill。"不要在没有文档的情况下继续执行，因为没有文档意味着你对工程的理解是盲目的。

**新工程**：工程刚创建，代码和文档都还不完整。docs/ 可能不存在。此时不停止服务，但在 task 文件中记录："新工程，无 docs/ 文档。分析将主要基于代码和用户描述。"后续分析和实现中需要主动补充文档。

怎么区分新老工程：看工程目录下是否已有源代码文件（如 src/、core/、app/ 等目录，或 .py、.ts 等源文件）。有 → 老工程。没有或只有骨架文件 → 新工程。

---

## 第三步：阅读工程文档

docs/ 目录存在时，阅读工程设计文档。先看目录结构，了解有哪些文档，再按与当前任务的相关性选择阅读顺序。

文档的典型内容和作用：
- 架构描述文档 → 了解工程的整体结构、模块划分、技术栈
- 模块说明文档 → 了解具体模块的职责和接口
- 流程描述文档 → 了解关键业务流程的数据流
- 设计方案文档 → 了解近期或正在进行的设计变更

阅读深度控制：文件标题和头部与任务明确相关 → 读取全文。可能相关但不确定 → 读取头部判断后再决定。明显无关 → 跳过。如果文档量大，用 agent 并行探索，每个 agent 读一个文档返回摘要。
