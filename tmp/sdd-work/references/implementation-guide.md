# 实施

阶段 B 加载。

## 实施原则

**按实施方案顺序执行。** 顺序存在是有原因的——前面的步骤可能是后面步骤的前提，顺序经过分析确认是最安全的路径。发现顺序有问题时停下来报告，不要自行调整。

**渐进实施验证。** 每完成一个步骤就运行相关测试。不要等所有步骤做完才测试。问题在刚引入时最容易定位，超范围改动能被立即发现。

**不超范围。** 推理链第 4 环定义了范围边界。改动在范围内 → 正常实施。发现需要改动范围外的文件 → 停止，评估是否需要扩展范围，报告用户。发现范围外的 bug → 记录到 task 文件中但不修复，修复超范围 bug 可能引入新的超范围改动。


## 每个步骤的执行

对实施顺序中的每个条目：回顾推理链中该步骤的目标 → 读取目标文件确认当前状态 → 执行规格驱动实现循环 → 记录完成的行为规格 。


## 回退

**测试修复失败**（尝试超过 3 次）。检查测试本身有没有 bug → 检查实现是否正确 → 两者都正确但测试仍失败 → 推理链可能有缺陷。

**发现推理链缺陷。** 停止实施 → 回到阶段 A 对应的断裂环节 → 重新分析和验证 → 重新走门控让用户确认 → 用户确认后从断裂点继续。回退不是失败，在错误前提上继续实施才是浪费。

**发现需要超范围改动。** 评估必要性（必须改还是可以变通）→ 必须改则报告用户等待确认 → 用户确认则扩展范围记录到 plan → 用户不确认则寻找范围内替代方案。


## 进度记录

在 task 文件中记录每个条目的完成状态：条目描述、实现行为规格满足情况、改动文件和行范围、测试验证结果、范围检查结果。这样在任何时候暂停和恢复都不会丢失上下文。
