# 门控呈现

门控阶段加载。你刚完成推理链的构建，现在要把它呈现给用户，等待确认。

---

## 怎么呈现

结论先行。用户关心结论是否正确，过程是支撑结论的工具。按三个层次呈现：

**第一层：推理链七环摘要。** 每环一句话。对每环标注置信度——确认（有代码证据或验证结果支撑）、高置信（有强间接证据）、待确认（基于推理但缺少直接证据）。用户需要知道哪些环节是铁板钉钉的，哪些还有不确定性。

**第二层：关键证据。** 支撑每环的证据链——文件:行号、验证结果、设计文档引用。不需要列出所有证据，只列支撑结论成立的必要证据。

**第三层：细节。** 完整的分析过程，按需展开。用户追问时才展开这一层。

直接按推理链七环的结构呈现，不要重新组织为其他格式。推理链是分析的核心产出，用户需要看到完整的推理链才能判断是否认可。

---

## 用户反馈的处理

**用户要求修改**——根据修改涉及的具体环节，回到阶段 A 对应步骤：修改任务定义或现状定位 → A1；修改验证结论 → A2；修改推理链细节 → A3。修改后重新走门控。不能只改一处就跳过其他环节的检查——修改可能引起连锁影响。

**用户要求缩小范围**——确认缩小后的范围仍然逻辑闭合（任务定义 → 现状定位 → 策略 → 行为规格仍然完整）。将被排除的部分标记为后续任务，记录到 task 文件。重新呈现缩小后的推理链。

**用户要求扩大范围**——评估扩展部分是否和当前任务在同一因果链上。是 → 扩展推理链，回到 A1/A2 补充；不是（独立问题）→ 记录为独立任务，不在本次推理链中混合。

**用户看不懂**——原因通常是术语过多或因果链跳跃。用自然语言重述，聚焦"所以呢"——每环对用户意味着什么。

**用户和推理链结论不一致**——不要急于修改。先理解异议来源：用户有我们没有的信息 → 补充后重新研究；不同意策略选择 → 讨论替代方案；不同意任务定义 → 回到意图理解重新对齐。

**用户说"看起来没问题，继续吧"**——这是确认。但如果用户明显没有仔细看就确认，简要复述最关键的风险和范围边界，确保用户知情。

---

## 冻结确认

用户确认后，在 task 文件中标记推理链为冻结状态。冻结意味着：阶段 B 以此为基准实施，偏离必须报告。记录用户的确认语（简要记录用户同意了什么）。
