# 规格驱动实现

阶段 B 的 B3 步骤加载。行为规格已细化，测试用例已设计，现在按实施顺序逐条实现。


## 实现循环

对实施顺序中的每个条目，四个步骤：

**对规格实现。** 读该条目的实现级行为规格，写代码让系统表现出规格描述的行为。实现的判断标准：对照规格的每一条——输入输出契约满足了吗？不变量成立吗？边界处理了吗？不是凭感觉写代码，是对着规格逐条实现。

**自查合规。** 逐条对照行为规格检查实现。不是"我觉得写得对"，而是拿着规格逐条验证：这条契约满足了吗？这个不变量成立吗？这个边界处理了吗？可以用 agent 做这个自查——给它规格和代码，让它逐条报告满足/不满足 + 原因（文件:行号）。不满足的立即修正，修正后重新自查。

**按需测试验证。** 为该条目中标注了 `[测试验证]` 的行为规格写测试。测试用例已在 B2 设计好——测什么行为、输入什么、期望什么输出、覆盖什么路径。这里把设计翻译为测试代码。写完后运行，确认通过。测试未通过 → 先检查实现是否正确，再检查测试本身是否有误。修正后重新运行。

**范围检查。** 确认改动在推理链的范围边界内。超范围 → 停下来报告，不继续推进。

循环直到该条目的所有行为规格满足、所有标注 `[测试验证]` 的测试通过。


## 测试验证的原则

不是所有行为规格都需要测试覆盖。验证方式在阶段 A 定义行为规格时已标注，B2 已设计具体用例。原则回顾：

- **核心业务逻辑和计算** → 必须测试。涉及正确性的，靠代码审查不够可靠。
- **数据转换和流转** → 应该测试。有明确数学约束或数据变换关系的。
- **简单赋值和配置** → 不需要测试。代码审查确认即可。
- **多模块协作** → 集成测试。单个模块的行为正确不代表协作正确。

能用单元测试验证的不用集成测试。测试越接近被测代码，反馈越快、定位越准。

测试写不出来 → 通常是行为规格不够具体。回到 B2 细化规格，而不是写一个模糊的测试凑数。


## 深度适配

不同复杂度的任务，实现循环的深度不同，步骤不减。

"改一个变量名"：规格是"修改后行为不变" → 改那一行 → 自查行为是否一致 → 不需要测试 → 范围检查。五分钟走完。

"修复一个时序 bug"：规格描述正确时序（observe 在 predict 之前）→ 调整代码顺序 → 自查时序正确 → 写测试验证正确时序和回归 → 范围检查。

"删除一个子系统"：规格描述删除后的行为（哪些功能移除、哪些接口变更、依赖模块如何适配）→ 逐步删除，每删一处自查 → 为受影响的模块写测试验证适配正确 → 逐文件确认删除范围。


## 范围检查的关键性

每个实现循环结束后必须做范围检查。推理链定义了明确的范围边界，实施过程中最常见的问题是"顺手改了不相关的东西"。范围检查就是防止这种蔓延——改动了范围外的文件，即使改得对，也要停下来报告，不能悄悄溜过去。
