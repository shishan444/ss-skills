### 工程研究

阶段 A 的研究步骤加载。你刚完成上下文扫描，现在要深入工程代码，建立对系统的结构性理解，找到任务的真正关键点。

研究不是信息采集。不是把相关代码读一遍、把发现列一遍就算完。研究的目标是建起一个结构模型——系统的模块边界在哪、数据怎么在边界之间流动、任务涉及的变化在这个结构里处于什么位置。有了这个模型，后续的根因判断、策略选择、范围界定才有承重的基础。

研究以循环方式推进：

```
循环 {
  架构理解（首轮：2 个 subagent 并行；后续轮：定向 subagent）
  → 根因分析 + 工程验证
  → 思考引导（三问决策）
      → 存在关键不确定性 → 封装 subagent → 下一轮
      → 不存在 → 研究完成
}
```

每轮循环包含三步。思考引导决定是否需要下一轮——有未消解的关键不确定性就继续，没有就结束，进入推理链构建。

---

step 1： 架构理解

每轮循环的起点。目的不是读代码，是看清系统的骨架——模块怎么划分、职责怎么分配、数据怎么流转。

1. 首轮：两路并行

派出 2 个 subagent 同时工作，一个从架构层看结构，一个从链路层看数据流动。

**架构层 subagent** 研究模块的职责划分。对任务涉及的每个模块回答四个问题：它拥有什么（管什么数据、担什么职责）、对外暴露什么（公开接口、输入输出契约）、依赖什么（需要从谁那里拿什么）、不知道什么（刻意不感知的部分）。产出是一张模块职责图，标注证据（文件:行号）。

职责边界不等于文件边界。一个叫 `utils.py` 的文件可能藏着核心业务逻辑，一个叫 `service.py` 的文件可能同时管着三件不相干的事。从代码实际做的事情推断职责，不从文件名推断。

**数据流 subagent** 追踪与任务相关的数据路径。数据从哪里进入系统、经过哪些模块、在每个边界上发生什么转换、最终到达哪里。关注点是数据跨越模块边界的方式——通过接口传参、共享状态、还是中间存储。产出是一张数据流图，标注每一跳的转换逻辑和证据（文件:行号）。

两个 subagent 的 prompt 按 SKILL.md "Agent 的使用"章节的骨架构造：

- **任务描述**：分别是"研究模块职责边界"和"追踪数据流路径"，说清楚从哪个模块开始、和什么任务相关
- **范围**：从任务直接涉及的模块出发，subagent 在追踪过程中自行判断需要看多远
- **上下文描述**：任务背景上下文重点内容描述
- **预期产出**：边界 subagent 产出模块职责图（每个模块的四个回答 + 文件:行号）；数据流 subagent 产出数据流图（完整路径 + 每一跳的转换 + 文件:行号）
- **约束**：任务隐含要求

2. 汇总分析

汇总 两个 subagent 研究返回详细后，把边界图和数据流图对照起来看。边界划分和数据流走向一致的地方，结构是清晰的；数据流穿越了不该穿越的边界，或者一个模块需要知道另一个模块的内部细节才能完成数据传递——这些不一致的地方往往是问题的真正所在。

汇总产出是一段结构性理解：用连贯的话说清楚系统这片代码的骨架长什么样、数据怎么在骨架上流动、哪些地方骨架和流动匹配、哪些地方有错位。不是罗列发现，是综合判断。

3. 汇总 后续轮：定向研究

从第二轮起，不再做全面扫描。上一轮思考引导产出了具体的不确定性条目和对应的研究任务——这些就是本轮 subagent 的任务。规划为任务序列，然后使用subagent 进行任务，可能是"确认模块 A 的接口是否被其他模块调用"，也可能是"追踪某条数据路径在异常情况下的走向"。subagent 分工由研究任务决定。

定向研究的结果整合进已有的结构性理解——补充、修正、或推翻之前的判断。

---

step 2： 根因分析与工程验证

架构理解给了你一张结构地图。现在用这张地图来分析任务的关键点。

不管任务是修 bug、加功能还是重构，分析方法是一样的：拿着任务在结构地图上定位——它落在哪个模块的边界内？涉及哪些数据流路径？会对现有结构施加什么压力？

具体的分析切入因任务性质而不同：

修 bug 时，从症状出发沿数据流路径追踪，结合模块边界判断——问题出在模块内部的逻辑错误，还是模块之间的契约不清？因果链的每一层都要有代码证据。"系统设计有问题"不是分析结果，"order/service.py:128 在调用 inventory 模块扣减库存时没有检查返回的失败状态，因为 service 层假设 repository 永远成功"才是。

加功能时，在结构地图上找落地位置——新功能的职责属于哪个模块的边界内？它需要的数据在当前流路径中能拿到吗？放进去之后模块的职责还是内聚的吗？

重构时，结构地图本身就是分析对象——哪些模块的边界已经模糊、哪些数据流在不必要地穿越多个模块、哪些职责被拆散在多个地方。

**每个分析判断都必须经过工程验证。** 验证的核心是证伪——对每个关键判断，主动问"什么事实如果存在，就说明我的判断是错的"，然后去代码里找。只找支持证据不找反面证据，不算验证。

验证范围只有一两个文件的明确位置时，自己直接读取；需要跨多个文件追踪时，用 agent。验证的具体方法——怎么设计证伪、怎么构造 agent prompt、怎么处理验证结果——加载 `verification-guide.md`。


### 思考引导

每轮循环做完架构理解和根因分析后，停下来回答三个问题。这三个问题的答案决定循环继续还是结束。

**第一问：我现在对系统的结构性理解是什么？**

把这一轮的发现和之前积累的理解合在一起，用连贯的话说出来——系统的模块边界在哪、数据怎么流、任务涉及的部分在结构中处于什么位置、结构有什么问题。不是重复 subagent 返回的原始发现，是消化后的综合判断。

如果说不出连贯的结构性理解——只能列出零散的事实（"模块 A 调了模块 B"、"这个函数返回了 null"）——说明架构理解阶段的产出还不够，需要补充研究再回答这一问。

**第二问：这个理解如何改变了我对任务的看法？**

研究之前你对任务有一个初始理解。现在研究过了，那个理解变了没有？

用户报了一个 bug，你研究后发现根因不是代码写错了，而是模块边界划分不合理导致数据在传递过程中丢失了关键字段——这时候你对任务的看法就从"修一行代码"变成了"这是一个边界问题"。这个变化直接影响后续推理链的策略和范围。

如果研究完了对任务的看法没有任何变化，两种可能：任务本来就简单直接，或者研究没有触碰到真实的结构层。后者需要警惕。

**第三问：还有什么不确定的东西，如果知道了答案会改变我的处理方式？**

这一问是过滤器。不确定的东西可能很多，但只有一种值得继续研究——知道了答案之后，你的处理方式会不一样的。

"我不确定模块 C 内部怎么实现的，但不管它怎么实现，我修改模块 A 的方式都不变"——不需要继续，记录为已知的不确定性，不阻塞。

"我不确定模块 A 的这个接口是不是还被其他三个地方调用，如果是的话我的修改范围会完全不同"——需要继续，这个不确定性直接影响决策。

**决策**

第三问的答案决定走向：

**答案为空**——没有会改变处理方式的不确定性。研究完成，进入 A3 构建推理链。

**答案不为空**——把每个不确定性条目转化为下一轮 subagent 的研究任务。转化方式：从"我不确定 X"推导出"让 agent 去查 Y——查到 Z 说明 X 成立，查到 W 说明 X 不成立"。然后进入下一轮循环。

---

## 每轮产出

每完成一轮循环，将该轮的思考引导写入 task 文件。格式：

```
## 研究第 N 轮

### 任务结构性理解
[第一问的回答——对任务的系统结构的综合判断]

### 任务认知变化
[第二问的回答——发现了什么信息，影响了什么内容，改变了对任务的什看法，本轮看法是什么，比如：xxx的情况发现，更新了认识，不是bug是设计结构问题，设计中没有对于这块的定义]

### 待消解的不确定性
[第三问的回答——列出条目和对应的研究任务，为空则写"无"]

### 决策
[继续研究（说明下一轮要消解什么）/ 研究完成]
```

每一轮的记录都追加保留，不覆盖。这些记录展示理解的演化轨迹——从最初的粗略认知到最终的结构性判断是怎么一步步走过来的。

---

## 容易踩的坑

**架构理解停在文件清单**——列出"项目有 A、B、C 三个模块"不是架构理解。要回答模块之间的关系：谁依赖谁、数据怎么流、边界在哪里清晰、在哪里模糊。

**把观察当诊断**——"数据从 A 流到 B"是观察。"数据从 A 到 B 的过程中丢失了状态信息，因为 B 的接口设计时没有考虑这个字段"是诊断。观察描述"是什么"，诊断回答"意味着什么"。

**过早收敛**——找到第一个看起来合理的根因就停。结构地图上可能有多个可疑点，在思考引导之前不要过早排除。

**确认偏误**——只找支持自己判断的证据。工程验证的核心是证伪：主动寻找能推翻判断的事实。

**思考引导走过场**——三问的回答变成"理解充分、无变化、无不确定性"的套话。如果第一问说不出具体的结构关系、第二问说不出认知的变化、第三问一个不确定性都列不出来，大概率是研究还不够深，不是真的什么都确定了。

**忽视设计文档**——工程文档（docs/）和代码可能不一致。先读设计文档理解意图，再用代码验证。两者矛盾时以设计文档的意图为准，代码实现偏差作为发现记录。

---

## 研究完成后

思考引导确认没有需要消解的关键不确定性，研究阶段结束。此时 task 文件中包含：

- 每轮循环的结构性理解（理解的演化轨迹）
- 经过工程验证的分析判断（附代码证据）
- 思考引导的完整记录（每轮的三问回答和决策）

这些是 A3 构建推理链的输入。推理链的"现状定位"环节从结构性理解中提取，不需要重新分析。
