# kb-tagger 设计追溯

## 失败模式

无约束的 LLM 在"知识库元数据与关联管理"任务上的默认行为：

### 标签相关（tag-gen / tag-normalize）

| # | 失败模式 | 根因 |
|---|---------|------|
| 1 | tag-normalize 不扫描全库直接操作 | 跳过了 Grep 扫描步骤 |
| 2 | tags / leads / aliases 混用——把"巴菲特"放进 tags，把"认知偏差"放进 leads | 不理解三字段的语义边界 |
| 3 | 标签偏向内容描述而非检索锚点 | 将"打标签"理解为"描述内容" |
| 4 | 为每篇笔记创造独特标签，导致标签膨胀 | 优先"为当前文件生成最准确的标签" |
| 5 | 覆盖已有 frontmatter 字段 | 不了解 frontmatter 结构的脆弱性 |
| 6 | tag-gen/normalize 误触 connections 字段 | 不理解字段分组和维护者边界 |
| 7 | 破坏人/LLM 维护分组注释结构 | 不了解分组注释的语义 |
| 8 | 别名越界——把"前景理论"当作"损失厌恶"的别名 | 分不清同义名和相关概念 |
| 9 | leads 偏向内容关键词而非检索线索 | 不理解 leads 的认知检索本质 |

### 链接相关（link）

| # | 失败模式 | 根因 |
|---|---------|------|
| 10 | link 只写中心笔记不写目标笔记 | 不理解双向链接的语义要求 |
| 11 | link 创建大量浅层关联——"相关"但无实质关系 | 不理解四种关系类型的判断标准 |
| 12 | link 覆盖已有链接而非追加 | 不理解增量补链原则 |
| 13 | link 写入 body 时破坏笔记结构 | 不了解 body 的六层分节结构 |

### 模式相关

| # | 失败模式 | 根因 |
|---|---------|------|
| 14 | 三种模式混淆——用 tag-gen 逻辑执行 link | 不理解模式的边界和触发条件 |

## 指导点设计

### FM1：tag-normalize 跳过扫描
**C：跳步？** 是。
→ 思维步骤：mode-tag-normalize.md Step 1 强制 Grep 扫描——不可跳过。

### FM2：tags / leads / aliases 混用
**A：不知好？** 是。
→ 评价视角：frontmatter-schema.md 三字段区分表 + 每个字段的判断测试。

### FM3：内容描述式标签
**A：不知好？** 是。
→ 评价视角：mode-tag-gen.md 标签质量三问 + 粒度参照表。

### FM4：标签膨胀
**B：权衡错？** 是。
→ 取舍规则：mode-tag-gen.md 优先使用简洁通用术语。

### FM5：覆盖已有字段
**D：隐患？** 是。
→ 代价描述：SKILL.md 反面代价——链接失效、数据丢失、不可修复。

### FM6：tag-gen/normalize 误触 connections
**D：隐患？** 是。
→ 代价描述：SKILL.md 字段维护者表 + 各 mode 文件明确标注"不触碰"。

### FM7：破坏分组结构
**D：隐患？** 是。
→ 代价描述：各 mode 文件写入规则——保留分组注释。

### FM8：别名越界
**A：不知好？** 是。
→ 评价视角：mode-tag-gen.md aliases 判断标准——"换说法仍指同一概念"。

### FM9：leads 偏向内容关键词
**A：不知好？** 是。
→ 评价视角：mode-tag-gen.md leads 区分——检索线索 vs 内容描述，正反示例。

### FM10：link 单向写入
**C：跳步？** 是。
→ 思维步骤：mode-link.md 双向写入检查清单。

### FM11：浅层关联
**A：不知好？** 是。
→ 评价视角：mode-link.md 四种关系判断标准 + "避免浅层关联"规则。

### FM12：覆盖已有链接
**C：跳步？** 是。
→ 思维步骤：mode-link.md 安全规则——只追加不删除。

### FM13：写入破坏 body
**D：隐患？** 是。
→ 代价描述：mode-link.md 安全规则——只在脉络段追加。

### FM14：模式混淆
**C：跳步？** 是。
→ 思维步骤：SKILL.md 模式识别表——根据用户意图加载对应 mode 文件。

## 强度判定

| # | 分歧点 | 翻转结果 | 强度 |
|---|--------|---------|------|
| 1 | tag-normalize 不扫描直接操作 | 产出根本不同 | 强 |
| 2 | tags/leads/aliases 混用 | 产出根本不同 | 强 |
| 3 | 内容描述式标签 | 产出根本不同 | 强 |
| 4 | 自由创建独特标签 | 产出方向不同 | 中 |
| 5 | 覆盖写入 frontmatter | 产出根本不同（数据丢失） | 强 |
| 6 | 误触 connections | 产出根本不同（数据丢失） | 强 |
| 7 | 破坏分组结构 | 产出不同 | 强 |
| 8 | 别名越界 | 产出根本不同 | 强 |
| 9 | leads 偏向内容 | 产出方向不同 | 中 |
| 10 | link 单向写入 | 产出根本不同（图谱不完整） | 强 |
| 11 | 浅层关联 | 产出方向不同 | 中 |
| 12 | 覆盖已有链接 | 产出根本不同（信息丢失） | 强 |
| 13 | 破坏 body 结构 | 产出根本不同 | 强 |
| 14 | 模式混淆 | 产出根本不同 | 强 |

12 强 + 3 中。元数据管理任务精度要求高，多数分歧点翻转后产出根本不同。

## 设计决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 三个模式拆分为独立文件 | 而非全部写在 SKILL.md | 按需加载，减少上下文占用 |
| frontmatter-schema 独立文件 | 而非内联在各 mode 文件 | 三个模式共享，避免重复 |
| kb_scan.py 仅服务 link 模式 | 而非三个模式共用 | tag-gen 逐文件处理无需扫描，tag-normalize 用 Grep |
| kb_scan.py 输出文本 | 而非 JSON | 便于 LLM 作为参考资料直接使用 |
| tag-gen 逐文件处理 | 而非批量 | 文件间互不依赖，简单可控 |
| 不使用 kb_write.py | LLM 用 Edit 直接写入 | 减少工具复杂度 |
| 无词频统计 | 基于内容理解而非数据分析 | 词频对标签质量无实质帮助 |
| disable-model-invocation: true | 手动触发 | 元数据更新是用户主动发起的维护操作 |

## 与 kb-note 的职责边界

| 职责 | kb-note | kb-tagger |
|------|---------|-----------|
| 创建新笔记 + 完整 frontmatter | 负责 | 不涉及 |
| tags/leads/aliases 生成 | 创建时生成 | 后续更新/补充（tag-gen） |
| 标签规范化 | 不涉及 | 负责（tag-normalize） |
| connections 管理 | 创建时建立 | 增量补充（link） |
