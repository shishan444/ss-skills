# kb-note 设计追溯

## 失败模式

无约束的 LLM 在"创建结构化知识笔记"任务上的默认行为：

1. 写出扁平的内容堆砌，没有层次区分（现象、证据、边界混在一起）
2. 跳过疆界层——大多数人和 LLM 都倾向于只写"是什么"，不写"在哪里不适用"
3. 关联要么没有（孤岛笔记），要么泛滥（什么都关联什么都等于没关联）
4. tags 和 leads 混淆——把"巴菲特"放进 tags 而非 leads
5. anchor 写成内容摘要而非核心判断——"介绍了损失厌恶的概念" vs "损失的心理权重约为等量收益的2-2.5倍"
6. 跳过扫描已有笔记图谱，导致新笔记无法融入知识图谱
7. 不与用户确认 anchor，导致核心判断与用户的真实理解偏差
8. 源头层硬写——没有明确触发问题时写出"在一本书里读到的"，这是来源不是源头

### 根因

LLM 将"写知识笔记"理解为"整理和呈现信息"，而非"锻造一个可检索、可关联、可演化的知识单元"。缺乏对笔记在知识图谱中扮演的角色的理解。

## 指导点设计

### 失败模式 1：扁平内容堆砌

**追问 A：不知道什么是好的？** 是。
→ 产出**评价视角**：六层结构——锚定/源头/根基/疆界/脉络/活用，每层有明确职责。

**追问 C：跳过步骤？** 是。LLM 倾向于把所有信息塞进一个"内容"区域。
→ 产出**思维步骤**：逐层锻造，每层有独立标题和约束。

### 失败模式 2：跳过疆界层

**追问 A：不知道什么是好的？** 是。
→ 产出**评价视角**：疆界是"这条知识在哪里不好用的地图"。至少 2 个边界条件。

**追问 D：未意识到隐患？** 是。
→ 产出**代价描述**：标注"不确定"比假装确定更有价值。

### 失败模式 3：关联缺失或泛滥

**追问 B：权衡点选错？** 是。
→ 产出**取舍规则**：宁可没有关联，不要硬凑。每条关联必须带完整上下文和明确类型判断。

### 失败模式 4：tags 和 leads 混淆

**追问 A：不知道什么是好的？** 是。
→ 产出**评价视角**：tags 回答"讲的是什么"，leads 回答"什么词会带你到这条笔记"。两者不应有重叠。

### 失败模式 5：anchor 写成内容摘要

**追问 A：不知道什么是好的？** 是。
→ 产出**评价视角**：anchor 是单句唤醒测试——删掉所有其他内容只留这一句，理解还能被激活吗？

### 失败模式 6：跳过扫描已有图谱

**追问 C：跳过步骤？** 是。
→ 产出**思维步骤**：步骤 2 强制扫描——不可跳过。

### 失败模式 7：不确认 anchor

**追问 C：跳过步骤？** 是。
→ 产出**思维步骤**：步骤 3 强制与用户确认核心判断。

### 失败模式 8：源头层硬写

**追问 B：权衡点选错？** 是。LLM 倾向于给每个层都填内容。
→ 产出**取舍规则**：源头为可选层。无明确触发问题时跳过，不要硬写。"在一本书里读到的"是来源信息，放根基层。

## 设计决策记录

| 决策 | 选择 | 放弃的方案 | 原因 |
|------|------|----------|------|
| 六层结构 | 锚定/源头/根基/疆界/脉络/活用 | 五层（无源头） | 源头捕捉知识获取的触发情境，与根基（证据）和活用（应用）正交 |
| 源头层可选 | 无触发问题时跳过 | 必填 | 硬写会产生低质量内容，违反"宁可空着不要硬凑"原则 |
| anchor 单句 | 单行字符串 | 列表多角度描述 | 单句约束笔记粒度；多角度由 tags+leads 补充 |
| tags 和 leads 分开 | 两个独立字段 | 合并为单一 tags | tags 是内容分类（Obsidian 原生搜索），leads 是检索线索（AI 脚本匹配），语义和用途不同 |
| 四种关联类型 | support/resonate/tension/instance | 三种或五种 | instance（抽象↔具体）出现频率高且判断标准清晰；composes（部分↔整体）暂缓，等实际需求再补 |
| composes 暂缓 | 观望 | 预先加入 | 边界与 support 有交叉，维护成本尚未验证 |
| connections 在 Frontmatter | 扁平列表+内联理由 | 仅在正文脉络层 | LLM 脚本扫描时需做图遍历，理由让 LLM 精准筛选是否加载关联笔记 |
| connections 格式 | `"[[笔记名]] 理由"` | 嵌套字典 | Obsidian 不支持嵌套 YAML |
| aliases 由 LLM 维护 | LLM 自动生成 | 手动填写 | 零人工成本，Obsidian 原生支持 |
| 空关联处理 | 不写该字段 | `[]` 空列表 | Obsidian 将空列表转为 `null` |
| 沉积层四要素 | 从→到→触发→关联变化 | 简单修改记录 | 四要素保证认知迁移事件的完整可追溯 |
| 维护者分组 | 人维护 vs LLM 维护 | 不区分 | 降低用户认知负担，明确哪些字段需要操心 |
| anchor 必须用户确认 | 写前确认 | LLM 直接生成 | 核心判断偏差会传染到所有其他字段 |
| 扫描已有图谱 | 写前必扫 | 直接写新笔记 | 跳过扫描是图谱断裂的根源 |

## Frontmatter 检索链路设计

人维护字段 + LLM 维护字段共同构成 LLM 两级检索的完整数据源：

```
第一级：Frontmatter 扫描
  ① tags 粗筛 → 几百条缩至几十条
  ② leads 关联筛选 → 几十条缩至十几条
  ③ anchor 语义精准匹配 → 十几条缩至几条
  ④ connections 图遍历 → 从几条扩展到相关图谱
    - support：因果追溯链
    - resonate：跨域迁移链
    - tension：矛盾检测链
    - instance：抽象→具体链

第二级：正文加载
  仅对第一级筛选出的笔记加载完整正文
```

每个检索维度对应一个不可替代的字段，缺了任何一步检索链都会断裂。
