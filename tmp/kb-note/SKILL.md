# 知识笔记创建

将零散知识内容锻造为结构化的 Obsidian 笔记。

## 流程

```
1. 读取环境配置
2. 扫描已有笔记图谱
3. 与用户确认笔记核心
4. 逐层锻造内容
5. 生成 Frontmatter
6. 写入文件
```

---

## 1. 读取环境配置

Read `CLAUDE.md` 获取知识库目录规则和文件组织约定。如果 CLAUDE.md 中没有定义知识库目录，询问用户笔记应存放的路径。

---

## 2. 扫描已有笔记图谱

在写入新笔记前，先扫描知识库目录下所有 `.md` 文件的 YAML Frontmatter。

提取每条已有笔记的：
- `anchor`：核心判断
- `tags`：内容标签
- `leads`：检索线索
- `aliases`：别名
- `support` / `resonate` / `tension` / `instance`：已有连接

这一步的目的：
1. 发现新笔记可以建立的关联（支撑、共振、张力、实例）
2. 确保标签和已有体系一致（复用优先于新建）
3. 检索线索不与已有笔记的 tags 过度重叠

---

## 3. 与用户确认笔记核心

笔记的核心判断是整条笔记的灵魂。**写之前必须和用户确认。**

向用户确认两件事：
1. **这条笔记的核心判断是什么？**（anchor 的候选）
2. **这条知识的来源是什么？**（帮助生成 leads）

如果用户已经提供了完整的知识内容，从中提取 anchor 候选，展示给用户确认。
不要自作主张——核心判断的表述必须经用户点头才算成立。

---

## 4. 逐层锻造内容

Read `references/note-template.md` 获取每一层的完整格式说明。

锻造顺序和要点：

### 4.1 锚定层

从第 3 步确认的 anchor 展开。补充必要的限定条件和背景。

**约束：**
- 必须是用户消化后的表述，不是照搬原文
- 如果无法用一段话说清，说明笔记粒度太粗，建议拆分

### 4.2 源头层（可选）

什么情境触发了这段知识？当时在解决什么具体问题？

**约束：**
- 如果没有明确的触发问题（随机阅读中遇到的），跳过这一层，不要硬写
- 如果有明确触发，写下来——它帮你日后回忆"我当时为什么要学这个"

### 4.3 根基层

填充支撑锚定判断的证据。

**约束：**
- 区分强证据（实验、数据、可验证来源）和弱证据（个人感受、未验证观点）
- 明确标注来源（谁说的、哪本书、哪个实验）
- 每个支撑点保持简洁——功能是日后判断知识是否还靠谱

### 4.4 疆界层

这是最容易被跳过的层，但恰恰是最有价值的层。

**约束：**
- 至少写 2 个边界条件
- 明确标注"不确定"的灰色地带——不确定比假装确定更有价值
- 疆界不是"补充说明"，是"这条知识在哪里不好用的地图"

### 4.5 脉络层

基于第 2 步扫描的已有笔记图谱，建立关联。

**约束：**
- 每条关联必须带完整的上下文（为什么相关、底层机制是什么、区别在哪）
- 使用格式：`类型 [[笔记名]]：完整描述`
  - 支撑 `[[笔记名]]：...`
  - 共振 `[[笔记名]]：...`
  - 张力 `[[笔记名]]：...`
  - 实例 `[[笔记名]]：...`
- 四种关联类型的判断标准：

| 类型 | 回答的问题 | 判断标准 |
|------|----------|---------|
| support | A 为 B 提供了什么基础？ | 如果 A 被推翻，B 会动摇吗？ |
| resonate | A 和 B 底层有什么共同结构？ | 跨领域能互相迁移洞察吗？ |
| tension | A 和 B 在哪里矛盾？ | 两者能同时成立吗？ |
| instance | A 在现实中长什么样？ | 能说"B 是 A 的一个例子"吗？ |

- 如果找不到有意义的关联，可以留空——宁可没有关联，不要硬凑

### 4.6 活用层

写出"知道这条知识后，在什么场景下会做出不同的决策"。

**约束：**
- 每个场景必须是完整的"触发条件 → 应对方式"闭环
- 如果写不出任何活用场景，标注"暂无明确活用场景，待积累"

---

## 5. 生成 Frontmatter

基于正文内容，生成 Frontmatter。字段分为两组：

### 5.1 人维护字段（必填）

#### anchor

从锚定层压缩为一句话。用第 3 步用户确认的版本。

**质量检验：** 删掉所有其他内容只留这一句，理解还能被激活吗？不能 → 重写。

#### tags

从锚定层和根基层提取内容分类。

**约束：**
- 2-5 个，不超过 5 个
- 只放"这条笔记讲的是什么"——领域、主题、分类
- 优先复用已有标签体系中的标签
- 不要放人名、意象、场景词——那些是 leads

#### leads

从知识的来源、联想、应用场景中提取检索线索。

**约束：**
- 问自己：用户日后会在搜索框里敲什么词来找这条知识？
- 典型来源：知识出处的人物/书名、类比意象、对立概念、应用场景关键词
- 不要放笔记本身涉及的主题词——那些是 tags
- `tags` 和 `leads` 不应有重叠项

### 5.2 LLM 维护字段（自动生成）

#### aliases

从笔记标题和 anchor 提取别名。包括英文名、同义词、其他常见叫法。

#### support / resonate / tension / instance

从脉络层提取关联，压缩为 Frontmatter 格式。

```yaml
support:
  - "[[笔记名]] 支撑理由"
resonate:
  - "[[笔记名]] 共振理由"
tension:
  - "[[笔记名]] 张力说明"
instance:
  - "[[笔记名]] 实例说明"
```

**约束：**
- 使用 Obsidian wikilink 格式 `[[笔记名]]`，必须用引号包裹
- 理由压缩为一句话——让 LLM 判断是否值得加载
- 如果某个类型没有关联，不写该字段（不要用空列表 `[]`，Obsidian 会将空列表转为 `null`）

---

## 6. 写入文件

将完整笔记写入知识库目录。

**文件命名：** 使用笔记标题作为文件名，`.md` 后缀。

**写入前检查：**
- YAML Frontmatter 格式正确（用 `---` 分隔）
- 人维护字段和 LLM 维护字段用注释分组
- 所有 `[[wikilinks]]` 用引号包裹
- 没有嵌套 YAML 结构（Obsidian 不支持）
- 正文使用 `##` 层级标题
- 沉积层 `## 沉积` 后留空——新笔记暂无迁移事件

**如果目标目录不存在：** 先询问用户确认路径，再创建。

---

## 反面代价

如果跳过扫描已有笔记图谱直接写新笔记，图谱会断裂——新笔记成为孤岛，无法被其他笔记的 connections 发现，也无法发现已有的可关联笔记。累积效应是知识库退化为松散的文件集合，失去图谱价值。

如果跳过与用户确认笔记核心，anchor 可能与用户真正的理解偏差——这个偏差会传染到 tags、leads、connections 的生成，导致整条笔记的检索和关联都偏了。

如果源头层没有明确触发问题却硬写，会产生低质量内容——"在一本书里读到的"是来源信息不是源头，应该放在根基层。
